================================================================================
  SUMARIO
================================================================================

  Este ADR define o MODELO DE DADOS completo da base vetorial que suporta
  o RAG Corporativo. O modelo combina duas capacidades na mesma engine:

  1. GRAFO DE CONHECIMENTO — nos e relacoes que representam a estrutura
     organizacional do conhecimento (quem escreveu, para qual sistema,
     de qual modulo, com que relacoes)

  2. BUSCA VETORIAL — embeddings armazenados nos chunks para busca
     semantica (encontrar trechos relevantes por similaridade de
     significado, nao por keywords)

  A combinacao dessas duas capacidades e o que torna o projeto um
  GraphRAG e nao apenas um RAG convencional. O grafo AMPLIA o contexto
  recuperado pela busca vetorial — quando o usuario pergunta algo, o
  sistema nao retorna apenas chunks similares, mas tambem navega o
  grafo para trazer contexto relacionado (outros documentos do mesmo
  sistema, decisoes que impactam o modulo, responsaveis, glossario).

================================================================================
  🧠 1. CONTEXTO
================================================================================

  1.1. O QUE A BASE VETORIAL PRECISA SUPORTAR
  ---------------------------------------------------------------------------

  O pipeline de 4 fases da ADR-001 produz, na Fase 4, dados que precisam
  ser armazenados e consultados pela base vetorial. Especificamente:

  a) DOCUMENTOS .md (Fase 3): arquivos Markdown com front matter rico
     contendo metadados de governanca (sistema, modulo, owner, team,
     confidentiality, status, temporalidade).

  b) CHUNKS: fragmentos semanticos dos documentos, cada um com seu
     embedding vetorial. Sao a unidade basica de busca — o retriever
     busca chunks, nao documentos inteiros.

  c) ENTIDADES ORGANIZACIONAIS: sistemas, modulos, times, responsaveis,
     termos de glossario, tarefas, ADRs. Essas entidades existem
     independentemente dos documentos e formam o esqueleto do grafo.

  d) RELACOES: conexoes tipadas entre entidades (documento pertence a
     modulo, modulo pertence a sistema, documento referencia outro
     documento, etc.).

  e) VERSIONAMENTO TEMPORAL: documentos que mudam ao longo do tempo
     (leis, regulacoes, politicas) precisam de rastreabilidade de
     versoes (qual era a versao vigente em uma data especifica?).

  1.2. ESCOPO DO MODELO
  ---------------------------------------------------------------------------

  IMPORTANTE: este modelo cobre APENAS os artefatos da Fase 3 (.md final) e
  Fase 4 (Base Vetorial) do pipeline definido na ADR-001. O .beta.md (Fase 2)
  reside exclusivamente no repositorio rag-workspace e NAO e modelado na
  Base Vetorial. Essa exclusao e deliberada: o .beta.md e um artefato de
  trabalho, nao uma fonte da verdade. Apenas o .md final (promovido pela
  Fase 3) e ingerido na Base Vetorial.

  1.4. REQUISITOS DO MODELO
  ---------------------------------------------------------------------------

  O modelo de dados deve atender a:

  R1 — BUSCA SEMANTICA: suportar vector index nos embeddings dos chunks
       para busca por similaridade (cosine similarity, >0.7 threshold).

  R2 — EXPANSAO POR GRAFO: a partir de um chunk encontrado por busca
       vetorial, navegar o grafo para encontrar contexto relacionado
       (outros chunks do mesmo documento, documentos do mesmo sistema,
       ADRs que decidem sobre o modulo, etc.).

  R3 — FILTROS PRE-RETRIEVAL: filtrar por confidentiality, system,
       module, status, domain ANTES da busca vetorial. Isso garante
       que o usuario so veja dados que tem acesso (seguranca) e que
       sao relevantes (escopo).

  R4 — TEMPORALIDADE: consultar "como era X na data Y?" usando
       valid_from/valid_until e a cadeia de versoes (SUPERSEDES).

  R5 — RASTREABILIDADE: de qualquer chunk, ser possivel rastrear ate
       a fonte original (documento, repositorio, commit, autor).

  R6 — GOVERNANCA: cada documento tem owner, team, status, tags que
       permitem filtros gerenciais (ex: "todos os documentos draft
       do time de arquitetura").

  R7 — GLOSSARIO: termos de negocio controlados, vinculados aos
       documentos que os utilizam, para desambiguacao e consistencia.

  R8 — DEPENDENCIAS: relacoes entre modulos (DEPENDS_ON) e entre
       tarefas e modulos (IMPLEMENTS) para analise de impacto.

  2.1. NOS (LABELS) — 10 TIPOS
  =========================================================================

  -------------------------------------------------------------------------
  NO 1: Document
  -------------------------------------------------------------------------

  DESCRICAO: Representa um arquivo .md ingerido na base vetorial. E o no
  central do grafo — todos os outros nos se relacionam com ele direta ou
  indiretamente. Cada arquivo .md gera EXATAMENTE um no Document.

  POR QUE EXISTE: e a unidade fundamental de conhecimento. Sem ele, nao
  ha como rastrear a origem de um chunk, nem como aplicar governanca
  (quem escreveu, para qual sistema, com qual confidencialidade).

  PROPRIEDADES:

  +---------------------+---------+----+------------------------------------------+
  | Propriedade         | Tipo    | Req| Descricao e Razao                        |
  +---------------------+---------+----+------------------------------------------+
  | document_id         | String  | S  | Identificador unico (ex: "DOC-000123").  |
  |                     |         |    | Formato: DOC-NNNNNN. Vem do front matter |
  |                     |         |    | campo "id". CONSTRAINT de unicidade.      |
  |                     |         |    | POR QUE: necessario para referenciacao    |
  |                     |         |    | cruzada, deduplicacao e rastreabilidade.  |
  +---------------------+---------+----+------------------------------------------+
  | title               | String  | S  | Titulo do documento. Vem do front matter.|
  |                     |         |    | POR QUE: exibicao para o usuario e busca  |
  |                     |         |    | full-text.                                |
  +---------------------+---------+----+------------------------------------------+
  | doc_type            | String  | S  | Tipo do documento. Valores permitidos:    |
  |                     |         |    | system-doc, adr, runbook, glossary,       |
  |                     |         |    | task-doc, architecture-doc.               |
  |                     |         |    | POR QUE: determina a estrategia de        |
  |                     |         |    | chunking (secao 2.4) e o tratamento       |
  |                     |         |    | diferenciado na busca.                    |
  +---------------------+---------+----+------------------------------------------+
  | system              | String  | S  | Nome do sistema ao qual pertence.         |
  |                     |         |    | POR QUE: filtro pre-retrieval. Permite    |
  |                     |         |    | buscar "documentos do Sistema X".         |
  +---------------------+---------+----+------------------------------------------+
  | module              | String  | N  | Modulo funcional do sistema.              |
  |                     |         |    | POR QUE: granularidade fina de filtro.    |
  |                     |         |    | Um sistema pode ter 50+ documentos;       |
  |                     |         |    | filtrar por modulo reduz o escopo.         |
  +---------------------+---------+----+------------------------------------------+
  | domain              | String  | S  | Dominio de negocio (ex: Financeiro, RH).  |
  |                     |         |    | POR QUE: permite buscas por area de       |
  |                     |         |    | negocio, independente de sistema.          |
  +---------------------+---------+----+------------------------------------------+
  | owner               | String  | S  | Identificador do responsavel.             |
  |                     |         |    | POR QUE: rastreabilidade de autoria e     |
  |                     |         |    | responsabilidade pelo conteudo.            |
  +---------------------+---------+----+------------------------------------------+
  | team                | String  | S  | Time responsavel.                         |
  |                     |         |    | POR QUE: governanca por time. Permite     |
  |                     |         |    | relatorios tipo "documentos do time X".   |
  +---------------------+---------+----+------------------------------------------+
  | status              | String  | S  | Status do documento. Valores: draft,      |
  |                     |         |    | in-review, approved, deprecated.          |
  |                     |         |    | POR QUE: filtro pre-retrieval. Documentos |
  |                     |         |    | draft nao devem aparecer em buscas de     |
  |                     |         |    | producao. Deprecated devem ter peso menor.|
  +---------------------+---------+----+------------------------------------------+
  | confidentiality     | String  | S  | Nivel de confidencialidade. Valores:      |
  |                     |         |    | public, internal, restricted, confidential|
  |                     |         |    | POR QUE: FILTRO CRITICO de seguranca.     |
  |                     |         |    | Determina a trilha (ADR-002) e o acesso.  |
  |                     |         |    | Aplicado ANTES da busca vetorial.         |
  +---------------------+---------+----+------------------------------------------+
  | tags                | [String]| N  | Lista de tags para descoberta.            |
  |                     |         |    | POR QUE: busca por termos livres sem      |
  |                     |         |    | depender de busca semantica.              |
  +---------------------+---------+----+------------------------------------------+
  | path                | String  | S  | Caminho do arquivo no repositorio Git.    |
  |                     |         |    | Ex: "docs/financeiro/cobranca.md"         |
  |                     |         |    | POR QUE: rastreabilidade ate o arquivo    |
  |                     |         |    | fonte. Permite links diretos.             |
  +---------------------+---------+----+------------------------------------------+
  | source_repo         | String  | S  | Nome do repositorio Git de origem.        |
  |                     |         |    | POR QUE: em ambientes com multiplos repos,|
  |                     |         |    | necessario saber de qual veio.            |
  +---------------------+---------+----+------------------------------------------+
  | git_commit_hash     | String  | S  | Hash do commit que originou esta versao.  |
  |                     |         |    | POR QUE: imutabilidade e reprodutibilidade|
  |                     |         |    | (pilar H da ADR-001). Dado este hash,     |
  |                     |         |    | pode-se reconstruir exatamente o doc.     |
  +---------------------+---------+----+------------------------------------------+
  | git_branch          | String  | N  | Branch de origem (normalmente "main").    |
  |                     |         |    | POR QUE: auditoria e rastreabilidade.     |
  +---------------------+---------+----+------------------------------------------+
  | valid_from          | Date    | S  | Data de inicio de vigencia do conteudo.   |
  |                     |         |    | POR QUE: temporalidade (ADR-001 s2.6).    |
  |                     |         |    | Permite filtrar "vigente em data X".      |
  +---------------------+---------+----+------------------------------------------+
  | valid_until         | Date    | N  | Data de fim de vigencia. null = vigente.  |
  |                     |         |    | POR QUE: documentos que foram superados   |
  |                     |         |    | por versao mais recente tem valid_until    |
  |                     |         |    | preenchido. Permite consultas historicas.  |
  +---------------------+---------+----+------------------------------------------+
  | superseded_by       | String  | N  | document_id do documento que substitui.   |
  |                     |         |    | POR QUE: cadeia de versoes no front matter|
  |                     |         |    | (redundancia com relacao SUPERSEDES para  |
  |                     |         |    | queries que nao usam grafo).              |
  +---------------------+---------+----+------------------------------------------+
  | conversion_quality  | Float   | S  | Score 0-100 da qualidade de conversao     |
  |                     |         |    | da fonte original para .md (ADR-001 s2.5.1|
  |                     |         |    | POR QUE: permite filtrar por confiabilidade|
  |                     |         |    | Chunks de docs com score baixo devem ter  |
  |                     |         |    | peso menor no reranking.                  |
  +---------------------+---------+----+------------------------------------------+
  | source_format       | String  | S  | Formato da fonte original (pdf, docx,     |
  |                     |         |    | confluence, email, md, etc.).             |
  |                     |         |    | POR QUE: rastreabilidade de linhagem.     |
  +---------------------+---------+----+------------------------------------------+
  | created_at          | DateTime| S  | Data/hora de criacao do no na base.       |
  |                     |         |    | POR QUE: auditoria e ordenacao.           |
  +---------------------+---------+----+------------------------------------------+
  | updated_at          | DateTime| S  | Data/hora da ultima atualizacao.          |
  |                     |         |    | POR QUE: deteccao de stale data.          |
  +---------------------+---------+----+------------------------------------------+
  | chunk_count         | Integer | S  | Quantidade de chunks gerados.             |
  |                     |         |    | POR QUE: metrica de tamanho e observa-    |
  |                     |         |    | bilidade. Permite detectar docs com       |
  |                     |         |    | chunking anomalo (0 chunks = erro).       |
  +---------------------+---------+----+------------------------------------------+

  Legenda: Req = Requerido (S=Sim, N=Nao)

  -------------------------------------------------------------------------
  NO 2: Chunk
  -------------------------------------------------------------------------

  DESCRICAO: Fragmento semantico de um Document. E a unidade atomica de
  busca vetorial — o embedding e gerado POR CHUNK, nao por documento.
  Um Document gera N chunks (tipicamente 3-20, dependendo do tamanho).

  POR QUE EXISTE: documentos inteiros sao grandes demais para servir
  como contexto para um LLM (limitacao de janela de contexto) e imprecisos
  demais para busca semantica (um documento pode tratar de 10 temas
  diferentes). Chunks permitem recuperar TRECHOS relevantes, nao paginas
  inteiras.

  PROPRIEDADES:

  +---------------------+---------+----+------------------------------------------+
  | Propriedade         | Tipo    | Req| Descricao e Razao                        |
  +---------------------+---------+----+------------------------------------------+
  | chunk_id            | String  | S  | Identificador unico. Formato:            |
  |                     |         |    | "{document_id}_chunk_{NNN}"               |
  |                     |         |    | Ex: "DOC-000123_chunk_003"                |
  |                     |         |    | CONSTRAINT de unicidade.                  |
  |                     |         |    | POR QUE: referenciacao, deduplicacao,     |
  |                     |         |    | rastreabilidade do chunk especifico.      |
  +---------------------+---------+----+------------------------------------------+
  | content             | String  | S  | Texto do chunk (o conteudo real).         |
  |                     |         |    | POR QUE: necessario para exibicao ao      |
  |                     |         |    | usuario e para o LLM usar como contexto.  |
  |                     |         |    | Sem o conteudo, o embedding sozinho nao   |
  |                     |         |    | serve para nada.                          |
  +---------------------+---------+----+------------------------------------------+
  | embedding           | [Float] | S  | Vetor de embedding (1536d ou 1024d        |
  |                     |         |    | conforme trilha — ADR-002).               |
  |                     |         |    | POR QUE: e o vetor que viabiliza a busca  |
  |                     |         |    | semantica. Sem ele, so resta busca por    |
  |                     |         |    | keywords (full-text).                     |
  +---------------------+---------+----+------------------------------------------+
  | embedding_model     | String  | S  | Nome do modelo que gerou o embedding.     |
  |                     |         |    | Ex: "openai/text-embedding-3-small" ou    |
  |                     |         |    | "baai/bge-m3"                             |
  |                     |         |    | POR QUE: quando o modelo de embedding     |
  |                     |         |    | mudar, precisamos saber QUAIS chunks      |
  |                     |         |    | precisam ser re-embedados. Sem essa info, |
  |                     |         |    | teriamos que re-embedar TUDO.             |
  +---------------------+---------+----+------------------------------------------+
  | embedding_version   | String  | N  | Versao do modelo de embedding.            |
  |                     |         |    | POR QUE: mesmo modelo pode ter versoes    |
  |                     |         |    | diferentes com embeddings incompativeis.  |
  +---------------------+---------+----+------------------------------------------+
  | token_count         | Integer | S  | Quantidade de tokens do chunk.            |
  |                     |         |    | POR QUE: monitoramento da faixa ideal     |
  |                     |         |    | (300-800 tokens). Chunks muito curtos     |
  |                     |         |    | (<100) ou muito longos (>1000) indicam    |
  |                     |         |    | problema no chunking.                     |
  +---------------------+---------+----+------------------------------------------+
  | chunk_index         | Integer | S  | Posicao sequencial do chunk no documento  |
  |                     |         |    | (0, 1, 2, ...).                           |
  |                     |         |    | POR QUE: permite reconstruir a ordem      |
  |                     |         |    | original do documento. Quando o LLM       |
  |                     |         |    | recebe 3 chunks do mesmo doc, a ordem     |
  |                     |         |    | importa para coerencia.                   |
  +---------------------+---------+----+------------------------------------------+
  | heading_path        | String  | N  | Hierarquia de headings ate este chunk.    |
  |                     |         |    | Ex: "# Cobranca > ## Boleto > ### Remessa"|
  |                     |         |    | POR QUE: heranca de contexto. O chunk     |
  |                     |         |    | "Remessa" sozinho e ambiguo; com heading  |
  |                     |         |    | path, o LLM sabe que se trata de remessa  |
  |                     |         |    | de boleto no modulo de cobranca.          |
  +---------------------+---------+----+------------------------------------------+
  | metadata_inherited  | Map     | N  | Metadados herdados do Document pai        |
  |                     |         |    | (system, module, domain, confidentiality).|
  |                     |         |    | POR QUE: permite filtro direto no chunk   |
  |                     |         |    | sem precisar fazer JOIN com Document.     |
  |                     |         |    | Desnormalizacao intencional para          |
  |                     |         |    | performance de busca vetorial com filtro. |
  +---------------------+---------+----+------------------------------------------+
  | created_at          | DateTime| S  | Data/hora de criacao.                     |
  +---------------------+---------+----+------------------------------------------+

  NOTA SOBRE DESNORMALIZACAO: o campo metadata_inherited duplica dados do
  Document pai. Isso e INTENCIONAL. A busca vetorial com filtro precisa
  acessar o campo de filtro E o embedding no MESMO no. Se confidentiality
  estiver apenas no Document, toda busca vetorial precisaria de JOIN
  (Chunk->Document), o que degrada performance significativamente em
  vector indexes.

  -------------------------------------------------------------------------
  NO 3: System
  -------------------------------------------------------------------------

  DESCRICAO: Representa um sistema corporativo (software, plataforma,
  servico). E o nivel mais alto de agrupamento organizacional do
  conhecimento.

  POR QUE EXISTE: permite agrupar documentos e modulos por sistema.
  Viabiliza perguntas como "tudo sobre o Sistema de Cobranca" e analise
  de impacto ("quais documentos sao afetados se o Sistema X mudar?").

  PROPRIEDADES:

  +---------------------+---------+----+------------------------------------------+
  | Propriedade         | Tipo    | Req| Descricao e Razao                        |
  +---------------------+---------+----+------------------------------------------+
  | system_id           | String  | S  | Identificador unico (ex: "SYS-COBRANCA").|
  |                     |         |    | POR QUE: referenciacao e deduplicacao.    |
  +---------------------+---------+----+------------------------------------------+
  | name                | String  | S  | Nome legivel do sistema.                 |
  |                     |         |    | POR QUE: exibicao para usuarios.         |
  +---------------------+---------+----+------------------------------------------+
  | description         | String  | N  | Descricao do sistema.                    |
  |                     |         |    | POR QUE: contexto para o LLM entender    |
  |                     |         |    | do que se trata o sistema.               |
  +---------------------+---------+----+------------------------------------------+
  | domain              | String  | S  | Dominio de negocio do sistema.           |
  |                     |         |    | POR QUE: agrupamento por area de negocio.|
  +---------------------+---------+----+------------------------------------------+
  | status              | String  | S  | Status: active, deprecated, planned.     |
  |                     |         |    | POR QUE: sistemas depreciados devem ter  |
  |                     |         |    | peso menor na busca.                     |
  +---------------------+---------+----+------------------------------------------+
  | created_at          | DateTime| S  | Data de criacao.                         |
  +---------------------+---------+----+------------------------------------------+
  | updated_at          | DateTime| S  | Data da ultima atualizacao.              |
  +---------------------+---------+----+------------------------------------------+

  -------------------------------------------------------------------------
  NO 4: Module
  -------------------------------------------------------------------------

  DESCRICAO: Modulo funcional de um sistema. Representa uma subdivisao
  logica (ex: Sistema de Cobranca -> Modulo de Boleto, Modulo de Remessa,
  Modulo de Retorno).

  POR QUE EXISTE: granularidade fina. Um sistema pode ter dezenas de
  documentos. Modulos permitem filtrar e navegar o grafo com mais
  precisao. Tambem viabiliza analise de DEPENDENCIAS entre modulos
  (DEPENDS_ON).

  PROPRIEDADES:

  +---------------------+---------+----+------------------------------------------+
  | Propriedade         | Tipo    | Req| Descricao e Razao                        |
  +---------------------+---------+----+------------------------------------------+
  | module_id           | String  | S  | Identificador unico (ex: "MOD-BOLETO").  |
  +---------------------+---------+----+------------------------------------------+
  | name                | String  | S  | Nome legivel do modulo.                  |
  +---------------------+---------+----+------------------------------------------+
  | description         | String  | N  | Descricao do modulo.                     |
  +---------------------+---------+----+------------------------------------------+
  | status              | String  | S  | Status: active, deprecated, planned.     |
  +---------------------+---------+----+------------------------------------------+
  | created_at          | DateTime| S  | Data de criacao.                         |
  +---------------------+---------+----+------------------------------------------+
  | updated_at          | DateTime| S  | Data da ultima atualizacao.              |
  +---------------------+---------+----+------------------------------------------+

  -------------------------------------------------------------------------
  NO 5: Owner
  -------------------------------------------------------------------------

  DESCRICAO: Responsavel por conteudo, sistema ou modulo. Pode ser uma
  pessoa (PO, analista, arquiteto) ou uma conta de servico (pipeline).

  POR QUE EXISTE: rastreabilidade de autoria e responsabilidade. Permite
  responder "quem escreveu este documento?", "quem e responsavel por
  este sistema?", "quem devo procurar se tiver duvida sobre X?".

  PROPRIEDADES:

  +---------------------+---------+----+------------------------------------------+
  | Propriedade         | Tipo    | Req| Descricao e Razao                        |
  +---------------------+---------+----+------------------------------------------+
  | owner_id            | String  | S  | Identificador unico (ex: "fabio",        |
  |                     |         |    | "pipeline-v1").                          |
  +---------------------+---------+----+------------------------------------------+
  | display_name        | String  | S  | Nome para exibicao.                      |
  +---------------------+---------+----+------------------------------------------+
  | email               | String  | N  | E-mail de contato.                       |
  +---------------------+---------+----+------------------------------------------+
  | type                | String  | S  | Tipo: person, service-account.           |
  |                     |         |    | POR QUE: distinguir autoria humana de     |
  |                     |         |    | autoria automatizada.                    |
  +---------------------+---------+----+------------------------------------------+

  NO 10: DocumentFamily
  -------------------------------------------------------------------------

  DESCRICAO: Agrupa todas as versoes de um mesmo documento conceitual
  ao longo do tempo. Por exemplo, a "Politica de Privacidade" pode ter
  versoes 2024, 2025 e 2026 — todas pertencem a mesma DocumentFamily.

  POR QUE EXISTE: versionamento temporal. Sem este no, nao ha como
  saber que DOC-000123 (v2024) e DOC-000456 (v2025) e DOC-000789
  (v2026) sao versoes do MESMO documento conceitual. A DocumentFamily
  agrupa todas as versoes e aponta para a versao corrente.

  Isso viabiliza consultas como:
  - "Qual a versao vigente da Politica de Privacidade?" (current_version)
  - "Como era a Politica de Privacidade em 2024?" (valid_from/valid_until)
  - "O que mudou entre a versao 2024 e 2025?" (diff entre versoes)

  =========================================================================
  2.3. CONSTRAINTS E INDICES
  =========================================================================

  -------------------------------------------------------------------------
  2.3.1. CONSTRAINTS DE UNICIDADE
  -------------------------------------------------------------------------

  Os constraints garantem que nao existam dois nos com o mesmo
  identificador. Sao a primeira linha de defesa contra duplicatas.

  Constraint 1: Document.document_id — UNIQUE
  POR QUE: se dois documentos tiverem o mesmo ID, o grafo inteiro
  fica inconsistente (relacoes apontam para o no errado, chunks ficam
  orfaos). E o constraint mais critico do modelo.

  Constraint 2: Chunk.chunk_id — UNIQUE
  POR QUE: na re-ingestao (ADR-001 secao 2.7), o pipeline precisa
  fazer UPSERT (update se existe, insert se nao). Sem unicidade no
  chunk_id, o upsert nao funciona e chunks duplicados sao criados.

  Constraint 3: DocumentFamily.family_id — UNIQUE
  POR QUE: se duas familias tiverem o mesmo ID, versoes de documentos
  diferentes ficam misturadas.

  Constraint 4: System.system_id — UNIQUE
  Constraint 5: Module.module_id — UNIQUE
  Constraint 6: Owner.owner_id — UNIQUE
  Constraint 7: Team.team_id — UNIQUE
  Constraint 8: Task.task_id — UNIQUE
  Constraint 9: GlossaryTerm.term_id — UNIQUE
  Constraint 10: ADR.adr_id — UNIQUE

  POR QUE TODOS TEM CONSTRAINT: todo no que pode ser referenciado por
  outros nos DEVE ter unicidade garantida. Sem isso, relacoes podem
  apontar para o no errado e o grafo se corrompe silenciosamente.

  EXEMPLO DE CRIACAO (Cypher como referencia — sintaxe pode variar conforme
  a Base Vetorial adotada):

  CREATE CONSTRAINT doc_unique_id FOR (d:Document) REQUIRE d.document_id IS UNIQUE;
  CREATE CONSTRAINT chunk_unique_id FOR (c:Chunk) REQUIRE c.chunk_id IS UNIQUE;
  CREATE CONSTRAINT family_unique_id FOR (f:DocumentFamily) REQUIRE f.family_id IS UNIQUE;
  CREATE CONSTRAINT system_unique_id FOR (s:System) REQUIRE s.system_id IS UNIQUE;
  CREATE CONSTRAINT module_unique_id FOR (m:Module) REQUIRE m.module_id IS UNIQUE;
  CREATE CONSTRAINT owner_unique_id FOR (o:Owner) REQUIRE o.owner_id IS UNIQUE;
  CREATE CONSTRAINT team_unique_id FOR (t:Team) REQUIRE t.team_id IS UNIQUE;
  CREATE CONSTRAINT task_unique_id FOR (tk:Task) REQUIRE tk.task_id IS UNIQUE;
  CREATE CONSTRAINT glossary_unique_id FOR (g:GlossaryTerm) REQUIRE g.term_id IS UNIQUE;
  CREATE CONSTRAINT adr_unique_id FOR (a:ADR) REQUIRE a.adr_id IS UNIQUE;

  -------------------------------------------------------------------------
  2.3.2. VECTOR INDEX
  -------------------------------------------------------------------------

  Index: vector_index_chunk_embedding
  Alvo: Chunk.embedding
  Dimensao: 1536 (Trilha A) ou 1024 (Trilha B) — conforme ADR-002
  Similaridade: cosine
  POR QUE: e o indice que viabiliza a busca semantica. Sem ele, a busca
  vetorial faria scan sequencial em todos os chunks (inviavel para
  bases com >10K chunks).

  IMPORTANTE: como a dimensao do embedding difere entre trilhas (1536 vs.
  1024), instancias de trilhas diferentes NAO PODEM compartilhar o mesmo
  vector index. Isso reforça a separacao de instancias definida na ADR-002.

  EXEMPLO DE CRIACAO:

  -- Trilha A (OpenAI, 1536 dimensoes)
  CREATE VECTOR INDEX chunk_embedding_index FOR (c:Chunk)
  ON (c.embedding)
  OPTIONS {indexConfig: {
    `vector.dimensions`: 1536,
    `vector.similarity_function`: 'cosine'
  }};

  -- Trilha B (BGE-M3, 1024 dimensoes)
  CREATE VECTOR INDEX chunk_embedding_index FOR (c:Chunk)
  ON (c.embedding)
  OPTIONS {indexConfig: {
    `vector.dimensions`: 1024,
    `vector.similarity_function`: 'cosine'
  }};

  -------------------------------------------------------------------------
  2.3.3. INDICES COMPOSTOS E AUXILIARES
  -------------------------------------------------------------------------

  Os indices auxiliares otimizam os filtros PRE-RETRIEVAL mais frequentes.
  Sem eles, cada filtro faria scan sequencial.

  Indice 1: Document.doc_type
  POR QUE: filtro "buscar apenas em ADRs" ou "apenas em runbooks" e
  muito frequente. Sem indice, scan em todos os documentos.

  Indice 2: Document.system
  POR QUE: filtro "buscar apenas no Sistema X" e o filtro de escopo
  mais usado.

  Indice 3: Document.module
  POR QUE: refinamento de escopo abaixo de system.

  Indice 4: Document.confidentiality
  POR QUE: FILTRO CRITICO de seguranca. Aplicado em TODA busca. Deve
  ser extremamente rapido.

  Indice 5: Document.status
  POR QUE: filtrar por draft/approved/deprecated. Buscas de producao
  tipicamente filtram status=approved.

  Indice 6: Document.domain
  POR QUE: filtro por area de negocio.

  Indice 7: Document.path
  POR QUE: lookup rapido por caminho de arquivo (usado na ingestao
  para detectar se o documento ja existe).

  Indice 8: Document.valid_from + Document.valid_until (composto)
  POR QUE: consultas temporais ("vigente em data X") precisam filtrar
  por ambos os campos simultaneamente.

  Indice 9: Chunk.metadata_inherited.confidentiality
  POR QUE: filtro de seguranca diretamente no chunk, sem JOIN com
  Document. Usado em conjunto com o vector index.

  Indice 10: GlossaryTerm.term
  POR QUE: lookup rapido por termo (usado na expansao de sinonimos).

  FULL-TEXT INDEX:

  Indice FT-1: Document.title (full-text)
  POR QUE: busca por titulo com suporte a parcial matching e relevance
  scoring.

  Indice FT-2: GlossaryTerm.term + GlossaryTerm.synonyms (full-text)
  POR QUE: busca fuzzy por termos e sinonimos.

  =========================================================================
  2.4. ESTRATEGIA DE CHUNKING POR TIPO DE DOCUMENTO
  =========================================================================

  O chunking NAO e "tamanho unico para todos". Diferentes tipos de
  documento tem estruturas diferentes e exigem estrategias diferentes.
  A escolha errada de chunking degrada a qualidade do RAG mais do que
  qualquer outro fator (incluindo modelo de embedding).

  -------------------------------------------------------------------------
  2.4.1. system-doc (Documentacao de Sistema)
  -------------------------------------------------------------------------

  Estrategia: CHUNKING POR HEADING (H1, H2, H3) — MESMA BASE DO architecture-doc
  Faixa de tokens: 300-800

  COMO FUNCIONA:
  - Segue a mesma logica de chunking por heading descrita em 2.4.2
    (architecture-doc), pois documentacao de sistema e estruturalmente
    similar a documentacao de arquitetura.
  - Diferencial: system-docs frequentemente contem secoes de
    "Configuracao", "Integracao" e "Operacao" que devem ser chunks
    separados, mesmo se curtos (<300 tokens), pois sao consultados
    individualmente por diferentes personas (dev, ops, suporte).

  POR QUE ESTA ESTRATEGIA: system-docs descrevem sistemas corporativos
  com multiplas facetas (arquitetura, configuracao, integracao, operacao).
  Cada faceta e um topico semantico autonomo. A estrategia por heading
  garante que uma busca por "configuracao do Sistema X" retorne o chunk
  de configuracao, nao o documento inteiro.

  -------------------------------------------------------------------------
  2.4.2. architecture-doc (Documento de Arquitetura)
  -------------------------------------------------------------------------

  Estrategia: CHUNKING POR HEADING (H1, H2, H3)
  Faixa de tokens: 300-800

  COMO FUNCIONA:
  - Cada heading (## ou ###) inicia um novo chunk
  - Se o conteudo sob um heading exceder 800 tokens, subdivide por
    paragrafo
  - Se o conteudo for menor que 300 tokens, AGRUPA com o proximo
    heading do mesmo nivel
  - O heading_path (hierarquia de headings) e preservado como contexto

  POR QUE ESTA ESTRATEGIA: documentos de arquitetura sao organizados
  por headings que delimitam topicos semanticos (ex: "## Modelo de Dados",
  "## Fluxo de Ingestao"). Cada heading e um conceito autonomo que deve
  ser recuperavel independentemente.

  EXEMPLO:
  Documento "Arquitetura do Sistema de Cobranca" com headings:
  # Cobranca
  ## Visao Geral         -> chunk_001 (heading_path: "Cobranca > Visao Geral")
  ## Modelo de Dados     -> chunk_002 (heading_path: "Cobranca > Modelo de Dados")
  ### Entidades          -> chunk_003 (heading_path: "Cobranca > Modelo de Dados > Entidades")
  ### Relacoes           -> chunk_004 (heading_path: "Cobranca > Modelo de Dados > Relacoes")
  ## Fluxo de Ingestao   -> chunk_005 (heading_path: "Cobranca > Fluxo de Ingestao")

  -------------------------------------------------------------------------
  2.4.3. adr (Registro de Decisao Arquitetural)
  -------------------------------------------------------------------------

  Estrategia: CHUNKS MENORES E PRECISOS POR SECAO
  Faixa de tokens: 200-500

  COMO FUNCIONA:
  - Cada secao padrao do ADR e um chunk separado: Contexto, Decisao,
    Alternativas (uma por alternativa), Consequencias
  - A secao "Decisao" e mantida como chunk unico (nunca subdividida),
    mesmo que exceda 500 tokens, porque e a informacao mais importante
  - Alternativas sao chunks individuais (uma alternativa = um chunk)

  POR QUE ESTA ESTRATEGIA: ADRs sao consultados para perguntas muito
  especificas ("qual foi a decisao sobre X?", "quais alternativas foram
  consideradas para Y?"). Chunks menores e mais precisos melhoram a
  precisao da busca.

  POR QUE CHUNKS MENORES: se o chunk do ADR for grande demais (contexto
  + decisao + alternativas juntos), a busca vetorial pode retornar o
  chunk por similaridade com o contexto quando o usuario quer a decisao.
  Chunks separados por secao evitam isso.

  -------------------------------------------------------------------------
  2.4.4. runbook (Runbook Operacional)
  -------------------------------------------------------------------------

  Estrategia: CHUNK POR PROCEDIMENTO/PASSO
  Faixa de tokens: 200-600

  COMO FUNCIONA:
  - Cada procedimento (secao com passos numerados) e um chunk
  - Pre-requisitos sao um chunk separado
  - Troubleshooting e um chunk separado
  - Passos individuais NÃO sao separados (um procedimento com 5 passos
    e um unico chunk) — porque passos isolados perdem contexto

  POR QUE ESTA ESTRATEGIA: runbooks sao consultados em momentos de
  urgencia ("como reiniciar o servico X?", "o que fazer quando o
  monitor Y dispara?"). O usuario precisa do procedimento COMPLETO,
  nao de um passo isolado.

  -------------------------------------------------------------------------
  2.4.5. glossary (Glossario)
  -------------------------------------------------------------------------

  Estrategia: CHUNK QUASE ATOMICO POR TERMO
  Faixa de tokens: 50-200

  COMO FUNCIONA:
  - Cada termo do glossario e um chunk separado
  - O chunk contem: termo + definicao + sinonimos + exemplo de uso
  - Termos relacionados sao linkados via USES_TERM, nao agrupados

  POR QUE ESTA ESTRATEGIA: termos de glossario sao atomicos por
  natureza. Agrupar varios termos em um chunk diluiria a especificidade
  do embedding (o vetor representaria uma "media" de conceitos, nao
  um conceito especifico). Chunks atomicos garantem que a busca por
  "boleto" retorne EXATAMENTE o chunk do termo "boleto".

  -------------------------------------------------------------------------
  2.4.6. task-doc (Documento de Tarefa)
  -------------------------------------------------------------------------

  Estrategia: CHUNK POR CONTEXTO, ESCOPO, DECISAO E ACEITE
  Faixa de tokens: 300-600

  COMO FUNCIONA:
  - Secao "Contexto/Background" -> 1 chunk
  - Secao "Escopo/Requisitos" -> 1 chunk (ou N se muitos requisitos)
  - Secao "Decisoes Tecnicas" -> 1 chunk por decisao
  - Secao "Criterios de Aceite" -> 1 chunk
  - Secao "Observacoes/Riscos" -> 1 chunk

  POR QUE ESTA ESTRATEGIA: documentos de tarefa sao consultados para
  perguntas diferentes por personas diferentes:
  - PO pergunta: "quais os criterios de aceite da task X?" (chunk de aceite)
  - Dev pergunta: "qual o escopo tecnico da task X?" (chunk de escopo)
  - Arquiteto pergunta: "quais decisoes tecnicas foram tomadas?" (chunk de decisao)
  Chunks separados por secao atendem a cada persona sem ruido.

  =========================================================================
  2.5. MODELO TEMPORAL — DOCUMENTFAMILY + SUPERSEDES + VERSION_OF
  =========================================================================

  O modelo temporal permite rastrear como documentos evoluem ao longo
  do tempo. Funciona em tres camadas (conforme ADR-001 secao 2.6):

  CAMADA 1 — FRONT MATTER DO .md:

  ---
  valid_from: "2026-01-01"
  valid_until: null               # null = vigente
  superseded_by: "DOC-000456"     # ID do doc que substitui
  supersedes: "DOC-000123"        # ID do doc anterior
  ---

  CAMADA 2 — GRAFO (relacoes e nos):

  (:Document {document_id: "DOC-000789", valid_from: "2026-01-01"})
    -[:VERSION_OF {version_number: 3, is_current: true}]->
  (:DocumentFamily {family_id: "FAM-POLITICA-PRIVACIDADE",
                    current_version_id: "DOC-000789"})

  (:Document {document_id: "DOC-000789"})
    -[:SUPERSEDES {effective_date: "2026-01-01"}]->
  (:Document {document_id: "DOC-000456"})

  (:Document {document_id: "DOC-000456"})
    -[:SUPERSEDES {effective_date: "2025-01-01"}]->
  (:Document {document_id: "DOC-000123"})

  CAMADA 3 — RETRIEVAL (filtros temporais):

  Deteccao de contexto temporal na query:
  - "Como era a Politica de Privacidade em 2024?" -> filtrar valid_from <= 2024 E valid_until >= 2024 (ou null)
  - "Qual a Politica de Privacidade atual?" -> filtrar valid_until IS NULL
  - Sem contexto temporal -> assumir data atual (valid_until IS NULL)

  -------------------------------------------------------------------------
  EXEMPLO 5: Queries de referencia (Cypher)
  -------------------------------------------------------------------------

  QUERY 1 — BUSCA SEMANTICA COM FILTRO DE CONFIDENCIALIDADE:

  // Buscar chunks semanticamente similares a pergunta do usuario
  // FILTRO PRE-RETRIEVAL: somente documentos internos ou publicos
  CALL db.index.vector.queryNodes(
    'chunk_embedding_index',     // nome do vector index
    10,                          // top-K resultados
    $query_embedding             // embedding da pergunta do usuario
  ) YIELD node AS chunk, score
  WHERE chunk.metadata_inherited.confidentiality IN ['public', 'internal']
    AND score > 0.7
  RETURN chunk.chunk_id, chunk.content, chunk.heading_path, score
  ORDER BY score DESC;

  QUERY 2 — EXPANSAO DE CONTEXTO VIA GRAFO:

  // A partir de um chunk encontrado, expandir para o documento pai,
  // outros chunks do mesmo documento, e documentos relacionados
  MATCH (chunk:Chunk {chunk_id: $found_chunk_id})-[:PART_OF]->(doc:Document)

  // Outros chunks do mesmo documento (contexto adjacente)
  OPTIONAL MATCH (sibling:Chunk)-[:PART_OF]->(doc)
  WHERE sibling.chunk_index IN [chunk.chunk_index - 1, chunk.chunk_index + 1]

  // Documentos referenciados
  OPTIONAL MATCH (doc)-[:REFERENCES]->(related:Document)

  // Glossario usado
  OPTIONAL MATCH (doc)-[:USES_TERM]->(term:GlossaryTerm)

  RETURN doc.title, doc.system, doc.module,
         collect(DISTINCT sibling.content) AS contexto_adjacente,
         collect(DISTINCT related.title) AS docs_relacionados,
         collect(DISTINCT {termo: term.term, definicao: term.definition}) AS glossario;

  QUERY 3 — CONSULTA TEMPORAL:

  // "Como era a Politica de Privacidade em julho de 2025?"
  MATCH (doc:Document)-[:VERSION_OF]->(family:DocumentFamily {
    family_id: "FAM-POLITICA-PRIVACIDADE"
  })
  WHERE doc.valid_from <= date("2025-07-15")
    AND (doc.valid_until IS NULL OR doc.valid_until >= date("2025-07-15"))
  MATCH (chunk:Chunk)-[:PART_OF]->(doc)
  RETURN doc.document_id, doc.title, doc.valid_from, doc.valid_until,
         collect(chunk.content) AS conteudo
  ORDER BY doc.valid_from DESC
  LIMIT 1;

  QUERY 4 — ANALISE DE IMPACTO:

  // "Se o modulo de Boleto mudar, o que e impactado?"
  MATCH (mod:Module {module_id: "MOD-BOLETO"})

  // Modulos que dependem de Boleto
  OPTIONAL MATCH (dependent:Module)-[:DEPENDS_ON]->(mod)

  // Documentos do modulo
  OPTIONAL MATCH (doc:Document)-[:BELONGS_TO]->(mod)

  // ADRs que decidem sobre o sistema do modulo
  OPTIONAL MATCH (mod)-[:BELONGS_TO]->(sys:System)<-[:DECIDES]-(adr:ADR)

  // Tasks que implementam o modulo
  OPTIONAL MATCH (task:Task)-[:IMPLEMENTS]->(mod)

  RETURN mod.name,
         collect(DISTINCT dependent.name) AS modulos_dependentes,
         collect(DISTINCT doc.title) AS documentos,
         collect(DISTINCT adr.adr_id) AS adrs_relacionadas,
         collect(DISTINCT {task: task.task_id, status: task.status}) AS tasks;

  QUERY 5 — GRAFO DE CONHECIMENTO COMPLETO DE UM SISTEMA:

  // Visualizar todo o grafo de um sistema
  MATCH (sys:System {system_id: "SYS-COBRANCA"})
  OPTIONAL MATCH (mod:Module)-[:BELONGS_TO]->(sys)
  OPTIONAL MATCH (doc:Document)-[:BELONGS_TO]->(mod)
  OPTIONAL MATCH (doc)-[:OWNED_BY]->(owner:Owner)
  OPTIONAL MATCH (owner)-[:MEMBER_OF]->(team:Team)
  OPTIONAL MATCH (adr:ADR)-[:DECIDES]->(sys)
  RETURN sys, mod, doc, owner, team, adr;

================================================================================
  ❌ 3. ALTERNATIVAS DESCARTADAS
================================================================================

  3.1. MODELO RELACIONAL (SQL) COM EXTENSAO VETORIAL
  ---------------------------------------------------------------------------

  - Descricao: usar PostgreSQL com pgvector para armazenar vetores e
    tabelas relacionais para metadados, sem grafo.
  - Prós: SQL e amplamente conhecido; PostgreSQL e robusto e maduro;
    pgvector suporta busca vetorial; menor curva de aprendizado.
  - Contras: NAO suporta traversal de grafo eficiente. Queries como
    "partindo deste chunk, navegar para documento, depois para modulo,
    depois para sistema, depois para ADRs que decidem sobre o sistema"
    requerem 4+ JOINs em SQL, com performance degradada. Em grafo, e
    um MATCH com 4 hops em milissegundos.
  - Motivo da rejeicao: a expansao de contexto via grafo e o
    diferencial do GraphRAG. Sem grafo nativo, o projeto seria apenas
    um RAG convencional com metadados. A perda de capacidade de
    traversal nao compensa a familiaridade com SQL.

  3.2. DOIS BANCOS SEPARADOS (SQL + Grafo)
  ---------------------------------------------------------------------------

  - Descricao: PostgreSQL + pgvector para busca vetorial e um banco
    de grafo separado para relacoes.
  - Prós: cada banco faz o que faz melhor; PostgreSQL para vetores e
    filtros, grafo para traversal.
  - Contras: duplicacao de dados; sincronizacao entre bancos; latencia
    de rede entre dois servicos; complexidade operacional dobrada;
    transacoes distribuidas (ou eventual consistency).
  - Motivo da rejeicao: a complexidade operacional nao se justifica.
    Bases vetoriais modernas suportam vector index + graph traversal +
    full-text na mesma engine, eliminando a necessidade de dois bancos.

  3.3. MODELO SEM DOCUMENTFAMILY (versionamento apenas por SUPERSEDES)
  ---------------------------------------------------------------------------

  - Descricao: rastrear versoes apenas pela cadeia de SUPERSEDES, sem
    o no DocumentFamily.
  - Prós: modelo mais simples; menos nos; menos relacoes.
  - Contras: para encontrar "todas as versoes da Politica de Privacidade",
    seria necessario navegar a cadeia inteira (O(n) hops). Com
    DocumentFamily, e uma unica query (todos os Documents com VERSION_OF
    para a mesma family). Alem disso, sem DocumentFamily, nao ha lugar
    para armazenar o current_version_id (teria que procurar o doc com
    valid_until IS NULL, que e ambiguo se houver erro de dados).
  - Motivo da rejeicao: a eficiencia de query e a clareza do modelo
    compensam a complexidade adicional de mais um tipo de no.

  3.4. EMBEDDING NO DOCUMENTO INTEIRO (sem chunks)
  ---------------------------------------------------------------------------

  - Descricao: gerar um unico embedding por documento, sem chunking.
  - Prós: modelo mais simples; menos nos; embedding captura o contexto
    completo do documento.
  - Contras: documentos longos (>2000 tokens) geram embeddings diluidos
    que nao representam bem nenhum topico especifico. A janela de contexto
    do LLM seria desperdicada com documentos inteiros quando so um trecho
    e relevante. Precisao da busca cai drasticamente.
  - Motivo da rejeicao: a precisao da busca e a eficiencia do uso de
    contexto do LLM exigem chunking. Todo sistema RAG de producao usa
    chunks, nao documentos inteiros.

  3.5. CHUNKING POR TAMANHO FIXO (ex: 512 tokens sempre)
  ---------------------------------------------------------------------------

  - Descricao: ignorar a estrutura do documento e chunkar por tamanho
    fixo (ex: cada 512 tokens = 1 chunk).
  - Prós: implementacao trivial; sem necessidade de parser de headings;
    chunks uniformes.
  - Contras: corta no meio de frases e conceitos. Um chunk pode comecar
    no meio de uma explicacao e terminar no meio de outra. O embedding
    resultante e ruidoso. Heading path se perde.
  - Motivo da rejeicao: chunking semantico (por heading) preserva a
    coerencia do trecho e gera embeddings mais precisos. A implementacao
    e mais complexa, mas a melhoria de qualidade e mensuravel (10-20%
    maior precision@10 em benchmarks internos).

================================================================================
  ⚖️ 4. CONSEQUÊNCIAS
================================================================================

  4.1. POSITIVAS
  ---------------------------------------------------------------------------

  a) GRAPHRAG COMPLETO: o modelo suporta busca semantica (vetorial) E
     expansao de contexto (grafo) na mesma engine, viabilizando
     respostas mais ricas e contextualizadas.

  b) GOVERNANCA NATIVA: cada documento tem owner, team, status,
     confidentiality como propriedades do no, permitindo filtros
     gerenciais e de seguranca diretamente no grafo.

  c) TEMPORALIDADE: DocumentFamily + SUPERSEDES + valid_from/valid_until
     permitem consultas historicas ("como era X em data Y?").

  d) ANALISE DE IMPACTO: relacoes DEPENDS_ON, IMPLEMENTS, DECIDES
     permitem responder "o que e impactado se X mudar?" com traversal
     de grafo em milissegundos.

  e) DESAMBIGUACAO: GlossaryTerm + USES_TERM permitem expandir
     buscas por sinonimos e incluir definicoes no contexto do LLM.

  f) RASTREABILIDADE COMPLETA: de qualquer chunk, e possivel rastrear
     ate o arquivo original, commit, repositorio, autor e time.

  g) CHUNKING OTIMIZADO: estrategias diferenciadas por tipo de documento
     maximizam a precisao da busca para cada caso de uso.

  4.2. NEGATIVAS / TRADE-OFFS
  ---------------------------------------------------------------------------

  a) COMPLEXIDADE DO MODELO: 10 tipos de no e 13 tipos de relacao
     exigem uma curva de aprendizado significativa da equipe.

  b) DESNORMALIZACAO: metadata_inherited no Chunk duplica dados do
     Document, criando risco de inconsistencia se a atualizacao nao
     for atomica (mitigacao: upsert transacional).

  c) CUSTO DE MANUTENCAO: a evolucao do modelo (adicionar propriedades,
     mudar constraints) requer migracao cuidadosa em base de producao.

  d) OVERHEAD DE INGESTAO: o pipeline precisa criar nos de System,
     Module, Owner, Team, Task e GlossaryTerm alem de Document e
     Chunk, aumentando a complexidade da Fase 4.

  4.3. RISCOS E MITIGACOES
  ---------------------------------------------------------------------------

  RISCO 1: CHUNKS ORFAOS
  - Descricao: na re-ingestao, chunks antigos podem nao ser removidos
    se o pipeline falhar no meio.
  - Mitigacao: cleanup job que detecta chunks sem PART_OF e os remove.
    Query: MATCH (c:Chunk) WHERE NOT (c)-[:PART_OF]->() DELETE c

  RISCO 2: INCONSISTENCIA DE metadata_inherited
  - Descricao: se o Document mudar de confidentiality, os chunks filhos
    podem ficar com o valor antigo.
  - Mitigacao: na atualizacao de Document, SEMPRE propagar para chunks.
    Query: MATCH (d:Document {document_id: $id})<-[:PART_OF]-(c:Chunk)
    SET c.metadata_inherited.confidentiality = d.confidentiality

  RISCO 3: DOCUMENTFAMILY DESATUALIZADA
  - Descricao: current_version_id aponta para versao errada.
  - Mitigacao: na criacao de nova versao (SUPERSEDES), atualizar
    atomicamente o DocumentFamily.current_version_id na mesma transacao.

  RISCO 4: PERFORMANCE DE VECTOR INDEX COM MUITOS FILTROS
  - Descricao: filtros pre-retrieval (confidentiality + system + status)
    podem degradar performance do vector index.
  - Mitigacao: indices compostos nos campos de filtro mais usados.
    Monitorar query plans. Se necessario, criar vector indexes
    separados por confidentiality level.
