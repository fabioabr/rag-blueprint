================================================================================
  SUMARIO
================================================================================

  Este ADR e a DECISAO MACRO que determina como o projeto RAG Corporativo
  escolhe entre servicos de nuvem e infraestrutura local (on-premise) para
  cada componente do pipeline. Trata-se da decisao mais cascateante do
  projeto: a partir dela, 4 escolhas pendentes sao resolvidas de uma vez
  (modelo de embedding, infraestrutura da base vetorial, LLM de geracao,
  modelo de reranking).

  A premissa central e: a CLASSIFICACAO DE CONFIDENCIALIDADE do documento
  (definida no front matter) determina qual trilha cada informacao percorre.
  Nao existe uma unica resposta — existem DUAS TRILHAS validas que o projeto
  suporta simultaneamente.

================================================================================
  🧠 1. CONTEXTO
================================================================================

  1.1. O PROBLEMA DA SOBERANIA
  ---------------------------------------------------------------------------

  Muitas organizacoes NAO PODEM enviar sua base de conhecimento para servicos
  externos. As razoes sao multiplas e frequentemente cumulativas:

  a) POLITICA DE SEGURANCA INTERNA
     - Empresas com politica de seguranca da informacao que classifica
       documentos internos como restritos proibem o envio de dados para
       APIs externas, independentemente de termos de uso do fornecedor.
     - Exemplo: uma empresa que classifica atas de reuniao de diretoria
       como "confidencial" nao pode enviar esses textos para a API da
       OpenAI gerar embeddings, mesmo que a OpenAI garanta nao usar
       para treinamento.

  b) REGULACAO SETORIAL
     - BACEN (Banco Central): resolucoes sobre sigilo bancario impedem que
       dados de clientes trafeguem por infraestrutura nao homologada.
     - CVM (Comissao de Valores Mobiliarios): informacoes privilegiadas
       de companhias abertas nao podem ser acessadas por terceiros.
     - LGPD (Lei Geral de Protecao de Dados): dados pessoais de titulares
       brasileiros devem ter consentimento especifico para transferencia
       internacional. Uma base de conhecimento corporativa que contenha
       nomes de funcionarios, CPFs em documentos, enderecos, etc., esta
       sujeita a LGPD.
     - SOX (Sarbanes-Oxley): empresas listadas em bolsas americanas tem
       requisitos adicionais de controle de informacao financeira.
     - HIPAA (para operacoes em saude nos EUA): dados de saude tem
       restricoes severas.

  c) CLAUSULAS CONTRATUAIS DE CONFIDENCIALIDADE
     - Contratos com clientes frequentemente incluem NDAs que proibem o
       compartilhamento de informacoes tecnicas com terceiros. Se a base
       de conhecimento contiver especificacoes de clientes, ela nao pode
       ir para APIs externas.
     - Contratos governamentais (em especial Defesa) exigem infraestrutura
       certificada e isolada.

  d) RISCO REPUTACIONAL
     - Mesmo sem violacao legal, o vazamento de informacao via API de
       terceiro gera dano reputacional. "Empresa X enviou dados para IA
       externa" e uma manchete que nenhum C-level quer ver.

  e) LATENCIA E DISPONIBILIDADE
     - Dependencia de API externa significa que o RAG para de funcionar
       quando o servico externo cai ou muda precos/limites.
     - Para ambientes criticos (sala de operacoes, suporte 24/7), a
       disponibilidade local pode ser requisito.

  1.2. POR QUE ESTA DECISAO E FUNDACIONAL
  ---------------------------------------------------------------------------

  O pipeline definido na ADR-001 tem 4 fases:
    Fase 1 (Selecao) -> Fase 2 (.beta.md) -> Fase 3 (.md final) -> Fase 4 (Base Vetorial)

  Nas Fases 3 e 4, o pipeline precisa de:
  - MODELO DE EMBEDDING para gerar vetores dos chunks
  - BASE VETORIAL para armazenar grafos + vetores + metadados
  - LLM para geracao de respostas a partir do contexto recuperado
  - MODELO DE RERANKING para reordenar resultados da busca hibrida

  Cada um desses componentes pode ser cloud OU local. A decisao sobre
  soberania de dados resolve todos de uma vez, porque o criterio e o mesmo:
  O DADO PODE SAIR DA INFRAESTRUTURA CONTROLADA PELA ORGANIZACAO?

  Se SIM -> Trilha Cloud (mais simples, mais barata para comecar)
  Se NAO -> Trilha On-Premise (mais complexa, mas soberana)

  1.3. CONTEXTO TECNICO ATUAL (Marco 2026)
  ---------------------------------------------------------------------------

  O ecossistema de modelos open-source atingiu maturidade suficiente para
  viabilizar a Trilha On-Premise com qualidade competitiva:

  - Embeddings: BGE-M3 (BAAI) alcanca 95%+ da qualidade do OpenAI em
    benchmarks multilingues, rodando em GPU consumer (8GB VRAM)
  - LLMs: Llama 3.1 70B e Qwen 2.5 72B competem com modelos proprietarios
    em tarefas de RAG (resposta com contexto fornecido)
  - Reranking: BGE-Reranker-v2-m3 atinge resultados comparaveis ao Cohere
    Rerank v3 em cenarios de re-ranking com passagens em portugues
  - Infraestrutura: bases vetoriais com suporte a grafo (como a usada
    neste projeto) possuem edicoes community com funcionalidades suficientes

  Isso NAO era verdade 18 meses atras. A viabilidade da Trilha On-Premise
  e uma janela recente que torna esta ADR possivel.

  1.4. ALINHAMENTO COM OS 8 PILARES DA ADR-001
  ---------------------------------------------------------------------------

  A ADR-001 define 8 pilares norteadores do pipeline. A decisao de soberania
  de dados impacta cada pilar da seguinte forma:

    Pilar   Descricao                         Impacto da ADR-002
    -----   ---------                         ------------------
    A       Segregacao de responsabilidades    Trilha A: operada pelo time de
                                              Engenharia de Dados (APIs cloud).
                                              Trilha B: operada pelo time de
                                              Infraestrutura/Ops (GPU, Docker).
                                              Ambas: pipeline e o mesmo, mudam
                                              apenas os adaptadores de infra.
    B       Desacoplamento de etapas           Ambas as trilhas usam interfaces
                                              abstratas (embedding, LLM, reranker).
                                              Trocar componente em uma trilha NAO
                                              impacta a outra.
    C       Metodo garantidor de qualidade     Gates de qualidade (QA score >= 90%)
                                              sao identicos em ambas as trilhas.
                                              A promocao .beta.md -> .md e agnostica
                                              a infraestrutura.
    D       Observabilidade e Governanca       Cada trilha possui metricas proprias
                                              (ver secao 5.4 — Observabilidade).
                                              Metricas de custo apenas na Trilha A.
                                              Metricas de GPU/hardware na Trilha B.
    E       Clareza da informacao              O campo confidentiality no front matter
                                              torna explicita a trilha de cada doc.
                                              Nao ha ambiguidade: o metadado decide.
    F       Versionamento                      Git controla ambas as trilhas. O
                                              embedding_model e registrado no chunk
                                              para saber qual modelo gerou o vetor.
    G       Rastreabilidade de origem          Ambas as trilhas registram no chunk:
                                              embedding_model, embedding_date,
                                              trilha (A ou B), versao do modelo.
    H       Reprodutibilidade                  Trilha A: deterministica (mesma API,
                                              mesmo resultado). Trilha B: modelos
                                              quantizados (Q4) podem ter variacao
                                              minima — mitigado fixando seed e versao
                                              exata do modelo.

  2.1. REGRA DE ROTEAMENTO
  ---------------------------------------------------------------------------

  O front matter de cada documento possui o campo "confidentiality".
  Conforme ADR-001 (secao 2.5), este campo esta presente DESDE o .beta.md
  (Fase 2) — excecao ao principio de "governanca so no .md final" — para
  permitir o roteamento pela trilha correta e garantir que dados sensiveis
  nao sejam processados por APIs cloud em nenhuma etapa do pipeline.

  Os 4 niveis de confidencialidade e suas trilhas:

  +------------------+--------------------+-----------------------------------+
  | Nivel            | Trilha Permitida   | Descricao                         |
  +------------------+--------------------+-----------------------------------+
  | public           | A (Cloud)          | Informacao publica, pode ser      |
  |                  |                    | processada em qualquer lugar.     |
  +------------------+--------------------+-----------------------------------+
  | internal         | A (Cloud)          | Uso interno, sem restricao legal  |
  |                  |                    | de envio externo. Cloud permitido.|
  +------------------+--------------------+-----------------------------------+
  | restricted       | B (On-Premise)     | Dados com restricao regulatoria   |
  |                  |                    | ou contratual. NAO pode sair.     |
  +------------------+--------------------+-----------------------------------+
  | confidential     | B (On-Premise)     | Altamente confidencial. Somente   |
  |                  |                    | infraestrutura isolada.           |
  +------------------+--------------------+-----------------------------------+

  REGRA: o pipeline de ingestao (Fase 4) VERIFICA o campo confidentiality
  ANTES de escolher qual modelo de embedding usar, qual instancia da base
  vetorial receber o dado, e qual LLM pode gerar respostas sobre ele.

  Essa verificacao e PRE-RETRIEVAL (antes da busca vetorial), nunca
  pos-retrieval. O dado restrito NUNCA chega a um componente cloud.

  2.2. COMPONENTES POR TRILHA — DETALHAMENTO
  ---------------------------------------------------------------------------

  Para cada componente do pipeline, ha uma opcao recomendada por trilha.
  A seguir, o detalhamento tecnico de cada escolha.

  -------------------------------------------------------------------------
  2.2.1. MODELO DE EMBEDDING
  -------------------------------------------------------------------------

  FUNCAO: Transformar chunks de texto em vetores numericos de alta dimensao
  que capturam o significado semantico. E o componente mais critico para
  qualidade de busca — um embedding ruim invalida todo o RAG.

  TRILHA A (Cloud) — OpenAI text-embedding-3-small
  .........................................................................

  - Dimensao: 1536 (padrao) ou ajustavel (512, 1024, 3072)
  - Custo: US$ 0,02 por 1 milhao de tokens (~750 mil palavras em pt-BR)
  - Qualidade: referencia de mercado em benchmarks MTEB multilingues
  - Latencia: ~100ms para batch de 100 chunks (via API)
  - Suporte a portugues: excelente (treinado em corpus multilingue massivo)
  - Vantagem: zero infraestrutura, alta qualidade, custo previsivel
  - Desvantagem: dados saem da sua rede; dependencia de disponibilidade
    da API; OpenAI pode mudar precos ou descontinuar o modelo
  - Exemplo de custo: base de 10.000 documentos com media de 5 chunks
    cada, 500 tokens por chunk = 25M tokens = US$ 0,50 total para
    indexacao completa. Custo marginal quase zero.

  POR QUE ESTA ESCOLHA: melhor relacao custo x qualidade x simplicidade
  para dados que PODEM sair. O custo e tao baixo que nao justifica
  manter infraestrutura propria para dados nao-restritos. A qualidade
  do embedding impacta diretamente a precisao do RAG, e este modelo
  consistentemente lidera benchmarks.

  ALTERNATIVAS CLOUD CONSIDERADAS:
  - Cohere embed-multilingual-v3.0: qualidade comparavel, custo similar,
    mas menor ecossistema e tooling. Seria segunda opcao.
  - Voyage AI voyage-multilingual-2: excelente qualidade, especialmente
    para code retrieval. Opcao valida para bases com muito codigo.
  - Google text-embedding-004: qualidade competitiva, mas menor track
    record e documentacao para RAG em portugues.

  TRILHA B (On-Premise) — BGE-M3 (BAAI)
  .........................................................................

  - Dimensao: 1024 (padrao)
  - Custo: hardware (GPU com 8GB+ VRAM) + energia. Sem custo por token.
  - Qualidade: 95%+ da qualidade do OpenAI em benchmarks multilingues.
    Em benchmarks especificos para portugues, a diferenca pode ser
    menor ainda (~97%).
  - Latencia: ~200ms para batch de 100 chunks em RTX 3070
  - Suporte a portugues: muito bom (treinado em 100+ linguas)
  - Modo hibrido nativo: BGE-M3 gera TRES tipos de representacao
    simultaneamente — dense, sparse (lexical) e colbert — permitindo
    busca hibrida (semantica + keyword) sem modelo adicional.
  - Vantagem: soberania total; sem custo recorrente; modo hibrido nativo;
    modelo open-source (MIT License)
  - Desvantagem: requer GPU dedicada; requer gerenciamento de
    infraestrutura; atualizacoes de modelo sao manuais; qualidade
    ~5% inferior ao melhor modelo cloud

  REQUISITOS DE HARDWARE:
  - Minimo: 1x GPU com 8GB VRAM (RTX 3060, RTX 4060, T4)
  - Recomendado: 1x GPU com 16GB+ VRAM (RTX 4070 Ti, A10G, L4)
  - CPU-only: possivel com ONNX Runtime, mas latencia 10x maior
    (~2 segundos por batch). Viavel para indexacao batch, nao para
    busca em tempo real.

  POR QUE ESTA ESCOLHA: unico modelo open-source que combina qualidade
  multilingue competitiva com modo hibrido nativo (dense + sparse +
  colbert). Para dados que NAO PODEM sair, e a melhor opcao disponivel
  em marco/2026. O fato de rodar em GPU consumer (8GB) torna a barreira
  de entrada baixa.

  ALTERNATIVAS ON-PREM CONSIDERADAS:
  - E5-Mistral-7B-Instruct: excelente qualidade, mas requer GPU com
    24GB+ VRAM. Custo de hardware proibitivo para MVP.
  - Nomic-Embed-Text-v1.5: boa qualidade, eficiente em recursos, mas
    desempenho inferior em portugues especificamente. Bom para ingles.
  - Multilingual-E5-Large: qualidade razoavel, mas geracoes anteriores
    ao BGE-M3 em benchmarks multilingues.

  -------------------------------------------------------------------------
  2.2.2. INFRAESTRUTURA DA BASE VETORIAL
  -------------------------------------------------------------------------

  FUNCAO: Armazenar o grafo de conhecimento (nos e relacoes), os vetores
  de embedding, e os metadados. Suportar busca vetorial (semantica),
  busca por grafo (traversal), e filtros por metadados — tudo na mesma
  engine.

  TRILHA A (Cloud) — Base Vetorial Gerenciada (Managed Cloud)
  .........................................................................

  - Tipo: instancia gerenciada pelo provedor (DBaaS)
  - Custo: a partir de ~US$ 65/mes para instancias basicas
  - Escalabilidade: vertical (upgrade de instancia) e horizontal
    (read replicas) gerenciada pelo provedor
  - Backup: automatizado pelo provedor
  - Atualizacoes: gerenciadas, sem downtime (na maioria dos provedores)
  - SLA: 99,9% tipico
  - Vantagem: zero operacao de infraestrutura; backup automatico;
    escalabilidade sob demanda; monitoramento nativo
  - Desvantagem: dados armazenados na infraestrutura do provedor;
    custo mensal fixo; vendor lock-in parcial; latencia de rede

  POR QUE ESTA ESCOLHA: para dados que PODEM sair, a operacao gerenciada
  elimina a necessidade de um DBA ou ops dedicado para a base vetorial.
  O custo e previsivel e a escalabilidade e automatica. A equipe pode
  focar no pipeline e nos modelos, nao na infraestrutura.

  TRILHA B (On-Premise) — Base Vetorial Self-Hosted (Community Edition)
  .........................................................................

  - Tipo: instalacao via Docker / Kubernetes na infraestrutura propria
  - Custo: hardware (servidor com 16GB+ RAM, SSD) + operacao
  - Edicao: Community Edition (open source, licenca GPLv3 ou similar)
  - Funcionalidades: grafo completo, vector index (desde versao 5.x+),
    full-text index, constraints, APOC procedures
  - Vantagem: soberania total; sem custo de licenca; controle de
    versao e atualizacao; isolamento de rede
  - Desvantagem: requer ops dedicado; backup manual; monitoramento
    manual; sem suporte enterprise; escalabilidade horizontal limitada
    na edicao community (single-instance, sem clustering nativo)

  REQUISITOS DE HARDWARE:
  - Minimo: 4 vCPU, 16GB RAM, 100GB SSD
  - Recomendado: 8 vCPU, 32GB RAM, 500GB NVMe
  - Para bases >1M chunks: considerar Enterprise Edition ou arquitetura
    com read replicas

  POR QUE ESTA ESCOLHA: a Community Edition oferece TODAS as
  funcionalidades necessarias para o MVP e fases subsequentes do RAG
  (vector index, graph traversal, full-text, constraints). A limitacao
  de single-instance nao e problema para bases corporativas tipicas
  (<500K chunks). A instalacao via Docker e trivial.

  DOCKER COMPOSE DE REFERENCIA:

  NOTA: as portas e variaveis de ambiente abaixo sao EXEMPLOS GENERICOS.
  Ajustar conforme a base vetorial escolhida pela organizacao.

  # Exemplo simplificado — ajustar para producao
  services:
    vectordb:
      image: <imagem-da-base-vetorial>:community-latest
      ports:
        - "<PORTA_HTTP>:<PORTA_HTTP>"      # interface web / HTTP API
        - "<PORTA_PROTOCOLO>:<PORTA_PROTOCOLO>"  # protocolo nativo de conexao
      volumes:
        - ./data:/data
        - ./logs:/logs
      environment:
        - AUTH=usuario/senha
        - MEMORY_HEAP_MAX=8G
        - MEMORY_CACHE=4G
      restart: unless-stopped

  -------------------------------------------------------------------------
  2.2.3. LLM (MODELO DE LINGUAGEM PARA GERACAO)
  -------------------------------------------------------------------------

  FUNCAO: Gerar respostas em linguagem natural a partir do contexto
  recuperado pelo RAG. O LLM recebe os chunks relevantes + a pergunta
  do usuario e produz uma resposta coerente, citando fontes.

  IMPORTANTE: O LLM NAO e usado apenas na Fase 4 (retrieval). Ele tambem
  e usado na Fase 2 (mineracao de .beta.md a partir de fontes brutas,
  conforme ADR-001 secao 2.1) e na Fase 3 (geracao do .md final com
  front matter rico). Portanto, a escolha do LLM impacta TODO o pipeline.
  A regra de roteamento por confidentiality se aplica DESDE a Fase 2:
  um documento restricted/confidential DEVE usar LLM local mesmo na
  mineracao do .beta.md.

  TRILHA A (Cloud) — Claude (Anthropic)
  .........................................................................

  - Modelo recomendado: Claude Sonnet (para tarefas de RAG) ou Claude
    Opus (para mineracao complexa na Fase 2)
  - Custo: variavel por token (input + output). Consultar precos vigentes.
  - Janela de contexto: 200K tokens (permite injecao de muitos chunks)
  - Qualidade em pt-BR: excelente, com compreensao de nuances e
    capacidade de seguir instrucoes complexas de formatacao
  - Vantagem: melhor qualidade de seguimento de instrucoes (instruction
    following) do mercado; janela de contexto generosa; respostas bem
    estruturadas; excelente para geracao de documentos tecnicos
  - Desvantagem: dados enviados para API externa; custo por token;
    dependencia de disponibilidade

  POR QUE ESTA ESCOLHA: para o pipeline completo (Fases 2, 3 e 4),
  a capacidade de seguir instrucoes complexas e decisiva. O LLM precisa
  gerar .beta.md com front matter estruturado, respeitar blocos LOCKED,
  formatar documentos segundo templates especificos. Claude demonstra
  superioridade consistente nesse tipo de tarefa.

  ALTERNATIVA CLOUD: OpenAI GPT-4o — qualidade similar para muitas
  tarefas, mas instruction following ligeiramente inferior em tarefas
  de formatacao estruturada. Boa segunda opcao.

  TRILHA B (On-Premise) — Llama 3.1 70B ou Qwen 2.5 72B
  .........................................................................

  OPCAO B1 — Llama 3.1 70B (Meta)
  - Licenca: Llama Community License (uso comercial ate 700M MAU)
  - Qualidade: ~85-90% do Claude em tarefas de RAG com contexto fornecido
  - Requisitos: 2x GPU com 24GB VRAM (ex: 2x RTX 4090) para inferencia
    quantizada (Q4). Ou 1x GPU com 80GB (A100/H100) para fp16.
  - Hosting: Ollama (simples, single-node) ou vLLM (producao, batching)
  - Vantagem: modelo mais testado e documentado da comunidade open-source;
    ecossistema enorme; suporte a portugues bom
  - Desvantagem: 70B parametros exige GPU significativa; instrucao
    following inferior a modelos proprietarios; pode "alucinar" mais
    em tarefas de formatacao estruturada

  OPCAO B2 — Qwen 2.5 72B (Alibaba)
  - Licenca: Apache 2.0 (uso comercial irrestrito)
  - Qualidade: comparavel ao Llama 3.1 70B, com vantagem em raciocinio
    logico e tarefas multilingues (treinado com forte enfase em
    multilingualismo)
  - Requisitos: similares ao Llama 3.1 70B
  - Vantagem: licenca mais permissiva (Apache 2.0); excelente em
    multilingue; forte em raciocinio e matematica
  - Desvantagem: ecossistema menor que Llama; menos documentacao em
    portugues especificamente; comunidade menor

  RECOMENDACAO: para ambiente on-premise, iniciar com Llama 3.1 70B
  (maior ecossistema e documentacao) e avaliar migracao para Qwen 2.5
  se o desempenho em portugues nao for satisfatorio.

  CONFIGURACAO DE REFERENCIA COM OLLAMA:

  # Instalar Ollama
  curl -fsSL https://ollama.ai/install.sh | sh

  # Baixar modelo (download unico de ~40GB para versao Q4)
  ollama pull llama3.1:70b-instruct-q4_K_M

  # Servir como API compativel com OpenAI
  ollama serve
  # Endpoint: http://localhost:11434/v1/chat/completions

  CONFIGURACAO DE REFERENCIA COM vLLM (PRODUCAO):

  # Docker com vLLM para batching e alta concorrencia
  docker run --gpus all -p 8000:8000 \
    vllm/vllm-openai:latest \
    --model meta-llama/Llama-3.1-70B-Instruct \
    --quantization awq \
    --max-model-len 32768 \
    --tensor-parallel-size 2

  POR QUE ESTAS ESCOLHAS: Llama e Qwen sao os unicos modelos open-source
  de 70B+ parametros com qualidade suficiente para as tres funcoes do
  pipeline (mineracao, geracao de .md, respostas RAG). Modelos menores
  (7B, 13B) falham em tarefas de formatacao estruturada e instruction
  following. O custo de GPU para 70B e justificavel pela soberania total.

  -------------------------------------------------------------------------
  2.2.4. MODELO DE RERANKING
  -------------------------------------------------------------------------

  FUNCAO: Reordenar os resultados da busca vetorial para melhorar a
  precisao. O reranker recebe a query original + N candidatos retornados
  pela busca e os reordena por relevancia, usando um modelo cross-encoder
  (mais preciso que o bi-encoder usado no embedding, porem mais lento).

  POR QUE RERANKING E NECESSARIO: a busca vetorial (bi-encoder) e rapida
  mas imprecisa. Ela compara query e documento em espacos separados. O
  reranker compara query e documento JUNTOS, capturando interacoes
  semanticas sutis. Em benchmarks, reranking melhora precision@10 em
  15-30% sobre busca vetorial pura.

  TRILHA A (Cloud) — Cohere Rerank v3
  .........................................................................

  - Custo: US$ 2,00 por 1.000 buscas (cada busca = 1 query + ate 1000
    documentos para reranking)
  - Qualidade: referencia de mercado para reranking multilingue
  - Latencia: ~150ms para reranking de 50 candidatos
  - Suporte a portugues: excelente
  - Vantagem: melhor qualidade disponivel; API simples (1 endpoint);
    suporta ate 4096 tokens por documento
  - Desvantagem: dados enviados para API; custo por busca (pode escalar
    rapido em ambientes com alta concorrencia)

  ESTIMATIVA DE CUSTO: 1.000 buscas/dia x 30 dias = 30.000 buscas/mes
  = US$ 60/mes. Para um time de 50 usuarios fazendo ~20 perguntas/dia
  cada = ~30.000 buscas/mes = US$ 60/mes. Custo aceitavel.

  POR QUE ESTA ESCOLHA: Cohere Rerank v3 e o reranker com melhor
  desempenho documentado em benchmarks multilingues (BEIR, MIRACL).
  A integracao e trivial (1 chamada de API). O custo e proporcional
  ao uso, sem custo fixo.

  TRILHA B (On-Premise) — BGE-Reranker-v2-m3 (BAAI)
  .........................................................................

  - Custo: hardware (GPU com 4GB+ VRAM) + energia
  - Qualidade: ~90-95% do Cohere Rerank v3 em benchmarks multilingues
  - Latencia: ~100ms para reranking de 50 candidatos em GPU; ~500ms em CPU
  - Suporte a portugues: bom (treinado em 100+ linguas, mesma familia
    do BGE-M3)
  - Vantagem: soberania total; sem custo recorrente; roda na mesma GPU
    do embedding (BGE-M3); modelo leve
  - Desvantagem: qualidade ligeiramente inferior ao Cohere; requer
    infraestrutura de serving; atualizacoes manuais

  SINERGIA COM BGE-M3: como ambos os modelos sao da familia BGE (BAAI),
  eles compartilham tokenizador e sao otimizados para trabalhar juntos.
  O BGE-M3 gera embeddings e o BGE-Reranker-v2-m3 reordena — a mesma
  GPU pode servir ambos (em sequencia, nao simultaneamente).

  POR QUE ESTA ESCOLHA: unico reranker open-source com qualidade
  competitiva em multilingue que roda em GPU consumer. A sinergia com
  BGE-M3 simplifica a stack on-premise (mesma familia de modelos, mesmo
  framework de serving).

  2.4. TRILHA HIBRIDA — O CENARIO MAIS COMUM
  ---------------------------------------------------------------------------

  Na pratica, a maioria das organizacoes usara a TRILHA HIBRIDA:

  - Documentos public e internal -> processados pela Trilha A (Cloud)
  - Documentos restricted e confidential -> processados pela Trilha B

  Isso significa que a organizacao opera DUAS instancias paralelas da
  base vetorial:

  INSTANCIA CLOUD (Trilha A):
  - Hospedada na nuvem (managed)
  - Recebe documents com confidentiality = public ou internal
  - Embeddings gerados pela OpenAI
  - LLM de resposta = Claude
  - Reranking = Cohere

  INSTANCIA ON-PREM (Trilha B):
  - Hospedada na infraestrutura propria
  - Recebe documents com confidentiality = restricted ou confidential
  - Embeddings gerados pelo BGE-M3
  - LLM de resposta = Llama/Qwen local
  - Reranking = BGE-Reranker

  IMPORTANTE — INCOMPATIBILIDADE DE EMBEDDINGS:
  Os vetores gerados pela OpenAI (1536d) e pelo BGE-M3 (1024d) NAO sao
  compativeis. Um chunk indexado com OpenAI so pode ser buscado com
  embedding de query gerado pela OpenAI. Isso reforça a separacao em
  duas instancias.

  Para busca cross-instancia (query que precisa de resultados de ambas),
  o retriever executa a busca em PARALELO nas duas instancias e faz
  merge dos resultados antes do reranking. Isso so e permitido quando o
  usuario tem acesso a ambos os niveis de confidencialidade.

  FLUXO DA BUSCA HIBRIDA:

  Pergunta do usuario
       |
       v
  [Verificar nivel de acesso do usuario]
       |
       +-- acesso = public/internal apenas
       |     |
       |     v
       |   [Busca na instancia CLOUD apenas]
       |     |
       |     v
       |   [Reranking Cohere]
       |     |
       |     v
       |   [Resposta via Claude]
       |
       +-- acesso = todos os niveis
             |
             v
           [Busca PARALELA: Cloud + On-Prem]
             |
             v
           [Merge de resultados]
             |
             v
           [Reranking local (BGE-Reranker)]  <-- usa reranker local
             |                                    porque resultados
             v                                    on-prem NAO PODEM
           [Resposta via LLM local]               ir para Cohere
             |
             (resultado final nao sai da infra)

  NOTA: quando resultados de ambas as instancias sao mesclados, o
  reranking e a geracao de resposta DEVEM usar componentes on-premise,
  pois os chunks restricted/confidential estao presentes. A regra e:
  O COMPONENTE MAIS RESTRITIVO DETERMINA A TRILHA.

  COMPATIBILIDADE DO RERANKER LOCAL COM CHUNKS CLOUD: o BGE-Reranker-v2-m3
  e um cross-encoder — ele recebe o TEXTO da query e o TEXTO do chunk,
  NAO os embeddings. Portanto, funciona corretamente independentemente de
  qual modelo gerou os embeddings originais (OpenAI ou BGE-M3). O reranker
  nao depende do espaco vetorial; ele avalia relevancia textual diretamente.

  2.5. DIMENSIONAMENTO DE CUSTOS
  ---------------------------------------------------------------------------

  CENARIO DE REFERENCIA: empresa com 10.000 documentos, media de 5
  chunks por documento (50.000 chunks total), 50 usuarios, 20 buscas
  por usuario por dia util (~22 dias/mes).

  TRILHA A — CUSTO MENSAL ESTIMADO:
  +-------------------+----------+------------------------------------------+
  | Componente        | Custo    | Calculo                                  |
  +-------------------+----------+------------------------------------------+
  | Embedding (index) | ~$0.50   | 25M tokens x $0.02/1M (unica vez)        |
  | Embedding (query) | ~$2.00   | 22K queries x ~100 tok x $0.02/1M        |
  | Base Vetorial     | ~$65.00  | Instancia basica managed                 |
  | LLM (respostas)   | ~$150.00 | 22K queries x ~2K tok out x preco/tok    |
  | Reranking         | ~$44.00  | 22K buscas x $2/1K                       |
  +-------------------+----------+------------------------------------------+
  | TOTAL MENSAL      | ~$261.50 |                                          |
  +-------------------+----------+------------------------------------------+

  TRILHA B — CUSTO ESTIMADO:
  +-------------------+------------+----------------------------------------+
  | Componente        | Custo      | Calculo                                |
  +-------------------+------------+----------------------------------------+
  | Hardware (GPU)    | ~$3,000    | 2x RTX 4090 (investimento unico)       |
  | Servidor          | ~$2,000    | 64GB RAM, NVMe (investimento unico)    |
  | Energia/mes       | ~$50       | ~500W medio x 24/7                     |
  | Operacao/mes      | ~$500      | Fracao de tempo de ops/infra            |
  | Base Vetorial     | $0         | Community Edition                      |
  +-------------------+------------+----------------------------------------+
  | INVESTIMENTO      | ~$5,000    | Unico                                  |
  | CUSTO MENSAL      | ~$550      | Recorrente                             |
  +-------------------+------------+----------------------------------------+

  BREAK-EVEN: Trilha B fica mais barata que Trilha A apos ~12 meses
  (considerando amortizacao do hardware). Para bases maiores ou mais
  usuarios, o break-even chega mais rapido.

  TRILHA HIBRIDA (A + B) — CUSTO CONSOLIDADO:
  +-------------------+------------+------------------------------------------+
  | Componente        | Custo      | Calculo                                  |
  +-------------------+------------+------------------------------------------+
  | Trilha A (mensal) | ~$261.50   | Embedding + Base Vetorial + LLM +        |
  |                   |            | Reranking (ver tabela acima)             |
  +-------------------+------------+------------------------------------------+
  | Trilha B (invest) | ~$5,000    | Hardware GPU + servidor (unica vez)       |
  | Trilha B (mensal) | ~$550      | Energia + operacao                       |
  +-------------------+------------+------------------------------------------+
  | INVESTIMENTO      | ~$5,000    | Unico (hardware Trilha B)                |
  | TOTAL MENSAL      | ~$811.50   | Trilha A ($261.50) + Trilha B ($550)     |
  +-------------------+------------+------------------------------------------+

  NOTA: nao ha economia de escala significativa no cenario hibrido — os
  custos sao essencialmente aditivos. A Trilha A processa dados
  public/internal; a Trilha B processa dados restricted/confidential.
  A proporcao de documentos em cada nivel determina o custo real.

  POREM: o custo da Trilha B NAO e a razao da escolha. A razao e
  SOBERANIA. Organizacoes que PRECISAM da Trilha B nao tem a opcao
  de usar a Trilha A para dados restritos, independentemente de custo.

================================================================================
  ❌ 3. ALTERNATIVAS DESCARTADAS
================================================================================

  3.1. TRILHA UNICA CLOUD
  ---------------------------------------------------------------------------

  - Descricao: usar somente componentes cloud para todos os niveis de
    confidencialidade, confiando nos termos de uso e contratos dos
    provedores (ex: DPA da OpenAI, contratos enterprise com SLA).
  - Por que descartada: insuficiente para ambientes regulados. Termos de
    uso de API NAO substituem requisitos regulatorios (BACEN, CVM). Mesmo
    com contrato enterprise, o dado trafega por infraestrutura de terceiro.
    Auditorias externas frequentemente questionam: "onde fisicamente estao
    os embeddings dos documentos confidenciais?". "Na AWS us-east-1 da
    OpenAI" nao e uma resposta aceitavel.

  3.2. TRILHA UNICA ON-PREMISE
  ---------------------------------------------------------------------------

  - Descricao: usar somente componentes on-premise para todos os niveis,
    mesmo para dados publicos.
  - Por que descartada: desperadica recursos e qualidade. Para dados
    public/internal, a qualidade dos modelos cloud e superior e o custo
    e menor. Forcar Trilha B para dados que nao precisam de soberania
    e over-engineering. Alem disso, nem toda organizacao tem GPUs
    disponveis — a Trilha A permite comecar o RAG sem investimento em
    hardware.

  3.3. ENCRYPTION-AT-REST + CLOUD
  ---------------------------------------------------------------------------

  - Descricao: enviar dados criptografados para cloud, descriptografar
    apenas no momento do processamento.
  - Por que descartada: tecnicamente inviavel para embeddings. O modelo
    de embedding precisa do TEXTO CLARO para gerar o vetor. Nao ha
    como gerar embedding de texto criptografado e obter resultados
    semanticamente uteis. Tecnicas de "homomorphic encryption" para
    busca vetorial existem em pesquisa academica, mas nao estao em
    producao (latencia 1000x+ maior, precisao degradada).

  3.4. PROVEDOR CLOUD UNICO COM REGIAO DEDICADA
  ---------------------------------------------------------------------------

  - Descricao: contratar regiao dedicada (ex: AWS GovCloud, Azure
    Government) com garantia de isolamento fisico e compliance.
  - Por que descartada: custo proibitivo para a maioria das organizacoes
    (10-50x mais caro que regiao padrao). Disponivel apenas para
    contratos governamentais especificos. Nao resolve o problema
    fundamental: o dado ainda esta na infraestrutura de um terceiro,
    mesmo que fisicamente isolado. Pode ser opcao para clientes
    governamentais em fases futuras, mas nao e a recomendacao geral.

  3.5. MODELOS DE EMBEDDING MENORES ON-PREMISE (7B, distilled)
  ---------------------------------------------------------------------------

  - Descricao: usar modelos de embedding menores (ex: all-MiniLM-L6-v2,
    E5-small) para reduzir requisito de GPU.
  - Por que descartada: qualidade insuficiente para portugues. Modelos
    pequenos (<1B parametros) tem desempenho significativamente inferior
    em linguas nao-inglesas. O BGE-M3, apesar de maior, roda em GPU
    consumer (8GB VRAM) e oferece qualidade multilingue competitiva.
    A economia de GPU nao compensa a perda de qualidade.

  3.6. LLM LOCAL PEQUENO (7B-13B) PARA TODAS AS TAREFAS
  ---------------------------------------------------------------------------

  - Descricao: usar Llama 3.1 8B ou Qwen 2.5 14B no lugar dos modelos
    de 70B+, reduzindo drasticamente o requisito de GPU.
  - Por que descartada: modelos de 7-13B falham consistentemente em tres
    tarefas criticas do pipeline:
    a) Geracao de .beta.md com front matter estruturado (Fase 2) —
       frequentemente omitem campos, erram formatacao YAML
    b) Respeito a blocos LOCKED (Fase 2) — tendem a reescrever blocos
       protegidos
    c) Respostas com citacao de fontes (Fase 4) — tendem a "alucinar"
       informacoes nao presentes no contexto fornecido
    Para a Fase 4 (respostas RAG simples), um modelo de 13B PODE ser
    suficiente como fallback, mas nao e recomendado como unico LLM.

================================================================================
  ⚖️ 4. CONSEQUÊNCIAS
================================================================================

  4.1. POSITIVAS
  ---------------------------------------------------------------------------

  a) COMPLIANCE GARANTIDO: organizacoes reguladas podem adotar o RAG
     sem violar politicas de seguranca, porque a Trilha B garante que
     dados restritos nunca saem da infraestrutura propria.

  b) FLEXIBILIDADE DE ADOCAO: organizacoes sem restricao podem comecar
     pela Trilha A (zero hardware) e migrar para Trilha B conforme
     necessidade. O pipeline e o mesmo — so mudam os componentes.

  c) DECISAO AUTOMATICA: o campo "confidentiality" no front matter
     determina automaticamente a trilha. Nao ha decisao humana no
     momento da ingestao — o pipeline lê o metadado e roteia.

  d) CUSTO OTIMIZADO: dados que podem ir para cloud usam cloud (melhor
     qualidade, menor custo operacional). Dados que precisam ficar
     locais usam local. Nao ha desperdicio.

  e) EVOLUCAO INDEPENDENTE: cada trilha pode evoluir separadamente.
     Se surgir um embedding cloud melhor, troca-se na Trilha A sem
     impactar a Trilha B. Se surgir um LLM open-source melhor,
     troca-se na Trilha B sem impactar a Trilha A.

  f) ALINHAMENTO COM ADR-001: as 4 fases do pipeline funcionam
     identicamente em ambas as trilhas. A soberania e uma preocupacao
     de INFRAESTRUTURA, nao de PROCESSO.

  4.2. NEGATIVAS / TRADE-OFFS
  ---------------------------------------------------------------------------

  a) COMPLEXIDADE OPERACIONAL: manter duas instancias da base vetorial,
     dois conjuntos de modelos, e dois fluxos de serving aumenta a
     complexidade operacional significativamente.

  b) INCONSISTENCIA DE QUALIDADE: a Trilha A (cloud) tera qualidade
     ligeiramente superior a Trilha B (on-premise) em embeddings,
     reranking e respostas do LLM. Usuarios que consultam documentos
     de ambos os niveis podem perceber diferenca.

  c) BUSCA CROSS-INSTANCIA: a busca que precisa cruzar dados das duas
     instancias e mais complexa (merge de resultados de embeddings
     incompativeis) e mais lenta.

  d) CUSTO DE HARDWARE: a Trilha B exige investimento inicial em GPUs
     (~$3-5K) que pode ser barreira para organizacoes menores.

  e) COMPETENCIA DA EQUIPE: a Trilha B exige equipe capaz de operar
     modelos de ML em producao (GPU drivers, CUDA, model serving,
     monitoramento de inferencia). Isso pode exigir contratacao ou
     treinamento.

  4.3. RISCOS E MITIGACOES
  ---------------------------------------------------------------------------

  RISCO 1: CLASSIFICACAO INCORRETA DE CONFIDENCIALIDADE
  - Descricao: documento confidential e classificado como internal por
    erro humano, sendo enviado para componentes cloud.
  - Probabilidade: MEDIA (erro humano e inevitavel)
  - Impacto: ALTO (violacao de politica, possivel violacao regulatoria)
  - Mitigacao: validacao automatica no pipeline de promocao (Fase 3).
    Regras: documentos de determinados dominios (ex: "financeiro",
    "juridico") devem ser no minimo restricted. Documentos com
    determinadas tags (ex: "dados-pessoais", "sigilo-bancario") devem
    ser no minimo confidential. Alerta bloqueante se a regra for violada.

  RISCO 2: DEGRADACAO DE QUALIDADE NA TRILHA B
  - Descricao: modelos open-source evoluem mais devagar que proprietarios,
    e a diferenca de qualidade pode aumentar com o tempo.
  - Probabilidade: BAIXA (tendencia atual e de convergencia, nao divergencia)
  - Impacto: MEDIO (respostas menos precisas para dados restritos)
  - Mitigacao: monitorar benchmarks trimestralmente. Se um novo modelo
    open-source superar o BGE-M3 ou Llama, migrar. O pipeline e agnostico
    ao modelo — basta trocar o endpoint.

  RISCO 3: INDISPONIBILIDADE DA TRILHA A
  - Descricao: API da OpenAI, Anthropic ou Cohere fora do ar impede
    buscas em dados public/internal.
  - Probabilidade: BAIXA (SLAs de 99.9%+)
  - Impacto: MEDIO (buscas em dados nao-restritos ficam indisponiveis)
  - Mitigacao: implementar fallback local para a Trilha A. Em caso de
    indisponibilidade cloud, rotear para BGE-M3/Llama local com
    qualidade reduzida mas funcional.
  - NOTA: este fallback so e possivel se a organizacao JA possui a
    infraestrutura da Trilha B. Organizacoes que operam SOMENTE a
    Trilha A nao tem fallback local — devem aceitar a indisponibilidade
    ou considerar provisionar infraestrutura minima on-premise para
    contingencia (ver Fase 3 da implementacao, secao 5.2).

  RISCO 4: CUSTO CLOUD ESCALANDO
  - Descricao: aumento de uso (mais usuarios, mais documentos) faz o
    custo cloud escalar alem do orcamento.
  - Probabilidade: MEDIA (crescimento e esperado)
  - Impacto: BAIXO (custo e incremental, nao explosivo)
  - Mitigacao: monitorar custo mensal. Se ultrapassar threshold definido,
    considerar migrar dados internal para Trilha B, mantendo apenas
    public na Trilha A.

  RISCO 5: VENDOR LOCK-IN
  - Descricao: dependencia de OpenAI, Anthropic ou Cohere pode ser
    problematica se mudarem precos, termos ou descontinuarem modelos.
  - Probabilidade: MEDIA (historico de mudancas de precos na industria)
  - Impacto: MEDIO (requer migracao de modelo e re-indexacao)
  - Mitigacao: a arquitetura usa abstracoes (interfaces de embedding,
    LLM e reranking). Trocar o provedor cloud NAO exige mudar o
    pipeline, apenas o adaptador. Re-indexacao de embeddings e
    automatizada pelo pipeline da Fase 4.

  🚀 5. IMPLEMENTAÇÃO
================================================================================

  5.1. MATRIZ DE DECISAO — COMO ESCOLHER SUA TRILHA
  ---------------------------------------------------------------------------

  Use esta matriz para determinar qual configuracao sua organizacao
  deve adotar:

  PERGUNTA 1: A organizacao possui ALGUM documento classificado como
  restricted ou confidential?
  - NAO -> Use Trilha A (Cloud) para tudo. Fim.
  - SIM -> Continue.

  PERGUNTA 2: TODOS os documentos sao restricted ou confidential?
  - SIM -> Use Trilha B (On-Premise) para tudo. Fim.
  - NAO -> Continue.

  PERGUNTA 3: A organizacao aceita operar duas instancias da base
  vetorial simultaneamente?
  - SIM -> Use Trilha Hibrida (A + B). Fim.
  - NAO -> Use Trilha B (On-Premise) para tudo (mais seguro,
           sacrifica qualidade para dados nao-restritos). Fim.

  PERGUNTA 4 (para Trilha Hibrida): a organizacao tem GPUs disponiveis
  ou orcamento para adquirir?
  - SIM -> Implementar Trilha B com GPU. Fim.
  - NAO -> Iniciar com Trilha A apenas para dados public/internal.
           Dados restricted/confidential NAO sao ingeridos ate ter
           infraestrutura. Planejar aquisicao de hardware.

  5.2. ORDEM DE IMPLEMENTACAO
  ---------------------------------------------------------------------------

  FASE 1 — MVP (Trilha A apenas, dados public/internal)
  - Configurar base vetorial managed (cloud)
  - Integrar OpenAI embedding API
  - Integrar Claude API para respostas
  - Ingerir documentos public e internal
  - Validar pipeline end-to-end
  - SEM reranking nesta fase (simplificar)
  - DOD: busca vetorial retorna resultados corretos para 10 queries de teste;
    LLM gera respostas com citacao de fonte; checklist Trilha A completo

  FASE 2 — RERANKING + QUALIDADE (Trilha A)
  - Adicionar Cohere Rerank v3
  - Medir melhoria de precisao (precision@10, MRR)
  - Ajustar parametros (top-K retrieval, top-N reranking)
  - DOD: precision@10 melhora >= 15% com reranking; metricas documentadas

  FASE 3 — TRILHA B (dados restricted/confidential)
  - Provisionar hardware (GPU + servidor)
  - Instalar base vetorial Community Edition via Docker
  - Configurar BGE-M3 para embeddings
  - Configurar Llama 3.1 70B ou Qwen 2.5 72B via Ollama/vLLM
  - Ingerir documentos restricted e confidential
  - Validar pipeline end-to-end na Trilha B
  - Adicionar BGE-Reranker-v2-m3
  - DOD: checklist Trilha B completo; nenhum dado restricted/confidential
    trafega por rede externa (validado por logs); latencia < 2s end-to-end

  FASE 4 — TRILHA HIBRIDA (busca cross-instancia)
  - Implementar retriever que busca em ambas as instancias
  - Implementar merge de resultados
  - Implementar controle de acesso por usuario
  - Testar cenarios de busca cross-confidentiality
  - DOD: busca cross-instancia retorna resultados mesclados corretamente;
    controle de acesso impede usuario sem permissao de ver dados restricted;
    reranking local funciona com chunks de ambas as trilhas

  [ ] API key da Anthropic (Claude) configurada e testada
  [ ] API key da Cohere configurada e testada
  [ ] Base vetorial managed provisionada e acessivel
  [ ] Pipeline de ingestao filtra confidentiality = public/internal

  5.4. OBSERVABILIDADE POR TRILHA (Pilar D — ADR-001)
  ---------------------------------------------------------------------------

  Cada trilha deve monitorar metricas especificas. O objetivo e detectar
  degradacao de qualidade, indisponibilidade e custos fora do esperado.

  METRICAS COMUNS (ambas as trilhas):
  - Latencia end-to-end (query -> resposta): p50, p95, p99
  - Taxa de erro de ingestao (chunks rejeitados / total)
  - Volume de chunks ingeridos por dia/semana
  - Precisao da busca (precision@10, MRR) — amostral, via golden set
  - Consistencia: total de chunks na base vs. total de documentos no repo

  METRICAS ESPECIFICAS — TRILHA A (Cloud):
  - Custo acumulado por mes (embedding + LLM + reranking + base vetorial)
  - Latencia de API por provedor (OpenAI, Anthropic, Cohere)
  - Taxa de rate-limiting / throttling
  - Disponibilidade percebida (% de requests com sucesso)

  METRICAS ESPECIFICAS — TRILHA B (On-Premise):
  - Uso de GPU (VRAM, utilizacao %)
  - Temperatura da GPU (alerta se > 85C)
  - Uso de RAM/CPU do servidor da base vetorial
  - Espaco em disco (alerta se < 20% livre)
  - Throughput de inferencia (tokens/segundo para LLM, chunks/segundo para embedding)

  ARMAZENAMENTO DE METRICAS: pasta process/metrics/ no rag-workspace
  (conforme estrutura da ADR-001). Formato sugerido: JSON lines, com
  dashboards gerados sob demanda.

================================================================================

  - TRILHA: conjunto de componentes (embedding, base vetorial, LLM,
    reranker) que processam dados de um nivel de confidencialidade.
  - SOBERANIA DE DADOS: principio de que a organizacao mantém controle
    total sobre onde seus dados sao armazenados e processados.
  - BI-ENCODER: modelo que codifica query e documento separadamente
    (rapido, menos preciso). Usado para embedding.
  - CROSS-ENCODER: modelo que codifica query e documento juntos (lento,
    mais preciso). Usado para reranking.
  - VRAM: memoria da GPU dedicada ao processamento de modelos de IA.
  - QUANTIZACAO (Q4, AWQ): tecnica que reduz o tamanho de modelos de IA
    comprimindo pesos de 16 bits para 4 bits, com perda minima de
    qualidade. Permite rodar modelos grandes em GPUs menores.
  - DBaaS: Database as a Service — banco de dados gerenciado na nuvem.
