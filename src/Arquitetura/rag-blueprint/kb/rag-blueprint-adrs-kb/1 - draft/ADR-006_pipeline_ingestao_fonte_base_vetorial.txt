📋 ADR-006 — Pipeline de Ingestao: Da Fonte a Base Vetorial
================================================================================

Status:         proposed
Data:           21/03/2026
Decisor:        Fabio A. B. Rodrigues
Escopo:         RAG Corporativo — Pipeline de ingestao de documentos .md na Base Vetorial
Relaciona-se:   ADR-001 (Pipeline 4 Fases), ADR-002 (Soberania), ADR-003 (Modelo de Dados), ADR-005 (Front Matter), ADR-007 (Retrieval Hibrido), ADR-009 (Modelos de Embedding)

================================================================================
🧠 1. CONTEXTO
================================================================================

Esta ADR define o pipeline completo que transforma documentos .md finais
(produzidos na Fase 3 do ADR-001) em conhecimento pesquisavel dentro da
Base Vetorial. A Fase 4 do ADR-001 mencionava "ingestao no banco vetorial"
de forma resumida — este documento detalha CADA etapa desse processo.

Por que este ADR e necessario:

  A Base Vetorial nao e um simples banco de dados onde se "joga" arquivos.
  Ela e uma estrutura sofisticada que combina vetores de embedding, nos de
  grafo, relacoes semanticas e metadados estruturados. Para que essa
  estrutura funcione corretamente, o pipeline de ingestao precisa ser
  rigoroso, previsivel e observavel.

Premissas fundamentais:

  1. IDEMPOTENCIA — Re-executar o pipeline sobre o mesmo conjunto de
     documentos DEVE produzir zero alteracoes. Isso e critico porque:
       - Webhooks podem disparar duplicados
       - Cron jobs podem sobrepor execucoes
       - Falhas parciais exigem re-tentativa segura
       - Sem idempotencia, cada execucao gera inconsistencias cumulativas

  2. RE-INDEXACAO — Quando o modelo de embedding muda (ex: de
     text-embedding-3-small para text-embedding-3-large), o pipeline
     precisa re-gerar TODOS os vetores sem downtime. Isso acontece porque
     vetores de modelos diferentes vivem em espacos dimensionais
     incompativeis — nao se pode misturar vetores de modelos distintos
     no mesmo indice.

  3. LIMPEZA DE ORFAOS — Se um documento e removido do repositorio
     knowledge-base, seus chunks na Base Vetorial devem ser eliminados.
     Chunks orfaos poluem os resultados de busca e consomem espaco.

  4. TRIGGER POR RELEASE — O pipeline nao roda continuamente. Ele e
     disparado por tags de release no repositorio knowledge-base (conforme
     ADR-001, Fase 3). Isso garante que apenas documentos aprovados e
     versionados entrem na Base Vetorial.

  5. COMPATIBILIDADE COM ADR-003 — O pipeline deve popular os nos e
     relacoes definidos no modelo de dados (Document, Chunk, System,
     Module, Owner, Team, etc.) respeitando constraints de unicidade
     e tipos de propriedades.

Diagrama do fluxo macro:

  [Repo knowledge-base]     [Pipeline de Ingestao - 7 Etapas]     [Base Vetorial]
        |                                                                |
        |   release tag                                                  |
        +----------> [1.Descoberta] --> [2.Parse] --> [3.Chunking]       |
                         |                |              |               |
                         v                v              v               |
                     [4.Embeddings] --> [5.Persistencia] --> [6.Indexacao]
                                            |                    |
                                            v                    v
                                      [7.Observabilidade] --> [Metricas]

================================================================================
✅ 2. DECISAO — PIPELINE DE 7 ETAPAS
================================================================================

O pipeline de ingestao e composto por 7 etapas sequenciais. Cada etapa
tem responsabilidade unica, entrada e saida bem definidas, e pode ser
monitorada independentemente.

Por que 7 etapas e nao menos?

  Separar em 7 etapas permite:
  - Isolar falhas (se o chunking falha, os embeddings nem comecam)
  - Medir gargalos (saber exatamente onde o tempo e gasto)
  - Evoluir independentemente (trocar modelo de embedding sem alterar parse)
  - Testar isoladamente (unit tests por etapa)

  Unificar etapas (ex: chunking + embedding juntos) criaria acoplamento
  desnecessario e dificultaria debug em producao.

--------------------------------------------------------------------------------
ETAPA 1 — DESCOBERTA
--------------------------------------------------------------------------------

Objetivo: Identificar QUAIS documentos .md precisam ser processados nesta
execucao do pipeline.

O que esta etapa faz:

  1. Recebe a versao da release (ex: v1.3.0) via parametro de execucao
  2. Le o manifesto da release (releases/v1.3/manifest.json) que lista
     todos os arquivos .md incluidos nesta release
  3. Para cada arquivo .md no manifesto:
     a. Calcula o hash SHA-256 do conteudo do arquivo
     b. Captura metadados do Git:
        - path: caminho relativo no repositorio
        - branch: branch de origem (sempre main para releases)
        - commit: hash do commit que gerou a release
        - last_modified: data do ultimo commit que alterou o arquivo
     c. Consulta a Base Vetorial para verificar se ja existe um no
        Document com o mesmo document_id (extraido do front matter)
     d. Compara o hash armazenado na Base Vetorial com o hash calculado

  4. Classifica cada documento em uma de 4 categorias:

     - NEW: documento nao existe na Base Vetorial (nenhum no Document
       com esse document_id). Acao: processar todas as etapas seguintes.

     - UPDATED: documento existe mas o hash mudou (conteudo foi alterado
       desde a ultima ingestao). Acao: re-processar todas as etapas,
       substituindo chunks antigos.

     - UNCHANGED: documento existe e o hash e identico (nenhuma alteracao).
       Acao: PULAR completamente. Nao processar nenhuma etapa seguinte.
       ESTE e o mecanismo central de idempotencia.

     - DELETED: documento existe na Base Vetorial mas NAO esta no manifesto
       da release (foi removido do repositorio). Acao: remover o no
       Document e todos os seus chunks na etapa de Persistencia.

  5. Gera um relatorio de descoberta:
     - Total de documentos no manifesto
     - Quantos NEW, UPDATED, UNCHANGED, DELETED
     - Lista detalhada de cada documento e sua classificacao

Por que o hash e critico para idempotencia:

  O hash SHA-256 e uma funcao deterministica: o mesmo conteudo SEMPRE gera
  o mesmo hash. Isso significa que se o pipeline roda 2 vezes sobre o
  mesmo conjunto de documentos, na segunda execucao TODOS serao
  classificados como UNCHANGED (porque o hash no arquivo corresponde ao
  hash armazenado na Base Vetorial). Resultado: zero processamento,
  zero alteracoes. Idempotencia garantida.

  Alternativas ao hash que foram descartadas:
  - Data de modificacao: nao e confiavel (git clone reseta timestamps)
  - Numero de versao no front matter: depende do autor lembrar de atualizar
  - Tamanho do arquivo: colisoes obvias (arquivos diferentes, mesmo tamanho)

Saida da etapa:
  Lista de documentos com classificacao (NEW/UPDATED/UNCHANGED/DELETED),
  hash, path, metadados Git. Apenas NEW e UPDATED seguem para a Etapa 2.

--------------------------------------------------------------------------------
ETAPA 2 — PARSE
--------------------------------------------------------------------------------

Objetivo: Extrair conteudo estruturado de cada documento .md, separando
metadados (front matter) de conteudo (corpo do documento).

O que esta etapa faz:

  1. Para cada documento classificado como NEW ou UPDATED:

     a. EXTRAI O FRONT MATTER:
        - Le o bloco YAML entre os delimitadores --- no inicio do arquivo
        - Converte YAML em um dicionario/objeto estruturado
        - Campos esperados conforme ADR-005:
          * id (obrigatorio): identificador unico do documento
          * doc_type (obrigatorio): tipo do documento
          * title (obrigatorio): titulo descritivo
          * system: sistema corporativo relacionado
          * module: modulo funcional
          * domain: dominio de negocio
          * owner: responsavel pelo conteudo
          * team: time responsavel
          * status: estado do documento (draft/in-review/approved/deprecated)
          * confidentiality: nivel de classificacao (public/internal/restricted/confidential)
          * tags: lista de termos para descoberta
          * valid_from: data de inicio de vigencia
          * valid_until: data de fim de vigencia (null = vigente)
          * superseded_by: ID do documento que substitui este
          * supersedes: ID do documento anterior
          * created_at: data de criacao
          * updated_at: data de ultima atualizacao
          * release_version: preenchido pelo pipeline (nao pelo autor)

     b. VALIDA CONTRA O SCHEMA (ADR-005):
        - Verifica presenca dos campos obrigatorios (id, doc_type, title)
        - Verifica tipos de dados (tags deve ser lista, datas devem ser
          formato ISO 8601, confidentiality deve ser um dos 4 valores)
        - Verifica consistencia (valid_until >= valid_from, se preenchidos)
        - Se QUALQUER validacao falha: o documento e REJEITADO

        Por que a validacao e bloqueante:

          A filosofia "garbage in, garbage out" se aplica diretamente aqui.
          Um documento sem front matter valido nao pode ser indexado
          corretamente — seus chunks ficariam sem metadados, sem filtros
          de seguranca (confidentiality), sem rastreabilidade (owner, system).
          E melhor rejeitar UM documento e corrigir na fonte do que poluir
          a Base Vetorial com dados incompletos.

          Em producao, o documento rejeitado:
          - E registrado no log com motivo detalhado da rejeicao
          - Gera um alerta para o time de governanca
          - NAO bloqueia o processamento dos demais documentos
          - Pode ser corrigido e re-ingerido na proxima release

     c. EXTRAI HIERARQUIA DE HEADINGS:
        - Identifica todos os headings (# H1, ## H2, ### H3, etc.)
        - Constroi uma arvore hierarquica do documento
        - Cada heading recebe um "breadcrumb path":
          Ex: "Modulo de Cobranca > Fluxo de Boleto > Regras de Validacao"
        - Este breadcrumb sera herdado pelos chunks para fornecer contexto

     d. EXTRAI LINKS INTERNOS:
        - Detecta wikilinks: [[DOC-000456|titulo do link]]
        - Detecta links Markdown: [texto](caminho/para/documento.md)
        - Para cada link, registra: documento origem, documento destino, texto
        - Esses links serao convertidos em relacoes REFERENCES na Base Vetorial

     e. IDENTIFICA BLOCOS DE CODIGO:
        - Detecta blocos ``` delimitados por triple backticks
        - Registra a linguagem declarada (```python, ```cypher, etc.)
        - Blocos de codigo recebem tratamento especial no chunking
          (nunca sao divididos no meio)

     f. DETECTA REFERENCIAS A ENTIDADES:
        - Identifica mencoes a sistemas (via front matter system ou tags)
        - Identifica mencoes a modulos (via front matter module)
        - Identifica termos de glossario (via tags ou deteccao automatica)
        - Estas deteccoes alimentam a criacao de relacoes na etapa de
          Persistencia

  2. Saida da etapa:
     Para cada documento, um objeto estruturado contendo:
     - front_matter: dicionario com todos os campos validados
     - headings: arvore hierarquica de headings com breadcrumbs
     - internal_links: lista de referencias a outros documentos
     - code_blocks: lista de blocos de codigo com linguagem
     - entity_references: lista de entidades detectadas
     - raw_content: conteudo sem front matter (para chunking)
     - validation_status: "valid" ou lista de erros

--------------------------------------------------------------------------------
ETAPA 3 — CHUNKING
--------------------------------------------------------------------------------

Objetivo: Dividir o conteudo de cada documento em fragmentos semanticos
(chunks) que serao individualmente indexados e pesquisaveis.

Por que nao indexar o documento inteiro como um unico bloco:

  Modelos de embedding geram vetores mais precisos para textos curtos e
  focados do que para textos longos e multitematicos. Um documento de
  3000 tokens sobre "Modulo de Cobranca" trata de boletos, remessas,
  regras BACEN e integracao bancaria — temas distintos. Se o documento
  inteiro gera UM unico vetor, esse vetor sera uma "media" de todos
  esses temas, e nao representara bem NENHUM deles individualmente.

  Alem disso, ao retornar o documento inteiro como contexto para o LLM,
  estamos desperdicando tokens de contexto com informacao irrelevante
  para a pergunta especifica do usuario.

  Chunks menores e focados permitem:
  - Vetores mais precisos (representam UM tema)
  - Contexto mais relevante para o LLM (menos ruido)
  - Melhor ranking (granularidade fina)
  - Rastreabilidade precisa (saber QUAL trecho respondeu a pergunta)

O que esta etapa faz:

  1. ESTRATEGIA PRIMARIA — DIVISAO POR HEADINGS:

     O chunking usa os headings do Markdown como fronteiras naturais de
     divisao. Cada secao delimitada por um heading (e seus sub-headings)
     forma um candidato a chunk.

     Exemplo:
       ## Modulo de Cobranca          --> Chunk 1 (inicio)
         texto sobre cobranca...
         ### Boleto                   --> Chunk 2 (inicio)
           texto sobre boleto...
         ### Remessa                  --> Chunk 3 (inicio)
           texto sobre remessa...

     Se uma secao excede 800 tokens, ela e subdividida em paragrafos,
     mantendo o breadcrumb do heading pai.

     Se uma secao tem menos de 300 tokens, ela e MESCLADA com a secao
     seguinte (para evitar chunks muito curtos que perdem contexto).

  2. FAIXA DE TOKENS: 300 a 800 tokens por chunk

     Por que 300 tokens minimo:
       Chunks muito curtos (menos de 100-200 tokens) nao carregam contexto
       suficiente para gerar embeddings significativos. Uma frase isolada
       como "O prazo e de 5 dias" nao tem significado sem o contexto de
       QUAL prazo e para QUE serve.

     Por que 800 tokens maximo:
       Chunks muito longos diluem o foco semantico — o vetor resultante
       tenta representar multiplos conceitos e perde precisao. Alem disso,
       modelos de embedding tem limites de tokens de entrada (512 para
       muitos modelos, 8192 para text-embedding-3-small). 800 tokens da
       margem para a maioria dos modelos.

     Por que nao uma faixa mais estreita (ex: 400-600):
       Documentos corporativos tem grande variacao de densidade. Glossarios
       tem termos curtos; documentos arquiteturais tem secoes densas.
       A faixa 300-800 acomoda ambos os extremos.

     INTERACAO COM LIMITE DO MODELO DE EMBEDDING (ver ADR-009):

       A faixa 300-800 tokens foi definida para COERENCIA SEMANTICA dos
       chunks, NAO por limitacao do modelo de embedding. O modelo primario
       (BGE-M3, ADR-009) suporta ate 8192 tokens de entrada — portanto a
       faixa 300-800 esta bem dentro do limite do modelo.

       Porem, a ADR-009 recomenda max_seq_length de 512 tokens para a
       configuracao operacional (GTX 1070, 8GB VRAM). Isso significa que
       o texto TOTAL enviado ao modelo (breadcrumb + conteudo) deve
       respeitar 512 tokens na configuracao recomendada.

       Regras de compatibilidade:
       - O token_count do chunk (300-800) refere-se ao CONTEUDO apenas
       - O breadcrumb (heading_path) adiciona ~20-80 tokens extras
       - O total enviado ao modelo = breadcrumb + conteudo
       - Se o modelo configurado tem max_seq_length de 512: chunks devem
         ser limitados a ~450 tokens (512 - margem para breadcrumb)
       - Se o modelo configurado tem max_seq_length de 8192: a faixa
         300-800 pode ser usada integralmente sem risco de truncamento
       - O pipeline DEVE validar: (chunk_tokens + breadcrumb_tokens) <=
         max_seq_length do modelo configurado. Se exceder, o chunk e
         truncado e um warning e registrado no log.
       - A configuracao max_seq_length e lida do arquivo de configuracao
         do modelo (ver ADR-009, secao 4.3)

       Resumo: a faixa 300-800 e segura para BGE-M3 (8192 max), mas na
       configuracao operacional recomendada (512 max_seq_length), o
       pipeline deve ajustar o limite superior para ~450 tokens.
       Ver ADR-009 para limites especificos por modelo.

  3. HERANCA DE METADADOS:

     CADA chunk herda TODOS os metadados do documento pai:
     - document_id: de qual documento veio
     - doc_type: tipo do documento
     - system, module, domain: classificacao tematica
     - owner, team: responsaveis
     - confidentiality: nivel de acesso (CRITICO para seguranca)
     - tags: termos de descoberta
     - valid_from, valid_until: vigencia temporal

     Por que herdar tudo:
       O chunk e a unidade de busca. Quando o retriever executa uma query,
       ele aplica filtros (ex: confidentiality = "internal") DIRETAMENTE
       nos chunks. Se o chunk nao tiver esses campos, o filtro nao funciona
       e dados confidenciais podem vazar.

  4. BREADCRUMB (CAMINHO DE HEADINGS):

     Cada chunk recebe um campo heading_path que registra a hierarquia
     de headings que o contem:

       chunk.heading_path = "Modulo de Cobranca > Boleto > Regras de Validacao"

     Esse breadcrumb e incluido no texto do chunk para dar contexto ao
     embedding. Sem ele, um chunk contendo "O prazo e de 5 dias uteis"
     perde o contexto de que se trata do prazo de COMPENSACAO de BOLETO
     do modulo de COBRANCA.

  5. ESTRATEGIA POR TIPO DE DOCUMENTO (doc_type):

     Nem todos os documentos devem ser chunked da mesma forma. O tipo
     do documento (definido no front matter) determina a estrategia:

     a. architecture-doc (documento arquitetural):
        - Chunking por heading, faixa padrao 300-800 tokens
        - Diagramas ASCII preservados inteiros dentro do chunk
        - Tabelas preservadas inteiras (nao dividir no meio)
        - Cada secao relevante gera um chunk

     b. adr (registro de decisao arquitetural):
        - Chunks menores e mais precisos
        - Cada secao estrutural do ADR gera EXATAMENTE 1 chunk:
          * Contexto = 1 chunk
          * Decisao = 1 chunk (pode subdividir se muito longo)
          * Alternativas = 1 chunk por alternativa
          * Consequencias = 1 chunk
        - Por que: ADRs sao consultados para perguntas especificas
          ("por que foi decidido X?"), e a resposta esta em UMA secao

     c. runbook (procedimento operacional):
        - Chunk por procedimento/passo operacional
        - Cada passo numerado gera um chunk independente
        - Prerequisitos e observacoes ficam como chunks separados
        - Por que: operadores buscam "como fazer X", e a resposta e
          UM procedimento especifico

     d. glossary (glossario):
        - Chunk QUASE atomico: cada termo = 1 chunk
        - Excecao: termos muito curtos (menos de 50 tokens) sao agrupados
          por letra ou categoria
        - Por que: glossarios sao consultados para definicoes especificas,
          e misturar termos dilui a precisao

     e. task-doc (documento de tarefa):
        - Chunks separados por secao logica:
          * Contexto/Motivacao = 1 chunk
          * Escopo/Requisitos = 1 chunk
          * Decisoes tecnicas = 1 chunk
          * Criterios de aceite = 1 chunk
        - Por que: perguntas sobre tarefas sao especificas
          ("qual o criterio de aceite da task CU-4567?")

  6. REGRAS ESPECIAIS:

     a. Blocos de codigo NUNCA sao divididos no meio. Se um bloco de
        codigo faz o chunk exceder 800 tokens, o bloco de codigo fica
        sozinho como um chunk (com o heading pai como contexto).

     b. Tabelas Markdown NUNCA sao divididas no meio. Se uma tabela e
        muito longa (mais de 800 tokens), ela e dividida em grupos de
        linhas, mantendo o cabecalho da tabela em cada chunk.

     c. Listas longas sao divididas em grupos tematicos, nao por
        quantidade fixa de itens.

  7. CHUNKING HIERARQUICO — DECISAO ADIADA:

     Chunking hierarquico e uma tecnica onde cada chunk tambem armazena
     uma versao resumida do documento pai (ou secao pai) para fornecer
     contexto mais amplo durante o retrieval.

     Por que NAO implementar agora:
     - Aumenta significativamente o armazenamento (cada chunk fica maior)
     - Requer geracao de resumos (custo de LLM por chunk)
     - O breadcrumb de headings ja fornece contexto suficiente para o MVP
     - Complexidade de re-indexacao aumenta (cada mudanca de resumo
       invalida todos os chunks filhos)

     Quando reconsiderar:
     - Quando metricas de retrieval mostrarem que chunks perdem contexto
       frequentemente (Recall@10 abaixo de 70%)
     - Quando o volume de documentos ultrapassar 10.000 (mais necessidade
       de desambiguacao)
     - Quando houver feedback de usuarios dizendo "a resposta estava certa
       mas faltou contexto"

Saida da etapa:
  Lista de chunks para cada documento, cada um contendo:
  - chunk_id: identificador unico ({document_id}_chunk_{N})
  - content: texto do chunk com breadcrumb prepended
  - heading_path: hierarquia de headings
  - token_count: numero de tokens
  - chunk_index: posicao sequencial no documento
  - all inherited metadata (system, module, confidentiality, etc.)

--------------------------------------------------------------------------------
ETAPA 4 — EMBEDDINGS
--------------------------------------------------------------------------------

Objetivo: Gerar um vetor de embedding para cada chunk, transformando
texto em uma representacao numerica pesquisavel por similaridade.

O que esta etapa faz:

  1. Para cada chunk gerado na Etapa 3:
     a. Prepara o texto de entrada para o modelo de embedding:
        - Inclui o breadcrumb (heading_path) como prefixo
        - Inclui o conteudo do chunk
        - Formato: "[heading_path]\n\n[content]"
     b. Envia para o modelo de embedding em LOTES (batch processing)
     c. Recebe o vetor de embedding (array de floats)
     d. Armazena o vetor no campo chunk.embedding

  2. ESCOLHA DO MODELO DE EMBEDDING:

     A escolha depende da estrategia de soberania definida na ADR-002:

     Track A — Cloud (OpenAI):
       - Modelo: text-embedding-3-small
       - Dimensoes: 1536
       - Custo: ~$0.02 por 1M tokens
       - Latencia: ~100ms por batch de 100 chunks
       - Qualidade: estado da arte para portugues
       - Risco: dependencia de API externa

     Track B — On-premises / Soberano:
       - Modelo: BGE-M3 (BAAI)
       - Dimensoes: 1024
       - Custo: zero (rodando local)
       - Latencia: depende do hardware (~200-500ms por batch de 100 em GPU)
       - Qualidade: excelente para multilingual, incluindo portugues
       - Risco: exige infraestrutura de GPU

     A decisao final de Track e tomada na ADR-002. O pipeline suporta
     AMBOS os tracks sem alteracao de codigo (apenas configuracao).

  3. REGISTRO DE MODELO E VERSAO:

     CADA chunk armazena:
     - embedding_model: nome do modelo (ex: "text-embedding-3-small")
     - embedding_model_version: versao especifica (ex: "2024-01-25")
     - embedding_dimensions: numero de dimensoes do vetor
     - embedded_at: timestamp de quando o embedding foi gerado

     Por que registrar modelo e versao e CRITICO:

       Modelos de embedding evoluem. Quando um modelo novo e lancado
       (ex: text-embedding-3-large), os vetores gerados pelo modelo
       antigo NAO sao compativeis com os do modelo novo. Eles vivem
       em espacos dimensionais diferentes.

       Se misturarmos vetores de modelos diferentes no mesmo indice,
       a busca por similaridade coseno retornara resultados ERRADOS
       (comparando distancias em espacos incompativeis).

       O registro de modelo permite:
       - Saber EXATAMENTE quais chunks precisam ser re-embedded
       - Validar que todos os chunks usam o mesmo modelo
       - Auditar quando e por que houve troca de modelo

  4. PROCESSAMENTO EM LOTES (BATCH):

     Embeddings sao gerados em lotes de 100 chunks por chamada de API
     (ou por inferencia local). Motivos:

     - Eficiencia: uma chamada de API com 100 textos e mais rapida que
       100 chamadas individuais (overhead de rede)
     - Rate limiting: APIs cloud tem limites de requisicoes por minuto;
       batching maximiza throughput dentro dos limites
     - Tolerancia a falha: se um batch falha, apenas 100 chunks precisam
       ser re-processados (nao o documento inteiro)

     Tamanho do batch e configuravel para ajustar a capacidade do
     endpoint (cloud ou local).

  5. TRATAMENTO DE ERROS:

     - Se um batch falha (timeout, rate limit, erro de API):
       * Retry com backoff exponencial (1s, 2s, 4s, 8s, max 60s)
       * Apos 3 retries, marcar batch como FAILED
       * Chunks de batches FAILED sao registrados no log
       * Pipeline continua com os demais batches
       * Chunks sem embedding NAO sao persistidos (Etapa 5 os ignora)

  6. COMPORTAMENTO PARA CHUNKS PARCIALMENTE FALHADOS:

     Se a geracao de embedding falha para ALGUNS chunks de um documento
     (mas nao todos), o comportamento e:

     a. Chunks com embedding gerado com sucesso: persistidos normalmente
        na Etapa 5.
     b. Chunks com falha: enviados para FILA DE RETRY (max 3 tentativas
        com backoff exponencial: 30s, 60s, 120s).
     c. Apos retries esgotados sem sucesso:
        - Chunks falhados sao registrados no log com motivo detalhado
        - Documento e marcado como "incomplete" no log de execucao
        - Alerta gerado para o time de governanca (tipo: embedding_failure)
     d. Documento so e considerado INGERIDO COM SUCESSO quando TODOS os
        seus chunks tem embedding gerado. Enquanto houver chunks falhados,
        o documento fica com status "partial" no log de execucao.
     e. Na proxima execucao do pipeline: documentos "incomplete" sao
        reprocessados do zero (idempotencia garante seguranca — a Etapa 1
        compara hashes e classifica como UPDATED, forcando reprocessamento
        completo de todas as etapas).
     f. Se um documento permanece "incomplete" por mais de 3 execucoes
        consecutivas: escalar alerta para nivel CRITICO — indica problema
        sistematico com o modelo de embedding ou conteudo especifico.

Saida da etapa:
  Chunks enriquecidos com vetor de embedding e metadados do modelo.
  Chunks falhados registrados em fila de retry e log de execucao.

--------------------------------------------------------------------------------
ETAPA 5 — PERSISTENCIA
--------------------------------------------------------------------------------

Objetivo: Gravar (ou atualizar) os dados na Base Vetorial, criando nos,
relacoes e removendo dados obsoletos.

O que esta etapa faz:

  1. UPSERT DE DOCUMENT (criar ou atualizar):

     Para cada documento classificado como NEW:
     - Cria um novo no :Document com todas as propriedades do front matter
     - Define document_id como chave primaria
     - Registra release_version, file_hash, file_path, git_commit

     Para cada documento classificado como UPDATED:
     - Atualiza as propriedades do no :Document existente
     - Atualiza release_version, file_hash, updated_at
     - NAO altera o document_id (chave primaria imutavel)

     Por que o padrao UPSERT (e nao DELETE + CREATE):
       O upsert preserva o no Document e suas relacoes externas
       (BELONGS_TO System, OWNED_BY Owner, etc.) que podem ter sido
       enriquecidas manualmente ou por outros pipelines. DELETE + CREATE
       destruiria essas relacoes, exigindo recriacao.

  2. GERENCIAMENTO DE CHUNKS:

     Para documentos UPDATED:
     a. Deleta TODOS os chunks antigos associados ao documento
        (via relacao PART_OF)
     b. Cria novos chunks com os dados da Etapa 3-4
     c. Cria relacoes (:Chunk)-[:PART_OF]->(:Document)

     Para documentos NEW:
     a. Cria chunks com os dados da Etapa 3-4
     b. Cria relacoes (:Chunk)-[:PART_OF]->(:Document)

     Por que deletar chunks antigos de documentos atualizados:
       O chunking pode gerar quantidade diferente de chunks na nova
       versao (secoes adicionadas ou removidas). Tentar "atualizar"
       chunks individualmente exigiria diff semantico entre chunks
       antigos e novos — complexidade desnecessaria. E mais simples
       e seguro: deletar todos, criar todos novamente.

  3. CRIACAO/ATUALIZACAO DE RELACOES:

     Com base nos metadados do front matter e deteccoes do Parse:

     a. (:Document)-[:BELONGS_TO]->(:Module)
        - Baseado em front matter.module
        - Se o no :Module nao existe, cria com propriedades basicas
        - Se existe, reutiliza

     b. (:Module)-[:BELONGS_TO]->(:System)
        - Baseado em front matter.system
        - Se o no :System nao existe, cria com propriedades basicas

     c. (:Document)-[:OWNED_BY]->(:Owner)
        - Baseado em front matter.owner

     d. (:Owner)-[:MEMBER_OF]->(:Team)
        - Baseado em front matter.team
        - Criacao de Owner e Team se nao existirem

     e. (:Document)-[:REFERENCES]->(:Document)
        - Baseado em links internos detectados no Parse
        - So cria se o documento referenciado existir na Base Vetorial

     f. (:Document)-[:USES_TERM]->(:GlossaryTerm)
        - Baseado em termos detectados no Parse
        - So cria se o termo existir no glossario indexado

     g. (:Document)-[:RELATES_TO_TASK]->(:Task)
        - Baseado em front matter.task_id (se presente)
        - Nota: campo generico task_id substitui clickup_task_id para
          evitar lock-in com ferramentas especificas

     h. (:Document)-[:SUPERSEDES]->(:Document)
        - Baseado em front matter.supersedes
        - Relacao temporal para versionamento de conhecimento

     i. (:Document)-[:VERSION_OF]->(:DocumentFamily)
        - Para documentos com supersedes ou superseded_by
        - DocumentFamily agrupa todas as versoes de um mesmo conceito

  4. REMOCAO DE ORFAOS:

     Para documentos classificados como DELETED na Etapa 1:
     a. Remove todos os chunks associados (via PART_OF)
     b. Remove todas as relacoes do no Document
     c. Remove o no Document

     Para documentos cujo arquivo nao existe mais no repositorio:
     a. Identifica nos Document cuja file_path nao corresponde a
        nenhum arquivo no manifesto da release
     b. Aplica o mesmo processo de remocao acima

     Por que a remocao de orfaos e necessaria:
       Sem limpeza, a Base Vetorial acumula "conhecimento fantasma" —
       chunks de documentos que nao existem mais, retornando informacoes
       desatualizadas ou incorretas em buscas. Em um cenario corporativo,
       isso pode levar a decisoes baseadas em informacao obsoleta.

  5. TRANSACIONALIDADE:

     Toda a etapa de Persistencia roda dentro de UMA transacao na
     Base Vetorial. Se qualquer operacao falha (criacao de no, relacao,
     remocao), a transacao inteira e revertida (rollback).

     Por que: consistencia. Um documento parcialmente persistido (com
     alguns chunks mas nao todos, ou sem relacoes) e pior que nenhum
     documento. O rollback garante atomicidade.

  6. TRANSACIONALIDADE EM RELEASES GRANDES (>1000 documentos):

     Para releases com mais de 1000 documentos, uma unica transacao e
     inviavel (risco de timeout, lock contention, consumo de memoria).
     Nesse caso, o pipeline adota BATCHING TRANSACIONAL:

     a. Releases com >1000 documentos sao processadas em LOTES de 100
        documentos (com seus respectivos chunks e relacoes).

     b. Cada lote e UMA transacao independente — se o lote falha,
        apenas aquele lote sofre rollback. Os demais continuam.

     c. CHECKPOINT apos cada lote bem-sucedido:
        - Registra no log: batch_id, ultimo document_id processado,
          timestamp, contadores de operacoes do lote
        - Checkpoint persistido em tabela de controle (nao em memoria)

     d. Se o pipeline CRASHAR no meio da execucao:
        - Na re-execucao, o pipeline le o ultimo checkpoint
        - Retoma processamento a partir do lote seguinte ao checkpoint
        - Documentos de lotes ja persistidos serao classificados como
          UNCHANGED pela Etapa 1 (idempotencia via hash garante seguranca)

     e. A release so e marcada como COMPLETA quando TODOS os lotes
        forem processados com sucesso. Se algum lote falhou e nao foi
        recuperado, a release fica com status "partial_failure" no log
        de execucao.

     f. Tamanho do lote (100) e configuravel via parametro do pipeline.
        Valores menores reduzem risco de falha por lote mas aumentam
        overhead transacional. Valores maiores aumentam throughput
        mas elevam risco de timeout.

Saida da etapa:
  Base Vetorial atualizada com nos Document, Chunk, relacoes e limpeza
  de orfaos. Contadores de operacoes realizadas (criados, atualizados,
  deletados).

--------------------------------------------------------------------------------
ETAPA 6 — INDEXACAO
--------------------------------------------------------------------------------

Objetivo: Criar e manter indices que tornam as buscas eficientes na
Base Vetorial.

O que esta etapa faz:

  1. VECTOR INDEX no campo Chunk.embedding:
     - Tipo: indice vetorial para busca por similaridade (coseno)
     - Aplicado em: todos os nos :Chunk
     - Campo: embedding
     - Configuracao: dimensoes conforme modelo, distancia = cosine

     Por que: sem indice vetorial, cada busca exigiria comparacao com
     TODOS os chunks (busca linear, O(n)). Com indice HNSW (Hierarchical
     Navigable Small World), a busca e aproximada mas O(log n) —
     viabilizando buscas em milissegundos mesmo com milhoes de chunks.

  2. FULL-TEXT INDEX no campo Chunk.content:
     - Tipo: indice full-text para busca por palavras-chave
     - Aplicado em: todos os nos :Chunk
     - Campo: content
     - Analisador: com suporte a portugues (stemming, stopwords)

     Por que: a busca vetorial e excelente para perguntas conceituais
     mas FALHA para termos exatos (codigos, siglas, nomes proprios).
     O full-text index complementa o vetor (busca hibrida, ver ADR-007).

     Nota: este indice pode ser criado na Fase 2 (Metadados) do rollout
     mas so e usado ativamente na Fase 4 (GraphRAG Corporativo).

  3. INDICES COMPOSTOS para filtros frequentes:
     - Chunk.system: filtrar por sistema
     - Chunk.module: filtrar por modulo
     - Chunk.confidentiality: filtrar por nivel de acesso
     - Chunk.doc_type: filtrar por tipo de documento
     - Chunk.domain: filtrar por dominio de negocio
     - Document.status: filtrar por estado do documento

     Por que: filtros pre-retrieval (aplicados ANTES da busca vetorial)
     precisam ser rapidos. Sem indice, cada filtro exige scan completo.

  4. CONSTRAINTS DE UNICIDADE:
     - Document.document_id: UNICO (chave primaria)
     - Chunk.chunk_id: UNICO (chave primaria)

     Por que: garantir que o pipeline de upsert funcione corretamente.
     Sem constraint, um bug no pipeline poderia criar documentos
     duplicados, gerando resultados duplicados nas buscas.

  5. INDICES AUXILIARES:
     - Document.file_path: para lookup rapido por caminho
     - Document.release_version: para identificar documentos de uma release
     - Document.updated_at: para queries temporais

     Por que: queries administrativas (ex: "quais documentos entraram
     na release v1.3?") precisam ser rapidas para observabilidade.

Saida da etapa:
  Todos os indices criados/verificados. Se um indice ja existe, a etapa
  e no-op (nao recria). Registro de quais indices foram criados vs
  ja existentes.

--------------------------------------------------------------------------------
ETAPA 7 — OBSERVABILIDADE
--------------------------------------------------------------------------------

Objetivo: Registrar metricas, logs e verificacoes de consistencia para
cada execucao do pipeline.

O que esta etapa faz:

  1. LOG DE EXECUCAO:
     Cada execucao do pipeline gera um registro completo:
     - pipeline_run_id: identificador unico da execucao
     - release_version: qual release disparou a execucao
     - started_at / finished_at: timestamps
     - duration_seconds: tempo total
     - trigger: como foi disparado (webhook, manual, cron)
     - status: success, partial_failure, failure

  2. CONTADORES POR ETAPA:
     - Descoberta: documentos NEW, UPDATED, UNCHANGED, DELETED
     - Parse: documentos validados, rejeitados (com motivos)
     - Chunking: total de chunks gerados, media de chunks por documento,
       distribuicao de tamanhos (histograma)
     - Embeddings: total de vetores gerados, batches processados,
       batches com falha
     - Persistencia: nos criados, atualizados, deletados; relacoes criadas,
       removidas; orfaos limpos
     - Indexacao: indices criados vs ja existentes

  3. METRICAS DE PERFORMANCE:
     - Tempo por etapa (em segundos)
     - Latencia media de embedding (ms por chunk)
     - Latencia media de write na Base Vetorial (ms por operacao)
     - Throughput: chunks processados por segundo
     - Tamanho total do indice vetorial apos execucao

  4. RASTREAMENTO DE ERROS:
     - Lista de documentos que falharam e em qual etapa
     - Motivo da falha (validacao, embedding timeout, DB error)
     - Stack trace para erros inesperados
     - Classificacao: erro transiente (retry possivel) vs permanente

  5. VERIFICACAO DE CONSISTENCIA:
     Apos a execucao, o pipeline verifica:
     - Todo documento no manifesto tem um no Document correspondente
       (exceto rejeitados)
     - Todo no Document tem pelo menos 1 chunk associado
     - Todo chunk tem embedding preenchido
     - Nenhum chunk orfao existe (chunk sem relacao PART_OF)
     - Todos os chunks usam o mesmo modelo de embedding
     Se qualquer verificacao falha, status = partial_failure

  6. METRICAS DE SAUDE (HEALTH):
     Registradas apos cada execucao para monitoramento continuo:
     - Total de documentos na Base Vetorial
     - Total de chunks
     - Total de relacoes por tipo
     - Tamanho do indice vetorial (em bytes)
     - Ultima execucao bem-sucedida (timestamp)
     - Cobertura: % de documentos no repo com correspondente na Base Vetorial

  7. METRICAS DE QUALIDADE DE RETRIEVAL (sanity check pos-ingestao):

     Apos cada ingestao bem-sucedida, o pipeline executa um teste de
     qualidade automatizado usando um GOLDEN SET de perguntas e respostas
     esperadas:

     a. Golden set: conjunto de 50-100 perguntas curadas manualmente,
        cada uma com a lista de chunks esperados como resposta (ground truth)
     b. Para cada pergunta do golden set:
        - Executar busca vetorial nos chunks recem-ingeridos
        - Recuperar top-10 resultados
        - Calcular Recall@10: % de chunks esperados presentes no top-10
     c. Metrica principal: Recall@10 medio sobre todo o golden set
     d. Threshold de aceitacao: Recall@10 >= 70% (valor inicial — ajustar
        conforme maturidade do sistema)
     e. Se Recall@10 cai abaixo do threshold:
        - Alerta gerado para o time de arquitetura
        - NAO bloqueia a ingestao (metricas sao informativas neste estagio)
        - Log registra degradacao com detalhes (quais perguntas falharam)
     f. Tendencia: comparar Recall@10 entre releases consecutivas para
        detectar degradacao gradual (ex: v1.2 = 85%, v1.3 = 78%, v1.4 = 72%)

     Ver ADR-007 para definicao completa de metricas de retrieval.
     Ver ADR-008 (quando criado) para thresholds e SLAs de qualidade.

Saida da etapa:
  Registro completo da execucao armazenado em tabela de auditoria.
  Metricas publicadas para dashboard de monitoramento.
  Resultado do golden set (Recall@10) registrado para tendencia.

--------------------------------------------------------------------------------
2.8. IDEMPOTENCIA (Complemento da Decisao)
--------------------------------------------------------------------------------

Definicao precisa:

  Um pipeline e IDEMPOTENTE quando executar a funcao f(x) duas vezes
  produz o MESMO resultado que executar uma vez:

    f(f(x)) = f(x)

  No contexto deste pipeline: se os documentos no repositorio NAO
  mudaram desde a ultima execucao, re-executar o pipeline deve resultar
  em ZERO alteracoes na Base Vetorial.

Como a idempotencia e garantida neste pipeline:

  1. Etapa 1 (Descoberta): a comparacao de hash SHA-256 classifica
     documentos inalterados como UNCHANGED. Esses documentos sao
     completamente ignorados — nao passam pelas etapas 2-6.

  2. Etapa 5 (Persistencia): o padrao UPSERT garante que, mesmo se
     um documento NEW for re-processado (por algum motivo), o resultado
     e identico — o no Document e atualizado com os mesmos dados.

  3. Etapa 6 (Indexacao): indices ja existentes nao sao recriados.
     A verificacao e CREATE IF NOT EXISTS.

Por que a idempotencia e critica:

  - WEBHOOKS DUPLICADOS: plataformas Git (GitHub, GitLab) podem enviar
    o mesmo webhook mais de uma vez. Sem idempotencia, cada webhook
    duplicado geraria chunks duplicados.

  - CRON OVERLAP: se o pipeline roda via cron e uma execucao demora mais
    que o intervalo, duas execucoes rodam simultaneamente sobre os mesmos
    dados. Com idempotencia, a segunda execucao e no-op.

  - RETRIES SEGUROS: se o pipeline falha na Etapa 5 (persistencia) e
    e re-executado, os documentos ja processados nas etapas 1-4 serao
    classificados como UNCHANGED e pulados. Apenas os que falharam serao
    re-processados.

  - TESTABILIDADE: rodar o pipeline 2 vezes e comparar o diff da
    Base Vetorial e o teste definitivo de idempotencia:
      $ pipeline run --release v1.3
      $ pipeline run --release v1.3  # segunda vez
      $ db diff  # deve ser VAZIO

--------------------------------------------------------------------------------
2.9. RE-INDEXACAO BLUE/GREEN
--------------------------------------------------------------------------------

Problema:

  Quando o modelo de embedding muda (ex: de text-embedding-3-small para
  text-embedding-3-large), TODOS os chunks precisam ser re-embedded.
  Nao se pode simplesmente sobrescrever os vetores antigos porque:

  1. Durante a re-embedding (que pode levar horas para milhares de chunks),
     o sistema ficaria com vetores misturados (antigos e novos no mesmo
     indice), gerando resultados inconsistentes.

  2. Se a re-embedding falha no meio, o sistema fica em estado corrompido
     (metade dos chunks com modelo A, metade com modelo B).

  3. Se o novo modelo gera resultados piores, nao ha rollback possivel.

Solucao — Blue/Green Index:

  Inspirado no deployment blue/green, o processo cria UM NOVO indice
  ao lado do antigo, sem altera-lo:

  Passo 1 — CRIAR NOVO INDICE:
    - Criar indice vetorial "v2" com as dimensoes do novo modelo
    - O indice "v1" continua ativo e servindo queries normalmente
    - Nenhum downtime ate aqui

  Passo 2 — RE-EMBED TODOS OS CHUNKS:
    - Para cada chunk na Base Vetorial:
      * Gerar novo embedding com o novo modelo
      * Armazenar em campo temporario (embedding_v2)
    - Esse processo pode levar horas — nao impacta o servico

  Passo 3 — VALIDACAO DE QUALIDADE:
    - Executar um "golden set" de 50-100 perguntas conhecidas
    - Para cada pergunta:
      * Buscar no indice v1 (modelo antigo) → top-10 resultados
      * Buscar no indice v2 (modelo novo) → top-10 resultados
      * Calcular Recall@10 para ambos
    - Criterio de aceitacao: Recall@10 do v2 >= Recall@10 do v1
    - Se o novo modelo e PIOR: abortar, deletar indice v2, manter v1

  Passo 4 — SWITCH ATOMICO:
    - Se validacao passou:
      * Mover embedding_v2 para embedding (campo principal)
      * Atualizar embedding_model e embedding_model_version
      * Atualizar indice ativo para v2
    - Operacao atomica (transacao unica)
    - Downtime: zero (switch e instantaneo)

  Passo 5 — CLEANUP:
    - Apos confirmacao de que v2 funciona em producao (24-48h):
      * Deletar indice v1
      * Remover campos temporarios
    - Manter v1 por 48h permite rollback rapido se problemas forem
      detectados em producao

Por que Blue/Green e nao "substituir direto":

  - ZERO DOWNTIME: o sistema nunca fica sem indice funcional
  - ROLLBACK: se o novo modelo e pior, basta nao fazer o switch
  - VALIDACAO: permite comparar qualidade ANTES de comprometer
  - SEGURANCA: nenhum dado e perdido durante o processo

--------------------------------------------------------------------------------
2.10. TRIGGER
--------------------------------------------------------------------------------

Mecanismo de disparo:

  O pipeline NAO roda continuamente nem em polling. Ele e disparado
  EXCLUSIVAMENTE por eventos de release no repositorio knowledge-base.

Fluxo:

  1. Mantenedor do knowledge-base cria uma release tag (ex: v1.3.0)
  2. GitHub/GitLab envia webhook para o servico de ingestao
  3. Webhook handler recebe POST com payload contendo:
     - release_version: "v1.3.0"
     - tag_commit: hash do commit da tag
     - release_notes: descricao das mudancas
  4. Handler envia POST /admin/ingest para o pipeline:
     {
       "release_version": "v1.3.0",
       "commit": "abc123def",
       "manifest_path": "releases/v1.3/manifest.json"
     }
  5. Pipeline le o manifesto e inicia as 7 etapas

Quem gera o manifest.json:

  O manifest.json e gerado AUTOMATICAMENTE pelo pipeline de promocao
  (Fase 3, ADR-001) no momento da criacao da release tag. O processo e:

  1. Pipeline de promocao varre a pasta docs/ do rag-knowledge-base
  2. Para cada arquivo .md encontrado:
     - Extrai document_id do front matter
     - Calcula hash SHA-256 do conteudo
     - Extrai doc_type do front matter
     - Captura last_modified do Git (data do ultimo commit que alterou o arquivo)
  3. Compara com o manifesto da release anterior (se existir):
     - Arquivo novo (nao existia antes) → action: "new"
     - Arquivo com hash diferente → action: "updated"
     - Arquivo que existia antes mas nao existe mais → action: "deleted"
     - Arquivo com hash identico → action: "unchanged" (incluido para completude)
  4. Gera o manifest.json com a lista completa
  5. Manifesto e ASSINADO pela service account do pipeline (campo
     "created_by" identifica a conta de servico, permitindo auditoria)

  O pipeline de ingestao (este ADR) CONSOME o manifesto — nunca o gera.

O manifesto da release (manifest.json) contem:

  {
    "version": "1.3.0",
    "created_at": "2026-03-21T10:00:00Z",
    "created_by": "pipeline-service-account",
    "documents": [
      {
        "path": "docs/financeiro/modulo-cobranca.md",
        "document_id": "DOC-000123",
        "action": "updated"
      },
      {
        "path": "docs/financeiro/modulo-pagamentos.md",
        "document_id": "DOC-000124",
        "action": "new"
      },
      {
        "path": "docs/rh/politica-ferias.md",
        "document_id": "DOC-000089",
        "action": "deleted"
      }
    ]
  }

Por que trigger por release e nao por commit:

  - Nem todo commit representa conhecimento "pronto" — commits podem
    ser parciais, experimentais ou de WIP
  - Releases representam conjuntos APROVADOS de mudancas (via PR review)
  - A release tag permite versionar o estado da Base Vetorial
    (saber exatamente qual conteudo esta indexado em cada momento)
  - Facilita rollback: se a release v1.3 tem problema, re-ingerir v1.2

Versao da release armazenada:

  O campo Document.release_version e atualizado com a versao da release
  que o ingeriu. Isso permite:
  - Saber qual release trouxe cada documento
  - Identificar documentos de releases antigas (nao atualizados ha muito)
  - Auditar o historico de ingestoes

--------------------------------------------------------------------------------
2.11. CONCORRENCIA DE RELEASES
--------------------------------------------------------------------------------

Problema:

  O que acontece se uma nova release (v1.4) e criada enquanto o pipeline
  da release anterior (v1.3) ainda esta em execucao? Execucoes simultaneas
  sobre a mesma Base Vetorial podem causar corrupcao ou inconsistencia
  (escritas concorrentes no mesmo indice, conflitos de lock, chunks
  parcialmente atualizados).

Solucao — Lock de Execucao:

  1. LOCK EXCLUSIVO: o pipeline adquire um lock exclusivo (via arquivo de
     lock ou flag em tabela de controle na Base Vetorial) ANTES de iniciar
     a Etapa 1. O lock registra: pipeline_run_id, release_version,
     started_at, PID do processo.

  2. SE O LOCK JA EXISTE (outro pipeline esta rodando):
     - A nova release e adicionada a uma FILA de execucao (tabela de
       controle com: release_version, queued_at, status = "queued")
     - O webhook handler retorna HTTP 202 (Accepted) com mensagem:
       "Release v1.4 enfileirada. Pipeline v1.3 em execucao."
     - Ao terminar a execucao corrente, o pipeline verifica a fila e
       inicia automaticamente a proxima release enfileirada.

  3. NUNCA rodar dois pipelines de ingestao simultaneamente contra a
     mesma Base Vetorial. Razoes:
     - Escritas concorrentes no mesmo indice vetorial podem gerar
       corrupcao (ex: dois processos tentando deletar e criar o mesmo
       chunk simultaneamente)
     - Transacoes concorrentes em Neo4j podem causar deadlocks
     - A verificacao de consistencia (Etapa 7) seria invalida se outro
       processo estiver alterando dados simultaneamente

  4. CLEANUP DO LOCK: o lock e liberado automaticamente quando o
     pipeline termina (sucesso ou falha). Se o processo crashar sem
     liberar o lock, um mecanismo de timeout (ex: lock expira apos
     4 horas) permite que a proxima execucao adquira o lock.

  5. ORDEM DE PROCESSAMENTO: releases enfileiradas sao processadas
     em ordem cronologica (FIFO). Se v1.4 e v1.5 estao na fila,
     v1.4 e processada primeiro.

================================================================================
❌ 3. ALTERNATIVAS DESCARTADAS
================================================================================

3.1 Pipeline em streaming (processamento continuo):
  - Descricao: cada commit dispara processamento imediato do arquivo alterado
  - Por que descartada: commits intermediarios geram ingestoes parciais;
    sem conceito de "conjunto aprovado"; impossibilita rollback coerente;
    gera excesso de re-processamento

3.2 Pipeline batch diario (cron):
  - Descricao: cron roda 1x por dia, processa todos os .md do repo
  - Por que descartada: desperdicio (re-processa documentos inalterados);
    latencia alta (mudanca de manha so disponivel no dia seguinte);
    sem vinculo com release (nao sabe qual versao esta processando)

3.3 Chunking por tamanho fixo (sem headings):
  - Descricao: dividir texto em blocos de exatamente 500 tokens
  - Por que descartada: ignora estrutura semantica do documento;
    corta frases e paragrafos no meio; perde contexto de headings;
    gera chunks que misturam temas diferentes

3.4 Embeddings sob demanda (lazy):
  - Descricao: gerar embedding apenas quando um chunk e buscado pela
    primeira vez (cache-aside)
  - Por que descartada: primeira busca seria lenta (latencia de embedding
    + indexacao); indice vetorial precisa de todos os vetores pre-computados
    para HNSW funcionar; complexidade de cache invalidation

3.5 Delete + Recreate (em vez de Upsert):
  - Descricao: a cada execucao, deletar TUDO e recriar do zero
  - Por que descartada: perda de relacoes externas; downtime durante
    reprocessamento; desperdicio de recursos (re-processar inalterados);
    sem idempotencia

================================================================================
⚖️ 4. CONSEQUENCIAS
================================================================================

4.1 Positivas:
  - Pipeline previsivel e reprodutivel (mesma entrada = mesma saida)
  - Idempotencia elimina riscos de re-execucao
  - Observabilidade completa de cada execucao
  - Re-indexacao sem downtime (Blue/Green)
  - Suporte a multiplos modelos de embedding (Track A e B do ADR-002)
  - Limpeza automatica de orfaos mantendo Base Vetorial limpa
  - Trigger por release garante que so conhecimento aprovado e indexado
  - Chunking adaptativo por tipo de documento maximiza qualidade de retrieval
  - Heranca de metadados nos chunks viabiliza filtros pre-retrieval

4.2 Negativas / Trade-offs:
  - Complexidade de implementacao (7 etapas, cada uma com logica propria)
  - Re-indexacao Blue/Green requer espaco temporario (2 indices simultaneos)
  - Validacao bloqueante no Parse pode rejeitar documentos com problemas
    menores (trade-off consciente: preferimos rejeitar a poluir)
  - Processamento em batch (por release) pode ter latencia de horas para
    releases grandes (>1000 documentos)
  - Chunking por tipo de documento requer manutenção das estrategias
    conforme novos doc_types sao adicionados

4.3 Riscos:
  - Hash collision (SHA-256): probabilidade negligivel (2^128 para colisao)
  - Embedding model deprecated: fornecedor descontinua modelo sem aviso;
    mitigacao: monitorar announcements, ter Track B como fallback
  - Manifesto desatualizado: release criada sem atualizar manifest.json;
    mitigacao: pipeline de PR valida manifesto automaticamente
  - Base Vetorial indisponivel durante persistencia: mitigacao: retry
    com backoff, alertas automaticos
