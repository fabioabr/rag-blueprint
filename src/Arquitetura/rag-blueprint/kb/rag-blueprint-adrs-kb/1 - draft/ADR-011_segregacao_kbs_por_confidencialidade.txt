🧠 1. CONTEXTO
================================================================================

1.1. O PROBLEMA: FILTRO PRE-RETRIEVAL NAO E SUFICIENTE
---------------------------------------------------------------------------

O ADR-004 estabeleceu 4 niveis de confidencialidade para toda a base de
conhecimento corporativa: PUBLIC, INTERNAL, RESTRICTED e CONFIDENTIAL.
Tambem definiu que o filtro pre-retrieval — aplicado antes da busca vetorial —
e INVIOLAVEL: nenhum chunk deve ser retornado ao usuario sem que o filtro de
acesso tenha sido aplicado.

Essa decisao esta correta como PRINCIPIO, mas apresenta uma fragilidade
critica quando analisada sob a otica de IMPLEMENTACAO:

    O filtro pre-retrieval e um controle na camada de SOFTWARE (query).
    Qualquer controle na camada de software e vulneravel a:

    (a) Bugs de implementacao — um erro no codigo do filtro pode permitir
        que chunks RESTRICTED sejam retornados em buscas PUBLIC. Exemplo:
        uma condicao WHERE mal formulada, um operador AND que deveria ser
        OR, um campo confidentiality nao indexado que causa fallback para
        full scan sem filtro.

    (b) Prompt injection — um atacante pode manipular a query para tentar
        bypassar o filtro. Mesmo com filtro pre-retrieval implementado
        corretamente, se o MCP Server que processa a query tiver acesso
        a TODOS os dados na mesma Base Vetorial, uma vulnerabilidade no
        parsing da query pode expor chunks de niveis superiores.

    (c) Misconfiguration — um deploy incorreto, uma variavel de ambiente
        errada, um perfil de acesso configurado de forma permissiva demais.
        Em sistemas complexos, erros de configuracao sao a causa numero 1
        de vazamento de dados (segundo relatorios da Verizon DBIR).

    (d) Escalacao de privilegios — se todos os dados estao na mesma Base
        Vetorial, qualquer vulnerabilidade que permita acesso direto a
        Base Vetorial (SQL injection no driver, credencial vazada, acesso
        administrativo comprometido) expoe TODOS os niveis de uma vez.

    (e) Complexidade de auditoria — auditar se um filtro de query esta
        funcionando corretamente em TODAS as situacoes e exponencialmente
        mais dificil do que auditar se dois servidores estao em redes
        separadas.

1.2. CONTEXTO REGULATORIO: RISCO INACEITAVEL
---------------------------------------------------------------------------

O projeto RAG Corporativo opera dentro de uma instituicao financeira regulada
pelo Banco Central do Brasil (BACEN), pela Comissao de Valores Mobiliarios
(CVM) e pela Lei Geral de Protecao de Dados (LGPD — Lei 13.709/2018).

Nesse contexto, o risco de vazamento de dados classificados como RESTRICTED
ou CONFIDENTIAL nao e apenas um problema tecnico — e um problema
REGULATORIO e LEGAL:

    - LGPD: multas de ate 2% do faturamento da empresa por incidente de
      vazamento de dados pessoais (art. 52). Alem disso, o titular tem
      direito a indenizacao por dano moral e material.

    - BACEN: a Resolucao 4.893/2021 exige que instituicoes financeiras
      mantenham controles de seguranca proporcionais a sensibilidade dos
      dados. Falhas podem resultar em sancoes administrativas, restricoes
      operacionais e ate intervencao.

    - CVM: informacoes privilegiadas (insider information) tem tratamento
      especifico pela Instrucao CVM 400 e pela Lei 6.385/1976. Vazamento
      pode configurar crime de uso de informacao privilegiada.

    - SOX (se aplicavel): controles insuficientes sobre informacao
      financeira podem resultar em sancoes nos EUA.

Para uma instituicao financeira, confiar EXCLUSIVAMENTE em um filtro de
query como unico mecanismo de protecao de dados RESTRICTED/CONFIDENTIAL
e INACEITAVEL do ponto de vista de compliance. Reguladores esperam
controles em MULTIPLAS CAMADAS, com isolamento fisico quando possivel.

1.3. A SOLUCAO: MOVER O CONTROLE DE ACESSO DA QUERY PARA A INFRAESTRUTURA
---------------------------------------------------------------------------

A ideia central deste ADR e simples e poderosa:

    Se dados RESTRICTED estao em uma Base Vetorial SEPARADA, com um MCP
    Server SEPARADO, em uma rede SEPARADA, entao e FISICAMENTE IMPOSSIVEL
    que um usuario acessando o MCP publico consiga ver dados RESTRICTED.

    Nao importa se ha bug no filtro, prompt injection, misconfiguration
    ou escalacao de privilegios no MCP publico — os dados RESTRICTED
    simplesmente NAO EXISTEM naquele ambiente.

Isso e o que chamamos de SEGURANCA POR ISOLAMENTO DE INFRAESTRUTURA —
em contraste com seguranca por filtro de software.

O ADR-001 ja antecipou essa possibilidade ao definir que cada contexto de
negocio pode ter sua KB separada (pasta kb/ com subpastas). Este ADR
formaliza essa separacao como DECISAO ARQUITETURAL para segregacao por
confidencialidade.

O ADR-002 reforca essa abordagem ao definir duas trilhas de infraestrutura:
    - Trilha A (Cloud): aceita para dados PUBLIC e INTERNAL
    - Trilha B (On-Premise): obrigatoria para dados RESTRICTED e CONFIDENTIAL

Com KBs segregadas, cada Base Vetorial pode residir na trilha correta:
    - Base Vetorial A (public+internal) → pode ser cloud
    - Base Vetorial B (restricted) → obrigatoriamente on-premises
    - Base Vetorial C (confidential) → obrigatoriamente on-premises, isolada

#decisao/isolamento-fisico

1.4. POR QUE AGORA
---------------------------------------------------------------------------

Essa decisao precisa ser tomada ANTES da implementacao do pipeline de
ingestao (ADR-006), porque o roteamento de documentos para a Base Vetorial
correta depende da arquitetura de segregacao estar definida. Se comecarmos
com uma unica Base Vetorial e depois precisarmos segregar, o custo de
migracao sera significativo:

    - Re-ingestao de todos os documentos
    - Recriacao de indices vetoriais
    - Reconfiguracao de todos os MCPs e agentes
    - Re-teste de todos os fluxos de acesso
    - Periodo de indisponibilidade durante migracao

Definir a segregacao agora permite que o pipeline ja seja construido com
suporte a multiplas KBs desde a Fase 1.

#decisao/segregacao-fisica

    NOTA IMPORTANTE — Filtro pre-retrieval como camada ADICIONAL:
    O filtro pre-retrieval definido no ADR-004 NAO e eliminado. Ele continua
    existindo como camada ADICIONAL de defesa DENTRO de cada KB. Exemplo:
    dentro da kb-restricted, o filtro pre-retrieval pode ser usado para
    restringir acesso por DOMINIO de negocio (um Manager de financeiro nao
    ve documentos restricted de compliance). A segregacao fisica substitui
    o filtro como UNICA barreira entre niveis, mas o filtro permanece util
    para granularidade DENTRO do mesmo nivel.

2.1. MODELO DE SEGREGACAO — 3 NIVEIS DE KB
--------------------------------------------------------------------------------

O modelo define 3 Bases de Conhecimento segregadas, cada uma com sua propria
Base Vetorial e seu proprio MCP Server:

............................................................................
NIVEL 1: KB PUBLICA + INTERNA (acesso geral)
............................................................................

    Identificador:      kb-public-internal
    Confidencialidade:  PUBLIC + INTERNAL
    Base Vetorial:      Instancia A

    Conteudo:
        - Documentacao de sistemas (arquitetura, diagramas, fluxos)
        - ADRs (decisoes arquiteturais) classificados como internal
        - Runbooks operacionais publicos (deploy, rollback, monitoring)
        - Glossario de termos tecnicos e de negocio (nao proprietarios)
        - Tutoriais, guias, onboarding de novos colaboradores
        - Documentacao de APIs (internas e externas)
        - Manuais de ferramentas e processos padrao
        - FAQ tecnico e operacional
        - Documentacao de bibliotecas e frameworks utilizados
        - Descricao de modulos e integracao entre sistemas

    MCP Server:         mcp-knowledge-public
    Acesso:             Todos os funcionarios autenticados, agentes de IA
                        genericos, assistentes de onboarding
    Trilha (ADR-002):   Trilha A (Cloud) — permitido usar cloud com controles
                        (DPA, nao-treinamento, logs)
    Infraestrutura:     Pode ser hospedada em cloud provider (AWS, Azure, GCP)
                        ou on-premises, conforme decisao de custo/performance
    Autenticacao:       Token de servico ou SSO corporativo
    Autorizacao:        Perfis Analyst, Manager, Director — todos tem acesso
    Auditoria:          Log de queries (quem perguntou o que, quando)

............................................................................
NIVEL 2: KB RESTRITA (acesso por cargo/funcao)
............................................................................

    Identificador:      kb-restricted
    Confidencialidade:  RESTRICTED
    Base Vetorial:      Instancia B (SEPARADA FISICAMENTE da Instancia A)

    Conteudo:
        - Politicas internas da instituicao
        - Dados financeiros nao publicados (balancetes, projecoes)
        - Relatorios de gestao e indicadores estrategicos
        - Post-mortems de incidentes de seguranca
        - Auditorias internas e relatorios de compliance
        - Estrategias de produto e mercado
        - Dados de performance de areas/squads
        - Mapeamento de riscos operacionais
        - Informacoes de PLD (Prevencao a Lavagem de Dinheiro)
        - Dados de fornecedores com clausula de confidencialidade
        - Regulacoes internas pre-publicacao
        - Relatorios de due diligence

    MCP Server:         mcp-knowledge-restricted
    Acesso:             Gestores (Manager), Diretoria (Director), agentes de IA
                        com perfil explicitamente configurado para RESTRICTED
    Trilha (ADR-002):   Trilha B (On-Premise) — OBRIGATORIO processamento local
    Infraestrutura:     Servidores on-premises da instituicao, em rede
                        corporativa protegida por firewall e segmentacao de rede
    Autenticacao:       SSO corporativo + MFA obrigatorio
    Autorizacao:        Manager: acesso restrito ao seu dominio de negocio
                        Director: acesso amplo a restricted
                        Agente IA: perfil explicito por dominio
    Auditoria:          Log detalhado de queries + log de acesso ao MCP +
                        alertas para padroes anomalos de consulta
    Controles extras:
        - Rede segmentada (VLAN dedicada ou subnet isolada)
        - Acesso ao servidor da Base Vetorial B restrito por IP/firewall
        - Backup criptografado com chaves gerenciadas pela instituicao
        - Monitoramento de acesso em tempo real (SIEM)
        - Revisao periodica de acessos (access review trimestral)

............................................................................
NIVEL 3: KB CONFIDENCIAL (acesso minimo)
............................................................................

    Identificador:      kb-confidential
    Confidencialidade:  CONFIDENTIAL
    Base Vetorial:      Instancia C (ISOLADA — rede separada se possivel)

    Conteudo:
        - Contratos com clientes e fornecedores
        - Dados pessoais anonimizados (residuos de PII em documentos)
        - Estrategia corporativa de longo prazo
        - Informacoes de M&A (fusoes e aquisicoes)
        - Dados de insider trading / informacoes privilegiadas
        - Processos judiciais em andamento
        - Dados de remuneracao de executivos
        - Planos de contingencia e continuidade de negocios (BCP/DRP)
        - Vulnerabilidades de seguranca nao corrigidas
        - Relatorios de penetration testing
        - Dados regulatorios confidenciais (comunicacao com BACEN/CVM)

    MCP Server:         mcp-knowledge-confidential
    Acesso:             Diretoria (Director) com justificativa registrada,
                        Compliance Officer, agentes de IA DEDICADOS (nao
                        compartilhados) com aprovacao documentada
    Trilha (ADR-002):   Trilha B (On-Premise) — OBRIGATORIO, com controles EXTRAS
    Infraestrutura:     Servidores on-premises DEDICADOS, preferencialmente em
                        rede FISICAMENTE SEPARADA (air-gapped ou DMZ dedicada)
    Autenticacao:       SSO corporativo + MFA obrigatorio + aprovacao just-in-time
                        (acesso sob demanda com tempo limitado)
    Autorizacao:        Director: com justificativa em log de auditoria
                        Compliance Officer: acesso funcional
                        Agente IA: acesso NEGADO por padrao; apenas com
                        configuracao explicita e aprovacao documentada
    Auditoria:          Log completo de TODAS as operacoes (queries, acessos,
                        tentativas negadas) + alertas automaticos para qualquer
                        acesso + revisao manual de logs semanalmente
    Controles extras:
        - Rede fisicamente separada ou fortemente segmentada
        - Acesso ao servidor requer VPN dedicada + aprovacao just-in-time
        - Dados em repouso criptografados (AES-256)
        - Dados em transito criptografados (TLS 1.3)
        - Chaves de criptografia em HSM (Hardware Security Module)
        - Backup com criptografia separada, armazenado em cofre
        - Monitoramento 24/7 com alertas para SOC
        - Pen test periodico no ambiente isolado
        - Retencao de logs: minimo 5 anos (requisito BACEN)
        - Acesso revogado automaticamente apos periodo configuravel

2.2. VANTAGENS DO MODELO DE SEGREGACAO FISICA
--------------------------------------------------------------------------------

(a) SEGURANCA FISICA — ELIMINACAO DE CLASSE INTEIRA DE VULNERABILIDADES

    Com dados RESTRICTED em uma Base Vetorial separada, qualquer
    vulnerabilidade na Base Vetorial A (public+internal) — bug, injection,
    misconfiguration, credencial vazada — NAO tem como acessar dados
    RESTRICTED. Eles simplesmente nao existem naquele ambiente.

    Isso elimina uma classe inteira de ataques:
        - Filtro pre-retrieval bugado → sem impacto nos dados restricted
        - Prompt injection no MCP publico → sem acesso aos dados restricted
        - Credencial do MCP publico vazada → so acessa dados public/internal
        - Admin da Base Vetorial A comprometido → dados restricted intocados

(b) ZERO RISCO DE DATA LEAKAGE VIA QUERY

    O MCP publico (mcp-knowledge-public) se conecta APENAS a Base Vetorial A.
    Ele nao tem credencial, endpoint, ou qualquer forma de acessar a Base
    Vetorial B ou C. Mesmo que um atacante tenha controle total do MCP
    publico, ele so consegue ver dados public/internal.

    Comparacao com filtro pre-retrieval:
        - Filtro na query: depende de codigo correto → pode falhar
        - Segregacao fisica: nao depende de codigo → nao pode falhar

(c) AUDITABILIDADE SIMPLIFICADA

    Auditar acesso e trivial: basta verificar quem tem acesso a qual MCP.
    Nao e preciso auditar queries individuais para verificar se o filtro
    de confidencialidade esta funcionando — o proprio acesso ao MCP ja
    define o nivel maximo de dados que o usuario pode ver.

    Para auditores e reguladores, isso e muito mais facil de demonstrar:
        "O funcionario X tem acesso ao MCP publico. O MCP publico se conecta
        apenas a Base Vetorial A. A Base Vetorial A contem apenas dados
        public e internal. Logo, o funcionario X nao tem como acessar
        dados restricted ou confidential."

(d) SIMPLICIDADE CONCEITUAL

    O filtro de acesso deixa de ser um componente complexo na query e
    passa a ser uma questao de "qual MCP voce acessa?". Isso simplifica:
        - Desenvolvimento: nao precisa implementar filtro complexo
        - Teste: basta testar que o MCP se conecta a Base Vetorial correta
        - Operacao: monitorar 3 MCPs separados e mais simples que
          monitorar 1 MCP com logica de filtro complexa
        - Onboarding: explicar "voce tem acesso ao MCP publico" e mais
          simples que explicar "voce tem acesso ao MCP com filtro X, Y, Z"

(e) COMPLIANCE NATIVO

    A segregacao fisica atende NATIVAMENTE os requisitos de:
        - BACEN: dados regulados em ambiente controlado e isolado
        - LGPD: dados pessoais com acesso restrito e auditavel
        - CVM: informacoes privilegiadas isoladas fisicamente
        - SOX: controles de acesso a informacao financeira documentaveis
        - ISO 27001: segmentacao de rede e controle de acesso por nivel

(f) ALINHAMENTO COM ADR-002 (SOBERANIA)

    A segregacao por KB se alinha perfeitamente com as duas trilhas do
    ADR-002:
        - KB public-internal → Trilha A (cloud permitido)
        - KB restricted → Trilha B (on-premises obrigatorio)
        - KB confidential → Trilha B (on-premises obrigatorio, reforçado)

    Cada Base Vetorial reside na infraestrutura correta por design, nao
    por configuracao.

2.3. DESAFIOS E SOLUCOES
--------------------------------------------------------------------------------

............................................................................
DESAFIO 1: BUSCA CROSS-KB (pergunta que precisa de dados de multiplos niveis)
............................................................................

    Problema:
        Um usuario com perfil Director pode fazer uma pergunta cujo
        contexto ideal requer chunks de multiplos niveis de confidencialidade.
        Exemplo: "Qual o impacto da nova regulacao do BACEN na arquitetura
        do sistema de pagamentos?" — pode precisar de chunks da KB publica
        (arquitetura do sistema) E da KB restrita (regulacao em analise).

    Solucao: AGENTE ORQUESTRADOR CROSS-KB

        Implementar um agente orquestrador que:

        1. Recebe a pergunta do usuario
        2. Identifica quais MCPs o usuario tem acesso (baseado no perfil)
        3. Envia a query para CADA MCP autorizado em PARALELO
        4. Recebe os resultados de cada MCP
        5. Funde os resultados usando RRF — Reciprocal Rank Fusion (ADR-007)
        6. Aplica reranking no conjunto fundido
        7. Monta o prompt com chunks ordenados por relevancia
        8. Gera a resposta

        Regras do orquestrador:
            - O orquestrador roda com o PERFIL DO USUARIO — ele so acessa
              os MCPs que o usuario tem direito
            - O orquestrador NUNCA mistura chunks de niveis diferentes na
              mesma resposta sem INDICAR A CLASSIFICACAO de cada chunk
            - Cada chunk no contexto deve ter metadado [PUBLIC], [INTERNAL],
              [RESTRICTED] ou [CONFIDENTIAL] para a LLM saber a origem
            - A resposta gerada deve incluir a classificacao do nivel mais
              alto de chunk utilizado
            - Se chunks CONFIDENTIAL foram usados, a resposta inteira e
              classificada como CONFIDENTIAL

        Exemplo de resposta do orquestrador:
            "Classificacao da resposta: RESTRICTED
             Fontes utilizadas:
             - [PUBLIC] Documentacao do Sistema de Pagamentos v2.3
             - [RESTRICTED] Analise de Impacto — Regulacao BACEN 2026/Q1

             Resposta: ..."

............................................................................
DESAFIO 2: MESMO DOCUMENTO EM MULTIPLOS NIVEIS
............................................................................

    Problema:
        Um documento pode ter secoes com diferentes niveis de
        confidencialidade. Exemplo: um runbook de deploy pode ter:
        - Secao "Pre-requisitos" → INTERNAL
        - Secao "Credenciais de producao" → RESTRICTED
        - Secao "Procedimento de deploy" → INTERNAL

    Solucao: SPLIT NA FASE 2 (BETA.MD)

        Opcao A — Split por secao (recomendada):
            Na Fase 2 do pipeline (ADR-001), durante a geracao do .beta.md,
            identificar secoes com diferentes niveis de confidencialidade
            e gerar .beta.md SEPARADOS para cada nivel:

            Documento original:
                runbook-deploy-pagamentos.beta.md (confidentiality: mixed)

            Documentos gerados:
                runbook-deploy-pagamentos.internal.beta.md
                    (confidentiality: internal — secoes public+internal)
                runbook-deploy-pagamentos.restricted.beta.md
                    (confidentiality: restricted — secoes restricted)

            Cada .beta.md separado segue o pipeline normal e vai para sua
            KB correspondente. Uma referencia cruzada (cross_ref) no front
            matter indica que os documentos sao partes do mesmo original.

        Opcao B — Classificacao conservadora (fallback):
            O documento inteiro e classificado pelo nivel MAIS ALTO presente.
            Se uma secao e RESTRICTED, todo o documento vai para a KB
            restrita.

            Vantagens: simples, seguro por padrao
            Desvantagens: usuarios com acesso apenas ao nivel publico perdem
            acesso a partes do documento que seriam public/internal

        Decisao: usar Opcao A como padrao, com Opcao B como fallback para
        casos onde o split nao e viavel (documentos muito curtos, secoes
        interdependentes, custo de split desproporcional).

        Interacao com re-ingestao (ADR-001 secao 2.7):
            Quando uma fonte original e atualizada (ex: PDF v2) e o documento
            foi splitado em multiplos .beta.md por confidencialidade, o pipeline
            de re-ingestao segue o padrao 1-fonte → N-betas:
            1. Diff de fontes: comparar v1 com v2 da fonte original
            2. Mapear mudancas para cada .beta.md splitado via campo cross_ref
            3. Aplicar o merge (ADR-001 secao 2.7) SEPARADAMENTE em cada
               .beta.md — respeitando blocos LOCKED de cada um
            4. Se o split por confidencialidade mudar (ex: secao que era
               INTERNAL agora e RESTRICTED), gerar alerta para revisao humana
            O ADR-006 (Pipeline de Ingestao) deve implementar esse fluxo
            como extensao da re-ingestao padrao do ADR-001.

DESAFIO 3: ROTEAMENTO NO PIPELINE DE INGESTAO
............................................................................

    Problema:
        O pipeline de ingestao (ADR-006) precisa saber para qual KB e Base
        Vetorial cada documento deve ser enviado.

    Solucao: ROTEAMENTO POR FRONT MATTER

        Diagrama do fluxo de roteamento:

        ┌──────────┐    ┌────────────┐    ┌──────────────────┐
        │  Fonte   │───→│  .beta.md  │───→│  .md final       │
        │ (Fase 1) │    │  (Fase 2)  │    │  (Fase 3)        │
        └──────────┘    └────────────┘    └────────┬─────────┘
                                                   │
                                          Le front matter:
                                          confidentiality = ?
                                                   │
                              ┌─────────────────────┼──────────────────┐
                              │                     │                  │
                              ▼                     ▼                  ▼
                    ┌──────────────────┐  ┌───────────────┐  ┌────────────────┐
                    │ public/internal  │  │  restricted   │  │ confidential   │
                    │ → Base Vetorial A│  │ → Base Vet. B │  │ → Base Vet. C  │
                    │   (Fase 4)      │  │   (Fase 4)    │  │   (Fase 4)     │
                    └──────────────────┘  └───────────────┘  └────────────────┘

        O campo `confidentiality` no front matter (ADR-004, ADR-005) e o
        UNICO determinante do roteamento:

        confidentiality: public    → KB kb-public-internal → Base Vetorial A
        confidentiality: internal  → KB kb-public-internal → Base Vetorial A
        confidentiality: restricted → KB kb-restricted → Base Vetorial B
        confidentiality: confidential → KB kb-confidential → Base Vetorial C

        Implementacao no pipeline:
            1. Parser le o front matter do .md
            2. Extrai o campo confidentiality
            3. Valida que o valor e um dos 4 niveis aceitos
            4. Se campo ausente ou invalido: REJEITAR documento (fail-safe)
               NAO usar default — ausencia de classificacao e um ERRO
            5. Rotear para a Base Vetorial correspondente
            6. Registrar no log de ingestao: documento X roteado para KB Y

        Fail-safe critico:
            Se o campo confidentiality estiver AUSENTE, o pipeline DEVE
            rejeitar o documento e notificar o responsavel. NAO existe
            default seguro — um documento sem classificacao pode conter
            dados confidenciais. A ausencia de classificacao e tratada
            como ERRO, nao como "public por padrao".

............................................................................
DESAFIO 4: MANUTENCAO DE MULTIPLAS INSTANCIAS
............................................................................

    Problema:
        Manter 3 Bases Vetoriais separadas (backup, update, monitoring,
        patching) multiplica o esforco operacional.

    Solucao: PIPELINE PARAMETRIZADO + AUTOMACAO

        O mesmo pipeline de ingestao (ADR-006) e utilizado para as 3 KBs,
        parametrizado por:
            - KB_TARGET: kb-public-internal | kb-restricted | kb-confidential
            - VECTOR_DB_HOST: endpoint da Base Vetorial correspondente
            - VECTOR_DB_CREDENTIALS: credenciais da Base Vetorial correspondente
            - MCP_ENDPOINT: endpoint do MCP correspondente

        Automacao (exemplo ILUSTRATIVO com Docker Compose — a ferramenta de
        orquestracao de containers sera definida conforme o ambiente; em
        producao regulada, Kubernetes ou ECS podem ser mais adequados):
            docker-compose --profile public up -d
            docker-compose --profile restricted up -d
            docker-compose --profile confidential up -d

        Cada profile define:
            - Servico da Base Vetorial (instancia separada)
            - Servico do MCP Server (instancia separada)
            - Servico de monitoring (Prometheus/Grafana por KB)
            - Volumes de dados isolados
            - Rede Docker isolada por profile

        Backup:
            - Cada Base Vetorial tem seu proprio schedule de backup
            - KB public-internal: backup diario, retencao 30 dias
            - KB restricted: backup diario, retencao 1 ano, criptografado
            - KB confidential: backup diario, retencao 5 anos, criptografado,
              armazenado em cofre fisico separado

        Monitoring:
            - Dashboard unificado com visao das 3 KBs
            - Alertas separados por KB (criticidade proporcional ao nivel)
            - KB confidential: alerta imediato para qualquer anomalia

         ADR-006 (Pipeline de Ingestao) aplica-se integralmente a cada
         instancia — mesmas etapas, mesmos quality gates, mesma
         observabilidade.

............................................................................
DESAFIO 5: WIKILINKS CROSS-KB
............................................................................

    Problema:
        O ADR-001 (secao 2.2) define que wikilinks sao segregados
        (.beta.md navega para .beta.md, .md navega para .md). A segregacao
        por confidencialidade adiciona uma SEGUNDA dimensao de segregacao.
        Um documento da KB public-internal pode fazer wikilink para um
        documento da KB restricted?

    Decisao: WIKILINKS CROSS-KB SAO PROIBIDOS

        Regras:
        - Documentos de uma KB NAO podem fazer wikilink para documentos de
          outra KB (ex: um .md em public-internal nao linka para um .md em
          restricted)
        - Motivo: um wikilink para documento restricted em um documento
          publico revela a EXISTENCIA de informacao restrita ao usuario sem
          acesso, violando o principio de need-to-know
        - Para referencias cruzadas entre KBs, usar o campo cross_ref no
          front matter (sem link navegavel) — visivel apenas para quem tem
          acesso ao documento referenciado
        - O pipeline valida: wikilinks que apontam para documentos em outra
          KB sao tratados como ERRO de ingestao

............................................................................
DESAFIO 6: DOCUMENTOS EXPIRADOS (TEMPORALIDADE) NAS KBs
............................................................................

    Problema:
        O ADR-001 (secao 2.6) define temporalidade com valid_from/valid_until.
        Documentos expirados (valid_until preenchido e ultrapassado)
        permanecem na Base Vetorial ou sao removidos?

    Decisao: DOCUMENTOS EXPIRADOS PERMANECEM, COM FILTRO TEMPORAL

        - Documentos expirados NAO sao removidos da Base Vetorial — permanecem
          para consultas historicas ("como era a politica em 2025?")
        - O filtro temporal (ADR-001 secao 2.6, Camada 3) e aplicado por cada
          MCP individualmente ANTES de retornar resultados
        - O orquestrador cross-KB NAO precisa tratar temporalidade — cada MCP
          ja filtra seus proprios resultados
        - Documentos expirados sao marcados no retrieval como [EXPIRADO] para
          que a LLM e o usuario saibam que a informacao pode estar desatualizada

#decisao/desafios #decisao/wikilinks #decisao/temporalidade

--------------------------------------------------------------------------------
2.7. RELACAO COM ADR-010 (GIT FLOW E RELEASE)
--------------------------------------------------------------------------------

    NOTA: O ADR-010 define versionamento semantico e releases como conceito
    geral. Este ADR-011 ESTENDE o ADR-010 para suportar releases independentes
    por KB. O ADR-010 deve ser atualizado para referenciar o modelo de releases
    por KB aqui definido, confirmando compatibilidade com o Git Flow existente.

A segregacao por KB impacta o fluxo de release e versionamento:

(a) RELEASE TAGS INDEPENDENTES POR KB
    Cada KB tem seu proprio ciclo de release:
        - kb-public-internal@v1.2.0
        - kb-restricted@v1.0.3
        - kb-confidential@v1.0.1

    Isso permite:
        - Atualizar a KB publica sem tocar na restrita
        - Fazer hotfix na KB restrita sem afetar a publica
        - Release da KB confidencial com ciclo proprio (mais lento,
          mais gates de aprovacao)

(b) STAGING/PROD POR KB
    Cada KB tem seu proprio ambiente de staging:
        - Base Vetorial A-staging (staging da KB publica)
        - Base Vetorial B-staging (staging da KB restrita)
        - Base Vetorial C-staging (staging da KB confidencial)

    O staging da KB confidencial pode ter acesso ainda mais restrito
    que o staging das demais.

(c) PIPELINE DE DEPLOY SEPARADO
    O deploy de cada KB e uma operacao independente:
        - Deploy KB publica: pode ser automatico (CI/CD)
        - Deploy KB restrita: requer aprovacao de Manager
        - Deploy KB confidencial: requer aprovacao de Director + Compliance

(d) ROLLBACK INDEPENDENTE (ADR-008)
    Se um deploy da KB publica introduz erro, o rollback afeta apenas a
    Base Vetorial A. As Bases Vetoriais B e C permanecem intactas.

❌ 3. ALTERNATIVAS DESCARTADAS
================================================================================

............................................................................
ALTERNATIVA A: Filtro pre-retrieval na query (modelo ADR-004 original)
............................................................................

    Descricao:
        Manter uma unica Base Vetorial com todos os documentos de todos os
        niveis de confidencialidade. O controle de acesso e feito
        exclusivamente via filtro pre-retrieval na query:

        query = vector_search(
            embedding=question_embedding,
            filter={"confidentiality": {"$in": user.allowed_levels}},
            top_k=10
        )

    Pros:
        - Simplicidade operacional: apenas uma Base Vetorial para manter
        - Menor custo de infraestrutura (um unico servidor)
        - Busca cross-nivel nativa (sem orquestrador)
        - Menor complexidade do pipeline de ingestao

    Contras:
        - VULNERAVEL a bugs no filtro: um erro no codigo expoe TODOS os dados
        - VULNERAVEL a prompt injection: se o MCP tem acesso a Base Vetorial
          unica, um prompt injection pode bypassar o filtro
        - VULNERAVEL a misconfiguration: variavel de ambiente errada, perfil
          configurado incorretamente
        - VULNERAVEL a escalacao de privilegios: credencial da Base Vetorial
          vazada expoe TODOS os niveis
        - Auditoria complexa: precisa verificar que CADA query tem filtro correto
        - NAO atende requisitos regulatorios de isolamento fisico (BACEN, LGPD)
        - Violacao do principio de defesa em profundidade

    Motivo da rejeicao:
        Para uma instituicao financeira regulada, confiar EXCLUSIVAMENTE em
        um filtro de query como mecanismo de protecao de dados
        RESTRICTED/CONFIDENTIAL e INACEITAVEL. O risco de um unico bug
        expor dados confidenciais e desproporcional ao beneficio de
        simplicidade operacional. Reguladores esperam controles em
        multiplas camadas com isolamento fisico.

............................................................................
ALTERNATIVA B: Banco unico com Row-Level Security (RLS)
............................................................................

    Descricao:
        Usar uma unica Base Vetorial com mecanismo de Row-Level Security
        nativo do banco, onde cada registro (chunk) tem uma politica de
        acesso que o banco aplica automaticamente com base no usuario
        conectado.

    Pros:
        - Seguranca mais robusta que filtro na query (aplicada pelo banco)
        - Uma unica Base Vetorial para manter
        - Transparente para a aplicacao (o banco filtra automaticamente)
        - Suportado por alguns bancos relacionais (PostgreSQL, SQL Server)

    Contras:
        - NEM TODA Base Vetorial suporta RLS de forma madura
        - Depende de implementacao especifica do vendor — lock-in
        - Performance: RLS pode degradar busca vetorial em bases grandes
        - Complexidade de configuracao: politicas de RLS sao dificeis de
          debugar e auditar
        - NAO garante isolamento fisico — dados ainda estao no mesmo disco,
          mesma memoria, mesmo processo
        - Se o admin do banco tem acesso, ve TUDO (sem isolamento de admin)
        - Auditoria de RLS e complexa e vendor-specific
        - NAO atende requisito de isolamento fisico do BACEN para dados
          altamente sensiveis

    Motivo da rejeicao:
        RLS e um controle na camada de software do banco, nao na camada de
        infraestrutura. Para dados PUBLIC e INTERNAL poderia ser suficiente,
        mas para RESTRICTED e CONFIDENTIAL o BACEN espera isolamento fisico
        ou equivalente. Alem disso, a maturidade de RLS em bases vetoriais
        e limitada, e a dependencia de vendor e indesejavel.

............................................................................
ALTERNATIVA C: Criptografia at-rest com decrypt por perfil
............................................................................

    Descricao:
        Armazenar todos os dados em uma unica Base Vetorial, mas com chunks
        de niveis superiores criptografados com chaves diferentes. Apenas
        usuarios/MCPs com a chave correta conseguem descriptografar e ler.

    Pros:
        - Dados RESTRICTED/CONFIDENTIAL sao ilegíveis sem a chave
        - Uma unica Base Vetorial para manter
        - Mesmo que a Base Vetorial seja comprometida, dados criptografados
          permanecem protegidos
        - Alinhado com boas praticas de seguranca (defense in depth)

    Contras:
        - INCOMPATIVEL com busca vetorial: embeddings criptografados nao
          podem ser comparados por similaridade — o embedding precisa estar
          em texto claro para a busca funcionar
        - Se embeddings estao em texto claro, a criptografia do texto do
          chunk e parcialmente inutil — o embedding ja revela informacao
          semantica sobre o conteudo
        - Latencia: decrypt em tempo de query adiciona overhead significativo
        - Gerenciamento de chaves: N chaves para N niveis, rotacao, revogacao
        - Complexidade de implementacao: muito superior as outras alternativas
        - Nao resolve o problema do embedding exposto

    Motivo da rejeicao:
        Criptografia at-rest e uma boa pratica COMPLEMENTAR, mas nao resolve
        o problema fundamental: em uma base vetorial, os embeddings precisam
        estar em texto claro para a busca funcionar. Um embedding, embora
        nao seja legível por humanos, contem representacao semantica do
        conteudo — um atacante com acesso aos embeddings pode reconstruir
        informacoes sobre o conteudo usando tecnicas de inversion attack.
        Alem disso, a complexidade criptografica e desproporcional quando
        a segregacao fisica resolve o problema de forma mais simples e
        mais segura.

............................................................................
ALTERNATIVA D: Segregacao por 4 KBs (uma por nivel)
............................................................................

    Descricao:
        Criar 4 KBs separadas, uma para cada nivel de confidencialidade:
        KB-public, KB-internal, KB-restricted, KB-confidential.

    Pros:
        - Maximo isolamento: cada nivel completamente separado
        - Controle granular por nivel

    Contras:
        - Complexidade operacional excessiva: 4 Bases Vetoriais, 4 MCPs,
          4 pipelines, 4 ambientes de staging
        - PUBLIC e INTERNAL tem perfis de acesso muito similares — separar
          nao agrega seguranca significativa
        - Busca cross-KB mais complexa (4 MCPs no orquestrador vs. 3)
        - Custo de infraestrutura maior sem beneficio proporcional

    Motivo da rejeicao:
        A separacao PUBLIC/INTERNAL nao agrega seguranca significativa
        porque ambos os niveis sao acessiveis por todos os colaboradores
        autenticados. A diferenca entre PUBLIC e INTERNAL e mais sobre
        disseminacao EXTERNA (pode sair da empresa?) do que sobre acesso
        INTERNO. Juntar em uma unica KB simplifica sem comprometer seguranca.

================================================================================
⚖️ 4. CONSEQUENCIAS
================================================================================

--------------------------------------------------------------------------------
4.1. CONSEQUENCIAS POSITIVAS
--------------------------------------------------------------------------------

(+) Seguranca reforçada por isolamento fisico — elimina classe inteira de
    vulnerabilidades relacionadas a filtro de query

(+) Compliance nativo com BACEN, LGPD, CVM — isolamento fisico atende
    requisitos regulatorios sem controles adicionais complexos

(+) Auditoria simplificada — "quem acessa qual MCP" e documentavel e
    verificavel de forma trivial

(+) Alinhamento com ADR-002 (soberania) — cada KB na trilha de
    infraestrutura correta por design

(+) Blast radius limitado — comprometimento de uma Base Vetorial nao
    afeta as demais

(+) Release independente — cada KB com seu proprio ciclo, sem acoplamento

(+) Escalabilidade independente — Base Vetorial A pode escalar para
    centenas de milhares de chunks sem afetar performance da B ou C

(+) Simplicidade conceitual — controle de acesso e "qual MCP voce acessa",
    nao "qual filtro a query aplica"

--------------------------------------------------------------------------------
4.2. CONSEQUENCIAS NEGATIVAS
--------------------------------------------------------------------------------

(-) Custo operacional maior — 3 Bases Vetoriais para manter (backup,
    monitoring, patching, upgrade)

    Mitigacao: pipeline parametrizado, automacao via Docker Compose com
    profiles, monitoring unificado

(-) Complexidade de busca cross-KB — precisa de agente orquestrador

    Mitigacao: orquestrador e um componente bem definido; RRF (ADR-007)
    ja define como fundir resultados de multiplas fontes

(-) Duplicacao potencial de chunks — mesmo documento split em multiplos
    niveis gera duplicacao parcial

    Mitigacao: cross_ref no front matter; orquestrador deduplica por
    document_id na fusao de resultados

(-) Custo de infraestrutura maior — 3 servidores (ou 3 containers) em
    vez de 1

    Mitigacao: KB restricted e confidential tendem a ter MUITO menos
    documentos que a KB publica — instancias menores sao suficientes

(-) Latencia na busca cross-KB — consultar 3 MCPs em paralelo e mais
    lento que consultar 1

    Mitigacao: queries em paralelo, timeout por MCP, resultado parcial
    se um MCP nao responder a tempo

--------------------------------------------------------------------------------
4.3. RISCOS E MITIGACOES
--------------------------------------------------------------------------------

RISCO 1: Classificacao incorreta de documento (documento RESTRICTED
         classificado como INTERNAL por erro humano)

    Probabilidade: MEDIA — depende de processo humano
    Impacto: ALTO — documento sensivel na KB errada
    Mitigacao:
        - Quality gate na Fase 2 (ADR-001): revisao humana da classificacao
        - Validacao automatica: se documento menciona keywords sensiveis
          (CPF, contrato, salario, etc.) e esta classificado como public,
          pipeline emite WARNING
        - Revisao periodica: scan trimestral de todos os documentos por
          nivel para identificar classificacoes inconsistentes
        - Se duvida: classificar pelo nivel MAIS ALTO (conservative approach)

RISCO 2: Orquestrador cross-KB com falha de seguranca

    Probabilidade: BAIXA — componente isolado e auditavel
    Impacto: ALTO — pode expor dados de nivel superior
    Mitigacao:
        - Orquestrador roda com perfil do USUARIO, nao com perfil admin
        - Orquestrador nao tem credenciais proprias — usa token do usuario
        - Log completo de todas as queries do orquestrador
        - Pen test especifico no orquestrador
        - Orquestrador e stateless — nao armazena chunks ou respostas

RISCO 3: Drift de configuracao entre ambientes

    Probabilidade: MEDIA — 3 ambientes para manter sincronizados
    Impacto: MEDIO — pode causar inconsistencia de comportamento
    Mitigacao:
        - Infrastructure as Code (Terraform/Pulumi) para todos os ambientes
        - Configuracao versionada no Git
        - Drift detection automatica (comparacao periodica)
        - Alertas para desvios de configuracao

RISCO 4: Documento sem classificacao entra no pipeline

    Probabilidade: MEDIA — depende de disciplina no front matter
    Impacto: ALTO — documento pode ir para a KB errada (ou nenhuma)
    Mitigacao:
        - Pipeline REJEITA documentos sem campo confidentiality (fail-safe)
        - Pre-commit hook no Git valida presenca do campo
        - Quality gate na Fase 2 bloqueia promocao sem classificacao
        - Dashboard de documentos sem classificacao para acompanhamento

--------------------------------------------------------------------------------
5.1. FASE 1 — MVP (apenas KB public-internal, sem segregacao)
--------------------------------------------------------------------------------

    Objetivo:
        Validar o pipeline completo com uma unica KB antes de adicionar
        complexidade de segregacao.

    Entregas:
        [ ] Base Vetorial A (public-internal) operacional
        [ ] MCP mcp-knowledge-public configurado e funcionando
        [ ] Pipeline de ingestao roteando documentos public e internal
            para Base Vetorial A
        [ ] Validacao de que o campo confidentiality e lido e registrado
        [ ] Documentos RESTRICTED e CONFIDENTIAL rejeitados pelo pipeline
            (com log e notificacao)
        [ ] Testes de busca semantica na KB publica
        [ ] Monitoring basico (uptime, queries, latencia)

    DOD (Definition of Done):
        - Pipeline ingere documentos public/internal com sucesso
        - Documentos restricted/confidential sao rejeitados com mensagem clara
        - Busca semantica retorna resultados relevantes
        - Logs registram todas as operacoes de ingestao

    Responsaveis:
        - Engenharia: setup da Base Vetorial A + pipeline
        - Seguranca: validacao do mecanismo de rejeicao
        - Produto: curadoria dos primeiros documentos public/internal

    Duracao estimada: 2-3 sprints

5.2. FASE 2 — Adicionar KB restricted
--------------------------------------------------------------------------------

    Objetivo:
        Implementar segregacao fisica com a segunda KB (restricted) em
        infraestrutura on-premises.

    Pre-requisitos:
