#1 — Encoding quebrado no H1 do B08.md
Crítico

O Problema

O documento formal B08_pendencias.md tem o título H1 grafado como "Pendencias — Recomacoes e Alternativas", sem os acentos corretos ("Pendências", "Recomendações"). O front matter YAML está correto, mas o corpo do documento apresenta o erro — indicando que a conversão do draft .txt para .md teve problema de encoding UTF-8.

O Impacto

B08 é o documento mais referenciado da série (13 pendências citadas por praticamente todos os outros docs). Um erro visível no título compromete a credibilidade do material em apresentações e revisões. Além disso, se o encoding falhou no H1, pode haver outros trechos com caracteres corrompidos no corpo do documento que não foram detectados.

Por que Resolver

É uma correção de 10 segundos (editar o H1), mas garante integridade do documento mais importante da série. Recomendação: re-converter B08 do draft com encoding UTF-8 explícito e validar todo o corpo.

Docs afetados: B08.md (direto) — todos os 16 docs restantes referenciam B08
#2 — Campo source_system vs system nos metadados
Crítico

O Problema

O documento B04 (Metadados e Governança), na seção 2.3, define source_system como campo obrigatório no front matter. Porém, os documentos B00, B01, B02 e B03 — e o próprio CLAUDE.md — usam consistentemente o campo system. Não está claro se são sinônimos, se source_system é um campo adicional, ou se é um erro de nomenclatura.

O Impacto

O front matter é a base de toda a governança e filtragem do RAG. Se o nome do campo diverge entre documentos, o pipeline de ingestão (Prata→Ouro) vai rejeitar documentos válidos ou indexar com campos errados. Além disso, qualquer validação de schema contra o front matter obrigatório vai falhar dependendo de qual documento o desenvolvedor consultar.

Por que Resolver

É uma decisão simples: padronizar como system (já é o nome usado em 4 dos 5 docs e no CLAUDE.md). Se source_system tem semântica diferente, B04 precisa explicitar a diferença. Corrigir agora evita que o pipeline seja implementado com dois nomes para o mesmo conceito.

Docs afetados: B04.md (origem) — B00, B01, B02, B03, CLAUDE.md (divergência)
#3 — Critério Recall@10 diverge entre documentos
Crítico

O Problema

O critério de transição entre fases — medido por Recall@10 — tem valores diferentes em três documentos:

B12 define "≥ 80%" como meta para Fase 2
B15 define "≥ 70%" como pré-requisito para transição Fase 1→2
B16 define "≥ 70%" como critério de conclusão do Marco 1

Para Fase 2→3, a divergência é ainda maior: B12 diz "≥ 80%", B15 diz "≥ 80%", B16 diz "≥ 85%".

O Impacto

O Recall@10 é o gate principal para avançar de fase. Se cada documento define um número diferente, a equipe não sabe quando uma fase está "concluída". Isso causa discussões improdutivas e atrasa a transição — ou pior, permite transição prematura se alguém citar o documento com o critério mais baixo.

Por que Resolver

Escolher um valor único e propagar para os 3 documentos. Sugestão: 70% para Fase 1→2 (MVP, critério realista), 80% para Fase 2→3 (maturidade), 85% para Fase 3→4 (GraphRAG pleno). Documentar em B12 (fonte autoritativa de SLAs) e apenas referenciar em B15 e B16.

Docs afetados: B12.md, B15.md, B16.md

#4 — Fluxo de .md no Bronze: passa ou não?
Inconsistência

O Problema

B01 afirma que "todo documento passa pela bronze, inclusive .md". Porém, a estrutura de pastas sugerida no mesmo documento trata markdown/ como uma subpasta separada — sugerindo que .md já em repositórios Git poderiam ter tratamento diferenciado. B02 assume que documentos .md chegam do bronze com metadados de conversão, mas isso não faz sentido para .md que já estão no formato final.

O Impacto

Se .md do Git passam pelo bronze, o pipeline precisa de lógica para detectar que o arquivo já é .md e tem conversion_quality: 100%. Se não passam e vão direto para prata, a premissa de "bronze como fonte única da verdade" se quebra — e o pipeline prata precisa aceitar duas origens.

Por que Resolver

Clarificar em B01 e B02: ".md de repositórios Git entram pela bronze como cópia, mas com conversion_quality: 100 e source_format: markdown, pulando as etapas de conversão". Isso preserva a premissa sem criar complexidade desnecessária.

Docs afetados: B01.txt/md, B02.txt/md
#5 — Dimensão de embedding: 1536 vs 1024
Inconsistência

O Problema

B15 assume fixamente 1536 dimensões (float32 = 6 KB por vetor) para todos os cálculos de capacity planning. B12 aceita "1536 ou 1024" nos testes de validação de embedding. Se o modelo mudar de OpenAI (1536) para BGE-M3 (768/1024), todas as estimativas de B15 ficam erradas e o vector index precisa ser recriado.

O Impacto

Capacity planning com dimensão errada subestima ou superestima RAM, disco e latência. Se a troca de modelo acontecer em Fase 2, a re-indexação não tem procedimento documentado — B13 (Operações) descreve re-indexação de chunks mas não de embeddings com dimensão diferente.

Por que Resolver

Parametrizar a dimensão em B15 (ex: "N dimensões, onde N depende do modelo — ver Pendência 1 em B08"). Adicionar em B13 um procedimento de re-embedding quando o modelo muda. Isso desacopla o capacity planning da escolha do modelo.

Docs afetados: B12.md, B15.md, B13.md
#6 — Contagem de relações: "Todas (11)" sem lista explícita
Inconsistência

O Problema

B07 (Visão Consolidada) usa "Todas (11)" na tabela de evolução de relações, mas não lista quais são as 11. B05 lista as relações, mas ao contar chegam exatamente 11. O CLAUDE.md lista 11 relações. O problema é que "Todas" é ambíguo sem referência cruzada — se B05 adicionar uma 12ª relação, B07 fica desatualizado sem que ninguém perceba.

O Impacto

Baixo para implementação (a lista está correta em B05), mas cria confusão em revisões e apresentações quando alguém questiona "quais são as 11?".

Por que Resolver

Substituir "Todas (11)" por "11 relações (ver B05)" ou listar as relações na tabela. Manter a contagem sincronizada com B05 como fonte autoritativa.

Docs afetados: B07.txt/md, B05.txt/md
#7 — Fluxo de rollback urgente (PII) fragmentado
Inconsistência

O Problema

O procedimento de rollback urgente para dados pessoais (PII) está distribuído em 3 documentos sem fluxo completo em nenhum:

B13 diz "seguir procedimento de incidente completo (B14, §14.13)".
B15 diz "notificar compliance (B14)".
B14 tem classificação de incidentes mas falta o playbook completo de comunicação com órgãos reguladores e clientes.

O Impacto

Em um incidente real com vazamento de PII, a equipe precisaria consultar 3 documentos para montar o procedimento. Sem SLA definido, sem responsável claro (quem ativa: ops ou compliance?), e sem playbook de comunicação — o tempo de resposta será maior do que o aceitável.

Por que Resolver

Definir B14 como o documento autoritativo para resposta a incidentes de PII, com playbook completo (passo a passo, SLA por severidade, quem comunica). B13 e B15 devem apenas referenciar "ver B14, §14.13" sem duplicar o procedimento. Em contexto bancário (BACEN), este é um requisito regulatório.

Docs afetados: B13.md, B14.md, B15.md
#8 — Latência da API: exemplos desconectados dos SLAs
Inconsistência

O Problema

B10 mostra exemplos de resposta da API com valores de latência (142ms para /search, 1850ms para /ask) que parecem arbitrários — não referencia os SLAs formais. B12 define SLAs reais (p50 < 200ms, p95 < 500ms para /search; p50 < 3s, p95 < 8s para /ask), mas B10 não menciona nem cruza com esses números.

O Impacto

Desenvolvedores lendo B10 podem assumir que 1850ms é latência aceitável para /ask (está dentro do p95 de 8s mas longe do p50 de 3s). Sem referência cruzada, os SLAs de B12 ficam desconhecidos para quem implementa a API.

Por que Resolver

Adicionar em B10 uma nota nos exemplos: "Valores ilustrativos — para SLAs formais, ver B12, §12.5". Ou melhor: incluir uma tabela resumida de SLAs em B10 referenciando B12 como fonte autoritativa.

Docs afetados: B10.txt/md, B12.txt/md

#9 — Modelo de embedding não definido
B03

O Problema

B03 tem a única pendência marcada explicitamente no texto: "⚠️ PENDÊNCIA: definir modelo de embedding". Não há decisão sobre qual modelo usar — OpenAI text-embedding-3-small? BGE-M3 local? Qual dimensão? Qual custo?

O Impacto

A busca vetorial é o core do MVP (Fase 1). Sem modelo definido, não é possível dimensionar infra, estimar custos, nem implementar o pipeline Prata→Ouro. A Pendência 1 de B08 oferece análise, mas não tem decisão final.

Por que Resolver

Definir pelo menos a escolha para MVP: "OpenAI text-embedding-3-small para Fase 1, reavaliar BGE-M3 em Fase 2". Isso desbloqueia implementação sem comprometer flexibilidade futura.

#10 — Fórmula de conversion_quality
B02

O Problema

B02 descreve os sinais que compõem o conversion_quality (formatação preservada, tabelas intactas, imagens extraídas, etc.) e os thresholds (≥80% auto, 30-79% revisão, <30% rejeição) — mas não há fórmula para combinar múltiplos sinais em um score 0-100.

O Impacto

Cada desenvolvedor vai implementar a fórmula de forma diferente. Sem especificação, o QA não consegue validar se o cálculo está correto. O limiar de 80% perde sentido se a fórmula for arbitrária.

Por que Resolver

Documentar a fórmula: média ponderada dos sinais? Mínimo entre sinais? Regra de corte? Pode ser simples para MVP e refinada depois.

#11 — Enriquecimento semântico sem "como"
B05

O Problema

B05 descreve NER, classificação automática e resumos como capacidades da Fase 3, mas não especifica como: qual modelo de NER? LLM ou classificador treinado? Spacy em português? Qual precisão mínima aceitável?

O Impacto

A escolha impacta custo (LLM via API vs modelo local), latência do pipeline e qualidade dos metadados extraídos. Sem definição, a Fase 3 fica especulativa.

Por que Resolver

Não bloqueia Fase 1-2, mas deve estar definido antes de iniciar Fase 3. Registrar a decisão como ADR quando chegar o momento.

#12 — RBAC sem integração com IdP
B05

O Problema

B05 define perfis (Analista, Gestor, Diretoria, Agente de IA) mas não diz como mapear usuário → perfil. LDAP? OIDC? Manual?

O Impacto

Sem integração com IdP corporativo, o RBAC será implementado de forma ad-hoc — com risco de segurança e inconsistência com os demais sistemas da empresa.

Por que Resolver

Fase 3+. Antes de implementar, definir se usará OIDC (padrão corporativo) ou se a camada de aplicação fará o mapeamento manualmente.

#13 — Hybrid search: fusão sem algoritmo
B06

O Problema

B06 descreve que 3 buscas (vetorial, keyword, grafo) rodam em paralelo e os resultados são "fusionados" — mas não define a estratégia: Reciprocal Rank Fusion? Weighted average? Threshold-based?

O Impacto

A qualidade do hybrid search depende diretamente do algoritmo de fusão. Sem definição, cada implementação vai produzir resultados diferentes.

Por que Resolver

Fase 4. Documentar a estratégia quando iniciar implementação do hybrid search. RRF é o padrão da indústria e boa recomendação inicial.

#14 — Agentes sem dispatcher/roteamento
B06

O Problema

B06 define 3 agentes especializados por domínio mas não descreve quem roteia a pergunta para o agente certo. Existe um dispatcher? O LLM decide? É baseado em metadados?

O Impacto

Sem roteamento, ou o usuário precisa saber qual agente usar (péssima UX) ou todas as perguntas vão para um agente genérico (perde o benefício da especialização).

Por que Resolver

Fase 4. Definir quando começar a implementar agentes especializados. Padrão: LLM classifica a intenção e roteia.

#15 — Referências técnicas insuficientes
B09

O Problema

B09 lista apenas 3 URLs de documentação Neo4j. Faltam referências para: modelos de embedding (OpenAI, BAAI/BGE), reranking (Cohere, BGE-Reranker), chunking (unstructured.io, LangChain), LLM (Claude, Llama), deploy (Docker, K8s).

O Impacto

Um desenvolvedor novo no projeto não tem ponto de partida para aprender os componentes. B09 deveria ser o "índice de leitura recomendada" da série.

Por que Resolver

Expandir à medida que decisões são tomadas. Cada pendência resolvida de B08 deve gerar uma entrada em B09 com a documentação oficial do componente escolhido.

#16 — B14 não é referenciado por B10-B13
B14

O Problema

B14 (Segurança) é o documento mais completo da série (1168 linhas), com checklists por fase. Porém, nenhum dos documentos B10 (API), B11 (Deploy), B12 (Testes) ou B13 (Operações) referencia explicitamente o checklist de segurança de B14 nas suas respectivas seções.

O Impacto

Segurança fica "isolada" em um documento que ninguém consulta proativamente. Checklists de B14 não são aplicados porque os docs operacionais não os mencionam.

Por que Resolver

Adicionar em cada doc (B10-B13) uma referência: "Antes de implementar, consultar checklist de segurança em B14, §14.12 — Checklist Fase X". Integração simples, impacto alto.
