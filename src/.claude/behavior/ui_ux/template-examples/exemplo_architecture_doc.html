Decisao Aprovada
Adotar modelo de 3 camadas (Bronze → Prata → Ouro) para o pipeline de maturidade de documentos, onde Bronze = captura bruta das fontes originais, Prata = conversao e normalizacao em Markdown padronizado, Ouro = ingestao no Neo4j com embeddings e modelo de grafo completo.

Contexto

O projeto RAG Corporativo precisa ingerir documentos Markdown de multiplas fontes (Git repos, wikis, docs internos) para alimentar um knowledge graph no Neo4j. A qualidade dos documentos varia muito — desde rascunhos sem formatacao ate docs formais com front matter completo. Sem um pipeline de maturidade, documentos de baixa qualidade contaminariam o grafo e degradariam a busca vetorial.

Decisao Detalhada — Camadas de Maturidade
Bronze — Captura Bruta
Captura bruta das fontes originais sem nenhuma transformacao. Preserva formato original (Markdown, texto, HTML). Registra linhagem completa: repositorio, path, commit hash, branch, timestamp de coleta. Objetivo: garantir rastreabilidade e possibilitar reprocessamento a qualquer momento.
Prata — Conversao e Normalizacao
Conversao para Markdown normalizado com estrutura padronizada. Validacao de front matter conforme schema minimo obrigatorio (id, doc_type, title, system, module, owner, status). Resolucao de links internos entre documentos. Classificacao por tipo de documento (ADR, runbook, glossary, system-doc).
Ouro — Ingestao no Neo4j
Chunking semantico com faixa de 300-800 tokens por chunk, respeitando limites de heading. Geracao de embeddings por chunk com registro de modelo e versao. Persistencia no Neo4j como nos Document e Chunk com todas as relacoes do modelo de grafo (PART_OF, BELONGS_TO, OWNED_BY, USES_TERM). Criacao de indices vetoriais e auxiliares.
Alternativas Consideradas
3 alternativas
Pipeline unico sem camadas
Rejeitada
Sem controle de qualidade intermediario
Erros se propagam diretamente ate o grafo, contaminando a busca vetorial e degradando resultados dos agentes.
Modelo Bronze-Prata-Ouro
Escolhida
Separacao clara de responsabilidades
Permite reprocessamento parcial por camada, validacao intermediaria e rastreabilidade completa da linhagem documental.
Pipeline event-driven com filas
Rejeitada
Complexidade excessiva para MVP
Overhead operacional com brokers de mensagem, dead-letter queues e monitoramento distribuido. Desproporcional para o volume inicial de documentos.
Consequencias
Positivas
Reprocessamento parcial por camada
Qualidade rastreavel por estagio
Independencia entre etapas do pipeline
Negativas
Mais armazenamento intermediario (3x)
Latencia maior no pipeline completo
Risco: Se o volume de documentos ultrapassar ~2000 .md, a camada Prata pode precisar de processamento assincrono via fila, o que reintroduziria parte da complexidade da alternativa rejeitada.
