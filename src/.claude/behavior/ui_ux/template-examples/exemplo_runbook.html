Atencao: Este procedimento requer janela de manutencao. Busca vetorial ficara indisponivel durante a re-indexacao.
Objetivo
Neo4j

Re-criar o indice vetorial dos embeddings armazenados no Neo4j quando houver mudanca de modelo de embedding, corrupcao do indice existente ou degradacao significativa de performance nas buscas semanticas. Este runbook cobre o fluxo completo: verificacao do estado atual, remocao do indice antigo, re-geracao dos vetores e criacao de um novo indice otimizado.

Responsavel: Fabio A. B. Rodrigues Time: Arquitetura e Produtividade Corporativa Modulo: Embeddings / Vector Index
Pre-requisitos
4 itens
Acesso SSH ao servidor Neo4j
Credenciais de acesso remoto ao servidor de producao onde o Neo4j esta hospedado. Confirmar com o time de infra.
Backup recente confirmado
Garantir que existe um backup completo do banco Neo4j (neo4j-admin dump) realizado nas ultimas 24 horas.
Janela de manutencao aprovada
Aprovacao formal do Change Advisory Board (CAB) para a janela de manutencao. Notificar usuarios afetados com antecedencia minima de 2 horas.
Neo4j 5.x+ instalado
Verificar a versao do Neo4j instalada. Vector indexes requerem Neo4j 5.11 ou superior.
Procedimento
5 etapas
Etapas de Execucao
Sequencial
1
Verificar status atual do indice
Pendente
Antes de qualquer alteracao, verificar o estado atual dos indices vetoriais no Neo4j. Registrar o nome, tipo, dimensao e quantidade de entradas populadas para referencia de rollback.
Comando
cypher> SHOW INDEXES YIELD name, type, state, populationPercent, entityType WHERE type = 'VECTOR' RETURN name, state, populationPercent, entityType;
Verificacao
O indice vetorial chunk_embedding_index deve aparecer com state: ONLINE e populationPercent: 100.0. Registrar o nome exato do indice.
Saida Esperada
// Exemplo de saida esperada: +---------------------------------------------------------------+ | name | state | populationPercent | entityType | +---------------------------------------------------------------+ | chunk_embedding_index | ONLINE | 100.0 | NODE | +---------------------------------------------------------------+
2
Dropar o indice vetorial existente
Pendente
Remover o indice vetorial existente. A partir deste ponto, a busca semantica ficara indisponivel ate a conclusao do procedimento. Certifique-se de que a janela de manutencao esta ativa.
Comando
cypher> DROP INDEX chunk_embedding_index IF EXISTS;
Verificacao
Executar SHOW INDEXES WHERE type = 'VECTOR' novamente. Nenhum indice vetorial deve ser retornado.
Observacao
O comando IF EXISTS evita erro caso o indice ja tenha sido removido. A remocao do indice nao apaga os dados de embedding nos nos — apenas a estrutura de indexacao.
3
Executar re-geracao de embeddings
Pendente
Executar o script Python de re-geracao de embeddings. Este script ira iterar por todos os chunks no Neo4j, gerar novos vetores usando o modelo configurado e atualizar a propriedade embedding de cada no Chunk.
Comando
$ cd /opt/rag-pipeline $ source .venv/bin/activate $ python scripts/regenerate_embeddings.py \ --neo4j-uri bolt://localhost:7687 \ --neo4j-user neo4j \ --neo4j-password "${NEO4J_PASSWORD}" \ --model text-embedding-3-small \ --batch-size 100 \ --dimension 1536 \ --verbose
Verificacao
O script deve finalizar com a mensagem Embeddings regenerados com sucesso. Total: XXXX chunks processados. Erros: 0. Verificar que o total corresponde ao numero esperado de chunks.
Observacao
O tempo de execucao depende da quantidade de chunks. Para ~10.000 chunks, esperar aproximadamente 15-20 minutos. Monitorar o uso de memoria do processo e da API de embeddings.
4
Criar novo indice vetorial
Pendente
Criar o novo indice vetorial com as configuracoes adequadas ao modelo de embedding utilizado. A dimensao deve corresponder exatamente ao modelo (ex: 1536 para text-embedding-3-small).
Comando
cypher> CREATE VECTOR INDEX chunk_embedding_index IF NOT EXISTS FOR (c:Chunk) ON (c.embedding) OPTIONS { indexConfig: { `vector.dimensions`: 1536, `vector.similarity_function`: 'cosine' } };
Verificacao
Executar SHOW INDEXES WHERE type = 'VECTOR'. O indice deve aparecer com state: ONLINE e populationPercent: 100.0. Aguardar ate 5 minutos para populacao completa em bases grandes.
Saida Esperada
// Apos populacao completa: +---------------------------------------------------------------+ | name | state | populationPercent | entityType | +---------------------------------------------------------------+ | chunk_embedding_index | ONLINE | 100.0 | NODE | +---------------------------------------------------------------+
5
Validar integridade — query de teste
Pendente
Executar uma busca vetorial de teste para confirmar que o indice esta funcional e retornando resultados relevantes. Usar uma query conhecida cujo resultado esperado seja previsivel.
Comando
cypher> // Gerar embedding de teste e executar busca CALL db.index.vector.queryNodes( 'chunk_embedding_index', 5, $test_embedding ) YIELD node, score RETURN node.chunk_id AS chunk_id, node.text AS text_preview, score ORDER BY score DESC;
Verificacao
A query deve retornar 5 resultados com scores entre 0.0 e 1.0 (similaridade cosseno). Os resultados devem ser semanticamente relevantes para a query de teste. Score do top-1 acima de 0.80 indica boa qualidade.
Observacao
Apos validacao bem-sucedida, comunicar o time que a busca vetorial esta restaurada e encerrar a janela de manutencao formalmente.
Procedimento de Reversao
Procedimento de Reversao: Em caso de falha em qualquer etapa, restaurar o backup do banco Neo4j e recriar o indice com as configuracoes anteriores. Tempo estimado de rollback: 20 minutos.
Etapas de Rollback
Reversao
Parar o Neo4j e restaurar backup
Interromper o servico Neo4j, restaurar o dump mais recente e reiniciar o servico.
$ sudo systemctl stop neo4j $ neo4j-admin database load --from-path=/backups/neo4j-latest.dump neo4j --overwrite-destination $ sudo systemctl start neo4j
Recriar indice anterior
Caso o backup ja contenha o indice, este passo nao e necessario. Caso contrario, recriar o indice com as configuracoes anteriores.
cypher> CREATE VECTOR INDEX chunk_embedding_index IF NOT EXISTS FOR (c:Chunk) ON (c.embedding) OPTIONS { indexConfig: { `vector.dimensions`: 1536, `vector.similarity_function`: 'cosine' } };
Validar restauracao
Executar a mesma query de teste da Etapa 5 para confirmar que o rollback foi bem-sucedido e a busca vetorial esta funcional.
cypher> SHOW INDEXES WHERE type = 'VECTOR';
Troubleshooting
3 problemas conhecidos
Problemas Conhecidos e Solucoes
Indice nao aparece apos criacao
Sintoma
Apos executar o comando CREATE VECTOR INDEX, o indice nao aparece na listagem de SHOW INDEXES ou aparece com state: POPULATING indefinidamente.
Causa Provavel
Transacao nao comitada. Se o comando foi executado dentro de uma transacao explicita (ex: :BEGIN no Neo4j Browser), o indice so sera visivel apos o :COMMIT.
Solucao
Verificar se ha transacoes pendentes com SHOW TRANSACTIONS. Executar :COMMIT ou finalizar a sessao e abrir uma nova. Se o indice permanecer em POPULATING, verificar os logs do Neo4j em /var/log/neo4j/neo4j.log.
Comando de Diagnostico
cypher> SHOW TRANSACTIONS YIELD transactionId, status, currentQuery RETURN transactionId, status, currentQuery;
Query vetorial retorna 0 resultados
Sintoma
A busca vetorial (db.index.vector.queryNodes) retorna um resultado vazio, mesmo com embeddings populados e indice no estado ONLINE.
Causa Provavel
Embeddings com dimensao incorreta. Se o modelo foi trocado (ex: de 1536 para 3072 dimensoes), mas o indice foi criado com a dimensao anterior, os vetores nao serao indexados.
Solucao
Verificar a dimensao dos embeddings armazenados vs. a dimensao configurada no indice. Dropar e recriar o indice com a dimensao correta. Confirmar com: MATCH (c:Chunk) RETURN size(c.embedding) LIMIT 1.
Comando de Diagnostico
cypher> MATCH (c:Chunk) WHERE c.embedding IS NOT NULL RETURN size(c.embedding) AS dimensao, count(*) AS total ORDER BY total DESC LIMIT 5;
Timeout na re-indexacao
Sintoma
O processo de criacao do indice vetorial trava ou o Neo4j fica irresponsivo durante a populacao do indice em bases com grande volume de chunks (>50.000).
Causa Provavel
Heap insuficiente. A construcao do indice HNSW em memoria requer heap proporcional ao numero de vetores e suas dimensoes. Para 50K vetores de 1536 dimensoes, recomenda-se no minimo 4GB de heap.
Solucao
Aumentar o heap maximo do Neo4j em neo4j.conf: configurar server.memory.heap.max_size=8g. Reiniciar o Neo4j e tentar novamente a criacao do indice. Monitorar o consumo de memoria com htop durante o processo.
Comando de Diagnostico
$ grep "heap" /etc/neo4j/neo4j.conf $ curl -u neo4j:${NEO4J_PASSWORD} http://localhost:7474/db/manage/server/jmx/domain/java.lang/name=Heap%20memory%20usage
