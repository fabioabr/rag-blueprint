Este glossário padroniza a terminologia utilizada nos documentos corporativos do pipeline RAG. Todos os termos listados devem ser referenciados de forma consistente em ADRs, runbooks e documentação técnica da Empresa XPTO.
Detalhamento dos Termos 8 termos
Ingestão Pipeline
Processo de captura, conversão e persistência de documentos no grafo Neo4j. Envolve as etapas de descoberta, parsing, chunking, geração de embeddings e upsert no banco de dados.
No projeto No pipeline RAG corporativo, a ingestão é o processo que transforma arquivos Markdown dos repositórios Git em nós Document e Chunk no Neo4j, preservando metadados do front matter e relações entre documentos.
Exemplo: Um arquivo ADR-001.md é ingerido e gera 1 nó Document + 12 nós Chunk com embeddings calculados pelo modelo text-embedding-3-small.
Relacionados: Chunking Embedding Front Matter
Chunking Pipeline
Processo de dividir um documento em fragmentos semânticos menores (chunks), respeitando limites de 300-800 tokens. Cada chunk herda os metadados do documento pai e recebe um identificador único.
No projeto O chunking é realizado por headings do Markdown, garantindo que cada fragmento mantenha coerência semântica. A estratégia varia conforme o tipo de documento: ADRs usam chunks menores e precisos, enquanto runbooks são divididos por procedimento.
Exemplo: Um documento de arquitetura com 5 seções H2 gera 5 chunks, cada um com tamanho entre 300 e 800 tokens, herdando sistema, módulo e tags do front matter.
Relacionados: Chunk Ingestão Embedding
Reranking Pipeline
Etapa de reordenação dos resultados de busca vetorial com base em relevância contextual. Utiliza um modelo cross-encoder para recalcular a pontuação de cada resultado considerando a pergunta original.
No projeto O reranking ocorre após a busca vetorial e a expansão de contexto via grafo, garantindo que os chunks mais relevantes para a pergunta do usuário sejam priorizados no prompt enviado ao LLM.
Exemplo: A busca vetorial retorna 20 chunks candidatos. O reranker reclassifica e seleciona os 5 mais relevantes para compor o contexto da resposta.
Relacionados: Hybrid Search Embedding GraphRAG
Chunk Dados
Fragmento semântico de um documento, unidade mínima de busca vetorial. Herda metadados do documento pai. Representado como nó no grafo Neo4j com relação (:Chunk)-[:PART_OF]->(:Document).
No projeto Cada chunk armazena o texto do fragmento, seu embedding vetorial, e metadados herdados (sistema, módulo, confidencialidade). É a unidade atômica consultada durante a busca semântica.
Exemplo: O chunk CHK-000123-003 contém o texto da seção "Fluxo de Cobrança" do documento DOC-000123, com 520 tokens e embedding de 1536 dimensões.
Relacionados: Chunking Embedding Ingestão
Embedding Dados
Representação vetorial de um chunk, gerada por modelo de linguagem. Permite busca por similaridade semântica. Armazenado como propriedade do nó Chunk no Neo4j com índice vetorial dedicado.
No projeto Os embeddings são gerados na etapa 4 do pipeline de ingestão, utilizando modelos como text-embedding-3-small da OpenAI. O vector index do Neo4j permite busca por similaridade cosseno sobre esses vetores.
Exemplo: Um chunk de 450 tokens gera um vetor de 1536 dimensões. A busca "como funciona a cobrança?" retorna os chunks cujos embeddings têm maior similaridade cosseno com o embedding da pergunta.
Relacionados: Chunk Hybrid Search Reranking
Front Matter Dados
Bloco YAML no início de um arquivo Markdown contendo metadados estruturados (id, tipo, sistema, módulo, tags, etc.). Define a identidade e classificação do documento na base de conhecimento.
No projeto Todo documento Markdown corporativo deve conter front matter com campos obrigatórios: id, doc_type, title, system, module, domain, owner, team, status, confidentiality, tags. Esses metadados são extraídos na etapa de parsing e persistidos como propriedades do nó Document.
Exemplo: ---\nid: DOC-000123\ndoc_type: system-doc\ntitle: Módulo de Cobrança\nsystem: Sistema Exemplo\n---
Relacionados: Ingestão Chunk
GraphRAG Arquitetura
Padrão arquitetural que combina Knowledge Graph com Retrieval-Augmented Generation para respostas contextualizadas. Vai além do RAG tradicional ao utilizar relações explícitas entre entidades para enriquecer o contexto recuperado.
No projeto O GraphRAG é implementado sobre Neo4j, combinando busca vetorial em chunks com travessia de grafo para expandir o contexto. Um agente pode, por exemplo, encontrar um chunk sobre cobrança e navegar até o sistema, módulo e ADRs relacionados.
Exemplo: Pergunta: "Quem é responsável pelo módulo de cobrança?". O GraphRAG encontra chunks relevantes via vetor, depois navega (:Document)-[:OWNED_BY]->(:Owner)-[:MEMBER_OF]->(:Team) para complementar a resposta.
Relacionados: Hybrid Search Reranking Chunk
Hybrid Search Arquitetura
Estratégia de busca que combina busca vetorial (similaridade semântica) com busca estruturada (filtros de metadados e travessia de grafo). Garante resultados mais precisos e contextualizados do que cada abordagem isolada.
No projeto A Hybrid Search é o mecanismo central do retriever. Primeiro aplica filtros de acesso (confidencialidade, domínio), depois executa busca vetorial nos chunks, e por fim expande o contexto via travessia do grafo Neo4j antes do reranking.
Exemplo: Busca por "fluxo de pagamento" filtra por confidentiality: internal e system: "Sistema Financeiro", recupera chunks por similaridade vetorial, e expande via relações [:BELONGS_TO] e [:DEPENDS_ON].
Relacionados: GraphRAG Embedding Reranking
